SECTION 1 — WHAT WAS THE PROBLEM
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Before P5-C, the condition step was visual pseudo-logic with no runtime effect.

executeConditionStep() evaluated its conditions correctly and returned:
  { success: true, output: { conditionResult: bool, nextSteps: number[] } }

But runStepLoop() was a plain for-loop:
  for (let i = startFromIndex; i < steps.length; i++) { ... }

The loop never read result.output.nextSteps.  It incremented i unconditionally.
Every condition step silently fell through to the next step in array order,
regardless of whether the condition matched or not.

The type system exposed this: ConditionStep.config had:
  onTrue:  number[]   — array of step indices (never acted upon)
  onFalse: number[]   — array of step indices (never acted upon)

WG-03 in validator.ts blocked any workflow that configured onTrue/onFalse
non-empty, with the message "branching logic blocked until Phase 4 (DAG execution)".
So no production workflow could express condition-based routing at all.

SECTION 2 — ROUTING ARCHITECTURE

2.1  Type System Changes (types.ts)
────────────────────────────────────

ConditionStep.config replaced:
  BEFORE:  onTrue: number[];   onFalse: number[];
  AFTER:   onTrueStepIndex: number | null;   onFalseStepIndex: number | null;

Design rationale:
  • Single integer target per branch (deterministic, no ambiguity).
  • null = "no routing for this branch; fall through linearly."
    This enables condition steps used purely for audit/diagnostics (record the
    evaluation result in step output without actually branching).
  • number = step.index of the target step (forward-only, validated).

StepResult gained a new optional field:
  nextStepIndex?: number;

  Set ONLY by executeConditionStep when onTrueStepIndex / onFalseStepIndex is
  non-null for the matched branch.  All other step types must never set this.
  Its absence tells runStepLoop to advance linearly — it must never default to 0.

2.2  Condition Step Handler (steps/condition.ts)
─────────────────────────────────────────────────

Before:
  returned nextSteps: number[] in output (array, ignored by executor).

After:
  const matched = evaluateConditions(step.config.conditions, data);
  const selectedNextStepIndex = matched
    ? step.config.onTrueStepIndex
    : step.config.onFalseStepIndex;

  result.output = { matched, selectedNextStepIndex, nextSteps: [...] }
  if (selectedNextStepIndex !== null) result.nextStepIndex = selectedNextStepIndex;

Key points:
  • output.matched — boolean evaluation result; always present in output.
  • output.selectedNextStepIndex — the step.index that was selected (or null).
    Diagnostic timelines and the audit trail read this field.
  • result.nextStepIndex — the routing signal consumed by the executor.
    Only set when the matched branch had a non-null target.
  • output.nextSteps — kept for backward compatibility with legacy readers.
  • The handler NEVER validates routing targets.
    All safety enforcement lives exclusively in resolveNextCursor() so the
    enforcement is in one place and testable independently.

2.3  Pure Routing Resolver (executor.ts — exported)
──────────────────────────────────────────────────────

A new exported pure function was extracted from the executor:

  export function resolveNextCursor(
    result:      StepResult,
    currentStep: WorkflowStep,
    steps:       WorkflowStep[],
  ): { nextCursor: number; routed: boolean }
     | { error: string; code: "BACKWARD_JUMP" | "SELF_LOOP" | "ROUTE_NOT_FOUND" }

Logic:
  1. If result.nextStepIndex === undefined (no routing signal):
       Find current step's array position via findIndex(s => s.index === currentStep.index).
       Return { nextCursor: currentPos + 1, routed: false }  ← identical to old i++.

  2. If result.nextStepIndex === currentStep.index:
       Return { error, code: "SELF_LOOP" }.

  3. If result.nextStepIndex < currentStep.index:
       Return { error, code: "BACKWARD_JUMP" }.

  4. Find target array position via steps.findIndex(s => s.index === targetIdx).
       If -1: Return { error, code: "ROUTE_NOT_FOUND" }.

  5. Return { nextCursor: targetArrayPos, routed: true }.

Why it uses step.index and not array position:
  The routing target in onTrueStepIndex / onFalseStepIndex is a step.index
  value, not an array offset.  For well-formed workflows where step.index equals
  array position, these are equivalent.  The findIndex() lookup handles the
  general case (e.g., non-contiguous indices) correctly.

Why it is exported:
  Pure functions should be testable without a database.  All 57 new tests call
  resolveNextCursor() directly — zero DB mocking required.

2.4  Executor Traversal Changes (executor.ts — runStepLoop)
──────────────────────────────────────────────────────────────

BEFORE (for-loop):
  for (let i = startFromIndex; i < steps.length; i++) {
    const step = steps[i]!;
    ...
    // i++ happens automatically
  }

AFTER (cursor-based while-loop):
  let cursor = startFromIndex;
  while (cursor < steps.length) {
    const step = steps[cursor]!;
    ...
    // ── Routing resolution (P5-C) ───────────────────────────────────────────
    const routeResult = resolveNextCursor(result, step, steps);

    if ("error" in routeResult) {
      // Log condition_route_invalid + guarded fail transition
      return;
    }

    if (routeResult.routed) {
      // Log condition_route_selected
    } else if (step.type === "condition" && result.nextStepIndex === undefined) {
      // Log condition_route_missing (linear fallthrough from condition step)
    }

    cursor = routeResult.nextCursor;   // sets cursor++ for linear, or jumps for routing
  }

Semantic equivalence with old for-loop:
  • startFromIndex works identically: initial=0, resume=currentStepIndex+1.
  • For all non-condition steps: routeResult.routed=false, cursor=prevPos+1.
    Identical behavior to i++.
  • currentStepIndex DB update uses cursor (was i): same value for well-formed steps.
  • All guarded transitions (step-fail, approval-pause, TTL, cancel) run at
    exactly the same points in the loop body — none were moved.

SECTION 3 — VALIDATION RULES (validator.ts)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

WG-03 (condition branching blocked) is LIFTED.

The old block is replaced by routing validation rules. A pre-pass now builds
allStepIndices (Set<number>) from the steps array before the per-step loop,
so routing target lookups are O(1) instead of O(n) per condition step.

Rules enforced at activation time (errors block publication):

  R-01  onTrueStepIndex / onFalseStepIndex must be a non-negative integer or null.
        Error codes: WG-03_INVALID_TRUE_ROUTE, WG-03_INVALID_FALSE_ROUTE
        (catches floats, strings, NaN, negative values)

  R-02  Self-loop detection: target must not equal current step.index.
        Error codes: WG-03_TRUE_ROUTE_SELF_LOOP, WG-03_FALSE_ROUTE_SELF_LOOP

  R-03  Forward-only: target must be strictly greater than current step.index.
        Error codes: WG-03_TRUE_ROUTE_BACKWARD, WG-03_FALSE_ROUTE_BACKWARD

  R-04  Target existence: allStepIndices must contain the target value.
        Error codes: WG-03_TRUE_ROUTE_NOT_FOUND, WG-03_FALSE_ROUTE_NOT_FOUND
        (error message lists all valid step indices for admin guidance)

  R-05  null targets are always valid (no routing on this branch — fall through).
        Not an error. Not a warning.

Validation is applied independently per branch:
  True and false branch errors are emitted separately.
  A step can have a valid true route and an invalid false route — both are reported.

Invariant: if validateWorkflow passes, resolveNextCursor will NEVER return an
error for any condition step at runtime (assuming the snapshot matches the
validated definition, which P5-A guarantees).

4.1  condition_route_selected  (level: INFO)

Emitted when a condition step successfully routes to a target.

  workflowId         number
  workspaceId        number
  stepIndex          number   — step.index of the condition step
  stepName           string
  matched            boolean  — result of condition evaluation

  availableRoutes    unknown  — output.nextSteps array (both branches' targets)

───────────────────────────────────────────────

Emitted when resolveNextCursor returns an error (routing violation).
Should never occur in validated workflows — indicates a bug or corrupt snapshot.

  executionId        number
  workflowId         number
  workspaceId        number
  stepIndex          number   — step.index of the condition step
  stepName           string
  routeCode          string   — "BACKWARD_JUMP" | "SELF_LOOP" | "ROUTE_NOT_FOUND"
  routeError         string   — full human-readable error from resolveNextCursor
  nextStepIndex      number   — the invalid target that was attempted
  availableRoutes    number[] — all step indices that were valid
  action             "condition_route_invalid"

Message: "[governance] P5-C: condition routing violation — failing execution safely"

4.4  Diagnostics Integration
──────────────────────────────

Execution timelines show routing decisions through the step output field
already stored in workflow_execution_steps.output:

  {
    "matched": true,
    "selectedNextStepIndex": 4,
    "nextSteps": [4]
  }

The GET /workflows/executions/:id diagnostics endpoint already returns step
logs with their output JSONB.  The routing decision for any condition step is
therefore immediately visible in the execution timeline without any new endpoint
or schema change:

  Step 2 (condition) → output.matched=true → output.selectedNextStepIndex=4
  Step 4 (notification) → ran because condition routed here
  Step 3 (task) → was NOT run (skipped by true-branch routing)

Any step that was skipped will simply have no log entry in workflow_execution_steps.
The absence of a step entry is itself observable.

5.1  Approval Resume (P4-E)
────────────────────────────

UNCHANGED and fully compatible.

resumeExecution() computes:
  resumeFromIndex = execution.currentStepIndex + 1

currentStepIndex is set to `cursor` (the array position) each iteration:
  .set({ currentStepIndex: cursor })

For well-formed workflows, cursor === step.index (array position equals logical
index).  The resume formula produces the correct array position for the next step
regardless of whether the paused step was reached via linear advance or a routing
jump.

Scenario: condition(0) routes to approval(2) → approval pauses.
  currentStepIndex = 2 (cursor at the approval step).
  Resume: startFromIndex = 3.
  runStepLoop starts at cursor=3. ✓

The approval step is never re-run.  No duplicate notifications.  No duplicate
approval records.  P4-E exact-once guarantee preserved.

5.2  TTL Check (P4-B)
──────────────────────

UNCHANGED and fully compatible.

The TTL check runs at the SAME position in each iteration — before routing
resolution.  If TTL is exceeded after a condition step completes (even during
a routing jump to a far step), the execution halts with status='timed_out'
before the cursor advances.  The guarded WHERE NOT IN terminal transition is
identical to pre-P5-C.

No routing jump can skip the TTL check.  Governance is enforced at every
inter-step boundary including routed boundaries.

5.3  Cancellation Check (P4-C)
────────────────────────────────

UNCHANGED and fully compatible.

Same as TTL: cancellation check runs before routing resolution.  Setting
cancelRequested=true during a condition-routed execution will halt the next
time the boundary is reached — no jump can bypass it.

5.4  Guarded Transitions (P4-D)
─────────────────────────────────

All guarded UPDATEs (WHERE status = X or WHERE status NOT IN terminal) are
UNCHANGED.  The cursor loop does not introduce any new status write paths.
The only new write is the guarded fail transition on routing violation:
  .where(and(eq(id, executionId), notTerminal(status)))
  ...same guard as all other failure transitions.

5.5  Snapshot Semantics (P5-A)
────────────────────────────────

UNCHANGED.

runStepLoop receives `steps: WorkflowStep[]` from the caller.
The caller (executeWorkflow / resumeExecution) resolves steps from the snapshot
or live definition exactly as in P5-A.  P5-C has no opinion on where the steps
come from — it receives the final array and traverses it.

A condition step in a snapshot uses the frozen onTrueStepIndex / onFalseStepIndex
from when the execution was triggered.  Live definition edits to routing cannot
affect in-flight executions.

5.6  Forward-Only Safety: Loop Termination Proof
──────────────────────────────────────────────────

CLAIM: The P5-C while-loop always terminates.

PROOF SKETCH:
  Let N = steps.length.  cursor starts at startFromIndex ∈ [0, N].

  Each iteration:
    Case A (linear): cursor = currentPos + 1 > currentPos ≥ cursor_before.
      cursor strictly increases.
    Case B (routed): cursor = targetArrayPos where
      steps[targetArrayPos].index > currentStep.index.
      Since steps are stored in ascending index order (index === array position
      for well-formed workflows), targetArrayPos > cursor_before.
      cursor strictly increases.
    Case C (error): function returns immediately. Loop terminates.
    Case D (governance halt): function returns immediately. Loop terminates.

  Since cursor strictly increases each iteration and the loop condition is
  cursor < N, the loop terminates in at most N iterations. □

  Backward jumps (which would cause cursor to decrease and risk infinite loops)
  are caught by resolveNextCursor() with code="BACKWARD_JUMP" and cause an
  immediate guarded fail → return.  They are also blocked at activation time
  by the validator.

  Self-loops (cursor unchanged) are caught with code="SELF_LOOP".

  Therefore: no infinite loop is possible at runtime. □

SECTION 6 — REMAINING DAG LIMITATIONS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

P5-C implements DETERMINISTIC FORWARD-ONLY BRANCHING — the minimal correct
subset of DAG execution needed for production condition routing.

The following DAG capabilities are deliberately NOT implemented:

NOT BUILT — Parallel execution
  Condition steps cannot fan out to run multiple branches simultaneously.
  onTrueStepIndex and onFalseStepIndex are single targets (not arrays).
  Parallel execution requires a join/merge semantics model, a fanout counter,
  and concurrent step execution — none of which is in scope for P5-C.

NOT BUILT — Branch merge / convergence point
  After a true-branch or false-branch completes, there is no "merge" step that
  automatically joins the two paths.  Both branches continue linearly to the end
  of the workflow (or to the next step reachable from their respective paths).
  A "merge" pattern can be modeled with a notification or task step placed at a
  high index that both branches route to — this works with the current model.

NOT BUILT — Cycle runtime recovery
  The runtime cannot detect cycles in snapshots that bypassed the validator.
  The only cycle protection is:
    a) Validator blocks self-loops and backward routes at activation.
    b) resolveNextCursor halts on any backward jump or self-loop at runtime.
  If somehow a valid-looking cycle reaches the runtime (bug in validator),
  the execution fails with condition_route_invalid (BACKWARD_JUMP or SELF_LOOP).

NOT BUILT — Multiple condition branches (switch/case)
  Each condition step has exactly one true target and one false target.
  There is no "match expression" with N branches.
  Workaround: chain multiple condition steps (each evaluates a different field).

NOT BUILT — Nested branching / sub-graphs
  Routing from a condition step to another condition step works correctly
  (the inner condition evaluates and routes to its targets).  However, there
  is no graph compiler to validate the full branching graph at activation time.
  Current validation only checks per-condition-step targets.

NOT BUILT — Dead code / unreachable step detection
  A step that can only be reached via a branch that is never taken is not
  flagged as unreachable.  The validator currently does not perform reachability
  analysis on the workflow graph.
  Impact: minor UX issue for workflow builders; no runtime impact.

8.2  What Is Ready for Workflow Builders
─────────────────────────────────────────

  Workflow admins can now configure condition steps with:
    - A single onTrueStepIndex pointing to any forward step.
    - A single onFalseStepIndex pointing to any forward step.
    - null for either branch (fall through linearly).

  These workflows pass the updated validator (WG-03 block lifted).
  The executor routes correctly at runtime.
  Routing decisions are visible in execution timelines.

  EXAMPLE — Priority-based notification routing:
    Step 0: condition (check priority=urgent) → onTrueStepIndex=2, onFalseStepIndex=1
    Step 1: notification (standard message to manager)
    Step 2: notification (urgent escalation to director + manager)
    Step 3: task (assigned to HR)

    Result:
      urgent tickets: 0 → 2 → 3
      other tickets:  0 → 1 → 2 (no skip in this case) OR 0 → 1 (if step 2 not after 1)

    Note: in the above, step 2 follows step 1 anyway, so both paths hit step 2.
    For real skipping: place the skipped step between the condition and the merge point.

8.3  What the Validation Engine Phase Adds
────────────────────────────────────────────

  P5-C validator (current) covers:
    ✓ Per-step: route targets valid, forward-only, non-self-loop.
    ✓ Per-step: route targets exist in the step set.
    ✓ Pre-activation gate (blocks publication).

  Future validation engine would add:
    ○ Graph-level reachability analysis (unreachable step detection).
    ○ Branch convergence validation (both branches eventually reach the same step).
    ○ Fanout safety estimation (how many notifications at worst case).
    ○ Step dependency analysis (does step 4 read step 2 output even if step 2 is skipped?).
    ○ Cross-step output reference validation (ctx.stepOutputs[i] — is i always run?).

  The current P5-C validation is necessary and sufficient for safe condition
  routing.  The future validation engine adds quality-of-life checks, not
  correctness guarantees.  P5-C is production-ready without it.

8.4  Known Limitation: Skipped Step Output References
───────────────────────────────────────────────────────

  If step 5 reads ctx.stepOutputs[3] (output of step 3), and a condition step
  at index 2 routes past step 3 (e.g., onTrueStepIndex=4), then step 3 never
  ran and stepOutputs[3] is undefined when step 5 executes.

  Current behavior:
    step 5 reads ctx.stepOutputs[3] → undefined → handler uses a default or skips.
    No crash — all step handlers must handle undefined stepOutputs gracefully.

  Detection: not currently possible at validation time (no dependency graph).
  Mitigation: workflow builders should ensure steps after a branch convergence
    do not depend on step outputs from steps that may be skipped.
  Future: cross-step output reference validation (validation engine phase).
