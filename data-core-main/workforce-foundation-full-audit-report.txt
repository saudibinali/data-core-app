EXECUTIVE SUMMARY
================================================================================

The platform implements a credible **HR Foundation** admin surface
(hr-foundation.tsx) backed by workspace-scoped master data tables and
POST /hr/foundation/seed defaults. This IS a real foundation architecture
for **configurable HR metadata**.

It is NOT yet **Runtime Organizational Intelligence**. Most foundation
entities are lookup catalogs; employee/contract/document rows use parallel
text fields without FK enforcement. Exceptions with real runtime impact:

  • hr_org_units → employees.orgUnitId
  • hr_job_titles / hr_job_grades → employee FKs
  • hr_leave_policies → leave requests, requiresApproval, balance mutations

Position seats (hr_positions) are fully configurable in Foundation but
NOT assigned to employees in application code (positionId unused).

Parallel legacy org (departments on users) remains outside Foundation UI.

Verdict: **Semi-structured HCM | Metadata-heavy foundation shell |
Partially operational org | Not enterprise workforce platform yet**

================================================================================
ORGANIZATIONAL STRUCTURE
================================================================================

SUPPORTED IN hr_org_units:
  company, division, department, team (+ unit in UI; branch in schema comment)

HIERARCHY:
  parentId tree — YES stored, NO chart, NO cycle validation, NO runtime walk

NOT SUPPORTED:
  Matrix org, dual reporting, policy inheritance, unit-head approval routing

EMPLOYEE BINDING:
  Single optional orgUnitId — NOT mandatory

LEGACY PARALLEL:
  flat departments table for users — NOT in Foundation page

================================================================================
POSITION / JOB MODEL
================================================================================

Job Title  = catalog (hr_job_titles) — USED on employee
Job Grade  = catalog (hr_job_grades) — USED on employee
Position   = seat (hr_positions) — CONFIG ONLY; employee occupies dept not seat
Role       = users.role (RBAC) — NOT HR job role

Employee model today: org unit + title + grade + optional free-text position

================================================================================
WORKFORCE RELATIONSHIPS
================================================================================

Direct manager: employees.directManagerId → used by CANONICAL LEAVE
Line manager:   users.lineManagerId      → used by WORKFLOWS
Dept manager:   departments.managerId    → legacy, limited use

NO: delegation, acting manager, alternate approver, org-graph escalation

================================================================================
WORKFLOW / APPROVAL DEPENDENCIES
================================================================================

Workflow engine:     users.lineManagerId + role — IGNORES foundation org
Leave:               employee manager + leave policy requiresApproval
Self-service:        employee userId link; policies list
Attendance:          employee userId; locations partial
Payroll:             employeeId; org optional

BLOCKERS for enterprise approvals:
  • Split manager models
  • No org-head resolver
  • No delegation
  • Foundation not wired to workflow steps
  • Optional org on hire → admin fallback approvers
