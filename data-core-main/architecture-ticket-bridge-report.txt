## Objective
Create a one-way Bridge Layer making appEventBus the official event entry point,
with a unidirectional bridge to the legacy eventDispatcher for backward compat.

## What Was Built

### 1. bridge.ts (NEW FILE)
Path: artifacts/api-server/src/lib/events/bridge.ts

The Bridge Layer:
  - Subscribes to ALL appEventBus events via appEventBus.subscribeToAll()
  - Translates TypedEvent → EventPayload (field mapping below)
  - Awaits eventDispatcher.dispatch() — writing to workspace_event_logs and
    triggering the WorkflowEngine and legacy notifications.ts listener
  - Errors are caught and logged — never propagate back to appEventBus callers
  - Guards: idempotent (calling registerBridge() twice is a no-op with warning)

Field mapping (TypedEvent → EventPayload):
  event.type                      → payload.event
  event.module                    → payload.module
  event.workspace.workspaceId     → payload.workspaceId
  event.actor.userId (if number)  → payload.triggeredBy
  event.data + _busEventId        → payload.data (with tracing fields injected)
    + _busCorrelationId

Tracing fields injected into EventPayload.data:
  _busEventId:        UUID from appEventBus.emit() — links workspace_event_logs
                      row back to the originating bus event
  _busCorrelationId:  correlationId for distributed tracing chains

Exported:
  registerBridge()      — registers the subscribeToAll() wildcard listener
  isBridgeRegistered()  — diagnostic helper for health checks and test setup

### 2. index.ts (UPDATED)
Path: artifacts/api-server/src/lib/events/index.ts

Registration order (side-effect imports):
  1. listeners/notifications (legacy eventDispatcher listeners)
  2. registerActivityListeners()     — 6 listeners on appEventBus (Ticket 04)
  3. registerNotificationBusListeners() — 6 listeners on appEventBus (Ticket 05)
  4. registerBridge()                — wildcard appEventBus → eventDispatcher

Rationale for order: domain listeners (2, 3) run in parallel with the bridge (4)
because appEventBus uses Promise.allSettled fan-out. Registering them before the
bridge means no ordering dependency — all run concurrently on each emit.

Updated exports: added registerBridge, isBridgeRegistered, EVENT_TYPES,
LEGACY_EVENT_NAMES to the barrel.

## No-Loop Guarantee

  appEventBus → bridge → eventDispatcher
                                 │
                   notifications.ts (DB insert only — no emit)
                   engine.ts (DB read/write only — no emit)

  eventDispatcher has NO reference to appEventBus.
  Legacy listeners (notifications.ts, engine.ts) do NOT emit to appEventBus.
  Therefore: no circular event cycles are possible.

================================================================================

## Bridge Benefit — approval events now fully wired

Before the bridge:
  - approval.created / approval.completed went ONLY to appEventBus
  - workspace_event_logs had NO record of these events
  - WorkflowEngine could NOT trigger on approval events

After the bridge:
  - approval.created / approval.completed go to appEventBus (typed listeners)
  - THEN bridge forwards to eventDispatcher → workspace_event_logs ✅
  - THEN WorkflowEngine evaluates any workflows triggered by "approval.created" ✅
  - Legacy notifications.ts listener receives them too (no-op: no approval handlers)

This is the primary new capability unlocked by the bridge.

## Known Debt

DEBT-BRIDGE-01: 4 route files still use eventDispatcher.dispatch() directly.
  Their events DO NOT reach appEventBus listeners.
  Each has an ARCH-TODO comment with exact migration steps.
  Priority: tickets.ts (HIGH) > admin.ts (MEDIUM) > forms.ts (LOW/complex)

DEBT-T05-01 (from Ticket 05, inherited):
  Legacy notifications.ts passes titleAr/messageAr/link fields that don't exist
  in the notifications table schema. Drizzle silently ignores extra fields.
  Impact: No functional breakage, but data is lost.
  Fix: Remove those fields or add columns before the next migration.

DEBT-BRIDGE-02: Bridge handler awaits eventDispatcher.dispatch() (includes DB write).
  At current scale this is acceptable (route uses void appEventBus.emit()).
  At high volume: make bridge dispatch fire-and-forget to reduce fanOut latency.
