T005 — Executor: waitForDelay pause + resumeDelayedExecution (executor.ts)
  Dispatch switch: "delay" case now calls executeDelayStep() instead of no-op.

  runStepLoop: added guarded running → waiting_delay transition after
  the waitForApproval block (mirrors P4-D model exactly):
    UPDATE SET status='waiting_delay', wakeAt, waitingReason='delay',
               scheduledStepIndex=cursor+1
    WHERE id=$id AND status='running'
    RETURNING id
  On empty return: transition_race_lost → log + return.
  On success: logs execution_delay_paused + returns (stops the loop).

  resumeDelayedExecution(executionId, now?) → string code
  New public export called by the scheduler.  Governance order:
    1. Fetch execution (SELECT)
    2. Pre-check: terminal → "already_terminal"
    3. Pre-check: status ≠ waiting_delay → "not_waiting_delay"
    4. Pre-check: cancelRequested=true → guarded → cancelled → "cancel_requested"
    5. Pre-check: TTL expired → guarded → timed_out → "ttl_expired"
    6. Pre-check: wakeAt > now → "wake_at_not_reached" (scheduler guard)
    7. Snapshot-first step resolution (P5-A model)
    8. Reconstruct ExecutionContext (JSONB + completed step rows)
    9. Guarded acquisition: waiting_delay → running (sets resumedAt=now)
       WHERE status='waiting_delay' → RETURNING id
       On empty: "transition_race_lost"
   10. void runStepLoop(…, scheduledStepIndex, …) → "ok"

T006 — Scheduler: WorkflowScheduler + singleton (scheduler.ts)
  Class WorkflowScheduler:
    start()    — starts the setTimeout polling loop (idempotent)
    stop()     — clears the timer, sets running=false (idempotent)
    pollOnce(now?) → { found, resumed, skipped }
      SELECT id, wakeAt FROM workflow_executions
      WHERE status='waiting_delay' AND wake_at <= now
      LIMIT 10
      → for each: resumeDelayedExecution(id, now)
      → emit scheduler_poll_cycle log with counters

  Design decisions:
    • setTimeout loop (not setInterval) — next cycle starts AFTER current
      completes, preventing overlapping poll cycles under load.
    • No per-execution timer IDs — all timing lives in DB (wake_at).
    • Guarded acquisition in resumeDelayedExecution ensures exactly-once
      wake-up even if two scheduler instances race (future-safe).
    • POLL_INTERVAL_MS = 15 000 (15 seconds) — bounded staleness, low DB load.
    • BATCH_SIZE = 10 — prevents large backlogs from flooding the event loop.

  Exported singleton: workflowScheduler

GOVERNANCE MODEL
────────────────

New state added to the lifecycle:

  pending → running → waiting_delay → running (wake) → completed
                    ↘ failed        ↗ cancelled (if cancelRequested at wake time)
                                     timed_out (if TTL expired before wake-up)

Guarded transitions (P4-D model, exactly-once):
  running → waiting_delay   WHERE status='running'
  waiting_delay → cancelled  WHERE status='waiting_delay'   (cancelRequested path)
  waiting_delay → timed_out  WHERE status='waiting_delay'   (TTL path)
  waiting_delay → running   WHERE status='waiting_delay'   (scheduler acquisition)

RESTART SAFETY:
  wakeAt + scheduledStepIndex persisted atomically in the running→waiting_delay
  transition.  On server restart: scheduler reads from DB on first poll cycle.
  No timer re-registration, no in-memory state.

IDEMPOTENCY:
  Scheduler SELECT + resumeDelayedExecution guarded UPDATE: if two processes
  race, exactly one wins.  The loser's UPDATE returns empty → transition_race_lost
  → skip silently.  This is already safe for future multi-replica deployments.


AUDIT EVENTS ADDED
──────────────────
  execution_delay_started      — delay step resolved wakeAt (step handler)
  execution_delay_paused       — running→waiting_delay succeeded (executor)
  execution_delay_resumed      — scheduler acquired execution (resumeDelayedExecution)
  execution_delay_cancelled    — cancelRequested=true at wake time
  execution_delay_ttl_expired  — TTL expired before scheduler wake-up
  scheduler_started            — engine started the scheduler
  scheduler_stopped            — scheduler stopped
  scheduler_poll_cycle         — each poll cycle (found/resumed/skipped counters)
  execution_delay_resume_blocked — scheduler skipped an execution (non-race reason)
