EXECUTIVE SUMMARY
================================================================================
P18-C produced a full leave migration blueprint from legacy hr_employee_leaves
to canonical leave_requests + leave_approval_steps. The canonical API
(routes/leave.ts) is already implemented with transactions, balance locks,
conflict detection, bus events, and notifications — but production databases
created from Drizzle journal 0000 lack the canonical tables (P18-B D1).

Production UI (hr-me-leave.tsx, hr-attendance.tsx, hr-employee-detail.tsx)
still uses legacy endpoints. Critical defect: self-service leave list calls
GET /hr/attendance/leaves which requires hr.manage — employees cannot list
their own requests reliably.

Recommended execution: L1 DB migration on staging first, then API tests,
then frontend bridge with legacy write freeze before broad rollout.

CANONICAL LEAVE DECISION
================================================================================
- Canonical tables: leave_requests, leave_approval_steps
- Canonical API: routes/leave.ts (POST/GET/PATCH approve|reject|withdraw)
- Canonical balances: hr_leave_balances (shared today)
- Legacy: hr_employee_leaves + hr.ts paths — freeze then read-only, no drop
- request_number: LRQ-YYYYMM-RAND (new); LRQ-MIG-{legacyId} (migration)
