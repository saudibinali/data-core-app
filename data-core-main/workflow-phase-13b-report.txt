1. BACKEND STATE MACHINE
   File: artifacts/api-server/src/lib/workspace-lifecycle.ts (NEW)

   Pure state machine — no DB, no HTTP, fully testable in isolation.

   Exports:
   • WorkspaceLifecycleState — 5 states: pending_activation | active |
     suspended | locked | archived
   • WorkspaceLifecycleAction — 5 actions: activate | suspend | restore |
     lock | archive
   • LIFECYCLE_STATE_TO_DB_STATUS — state → DB column value map
   • deriveLifecycleState(workspaceStatus) — DB status → lifecycle state
   • lifecycleStateToDbStatus(state) — lifecycle state → DB column value
   • LIFECYCLE_ACTION_MODEL — full action definitions with allowedFrom[],
     targetState, severity, requiresReason=true, requiresConfirmation=true,
     isDestructive=false, auditEventType
   • ALL_LIFECYCLE_ACTIONS — ordered list of 5 actions
   • isTransitionAllowed(currentState, action) → boolean
   • getAllowedActionsFrom(currentState) → action[]
   • REASON_MIN_LENGTH = 10
   • validateLifecycleRequest(req, currentState) → LifecycleValidationResult
   • validateLifecycleActorRole(role) → boolean (super_admin only)
   • buildLifecycleAuditPayload(...) → LifecycleAuditPayload

   Transition table:
     pending_activation → activate → active
     active             → suspend  → suspended
     active             → lock     → locked
     active             → archive  → archived
     suspended          → restore  → active
     suspended          → lock     → locked
     suspended          → archive  → archived
     locked             → restore  → active
     locked             → suspend  → suspended
     locked             → archive  → archived
     archived           → restore  → active
     archived           → suspend  → suspended

   Validation order (validateLifecycleRequest):
     1. Action must be a known action     → UNKNOWN_ACTION
     2. Reason ≥ REASON_MIN_LENGTH chars  → REASON_REQUIRED
     3. confirmation === true             → CONFIRMATION_REQUIRED
     4. Transition must be allowed        → TRANSITION_NOT_ALLOWED

3. BACKEND ROUTE
   File: artifacts/api-server/src/routes/tenants.ts — PATCH route added

   PATCH /platform/tenants/:tenantId/lifecycle
   Auth: requireAuth + requireSuperAdmin
   Body: { action, reason, internalNote?, confirmation }
   Returns: { tenant: PlatformTenantProfile, lifecycleEvent: LifecycleAuditPayload }

   Route logic:
     1. Validate tenantId (positive integer)
     2. Load workspace from DB — 404 if not found
     3. Derive current lifecycle state from workspace.status
     4. validateLifecycleRequest() — 400 on any validation failure
     5. Compute targetState and newDbStatus
     6. UPDATE workspaces SET status = newDbStatus
     7. INSERT into activity_logs (activityLogsTable) — full audit record
     8. req.log.info — structured log with actorId, tenantId, transition detail
     9. Return updated PlatformTenantProfile + LifecycleAuditPayload

   Audit log schema (activity_logs):
     userId     = req.userId (actor)
     workspaceId = workspace.id
     action     = auditEventType (e.g. "workspace_lifecycle_suspended")
     metadata   = JSON.stringify({ action, previousState, targetState,
                                   reason, internalNote, tenantId })

SAFETY NOTES
────────────────────────────────────────────────────────────────

• No workspace deletion — archive writes "disabled" to status, data preserved
• No billing/payment/HR/email actions in scope
• No autonomous suspension — all transitions are operator-initiated
• No AI decisions — pure deterministic rule-based state machine
• Reason required (min 10 chars) for every transition
• Explicit boolean confirmation required for every transition
• Super-admin role enforced at both HTTP middleware and pure function level
• All transitions audit-logged to activity_logs before response is sent
• Invalid transitions fail with structured error code (TRANSITION_NOT_ALLOWED)
• Unknown states default to pending_activation (fail-closed)
