2.2  Per-event display (ForensicEventCard)

  Every event card renders:

  Row 1 (header):
    [eventType mono badge]  [IntegrityStatusBadge]  [RetentionBadge]  #NNN

  Row 2 (integrity warning — only for non-healthy):
    ⚠ {INTEGRITY_STATUS_MAP[status].description}

  Row 3 (entity):
    🏷 entityType  entityId (mono)  👤 operatorId (mono)

  Row 4 (timestamps):
    🕐 Occurred: {fmtDate(occurredAt)}
       Recorded: {fmtDate(recordedAt)}  (if present)

  Row 5 (chain / hash):
    🔗 Chain: chainId (mono, truncated)
    # Hash: hashPreview (mono)

  Row 6 (evidence ref):
    Evidence: evidenceRef (mono)

  Left border color by integrity tier:
    healthy    → emerald-400

2.4  Safety guarantees

  - No export or download triggers
  - No AI summary cards
  - Compromised and orphaned events are never filtered out or hidden
  - Non-healthy status description is always shown inline on each affected card
  - Component is purely presentational (no internal query calls)

SECTION 3 — ENTITY-SCOPED FORENSIC REVIEW BEHAVIOR
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The "Forensic Entity Investigation" card is the P12-B centrepiece.

3.1  Interaction flow

  1.  Operator types an entity ID (workspace ID, user ID, ticket ID, etc.)
      into the entity ID input field (font-mono, data-testid: forensic-entity-id-input).

  2.  Optional: operator selects an entity type from the dropdown
      (decorative hint only in P12-B — does not filter the API call, which
       uses entity ID alone since the forensics endpoint is :entityId-based).

  3.  Operator clicks "Review Timeline" or presses Enter.
      Button is disabled while entity ID is empty.

  4.  activeEntityId is set to the trimmed entityId. This enables
      useGovernanceForensicTimeline(activeEntityId), which fires
      GET /api/platform/compliance/forensics/{entityId}.

  5.  GovernanceForensicTimeline renders with the returned events
      (mapped from the raw API response to the ForensicEvent interface).

  6.  The "Clear" button resets both activeEntityId and entityIdInput,
      returning to the "no entity selected" empty state.

3.2  No-entity-selected state

  When activeEntityId is undefined:
    - GovernanceForensicTimeline is not mounted
    - No API call fires
    - A dashed-border card shows the FORENSIC_EMPTY_STATE.noEntitySelected
      title and description
    - data-testid: forensic-no-entity

3.3  Query disabled behavior

  useGovernanceForensicTimeline is guarded by:
    enabled: typeof entityId === "string" && entityId.trim().length > 0

  Empty string, whitespace-only string, and undefined all produce `enabled: false`.
  TanStack Query will not fire the queryFn. No network request is made.
  No error state is shown for a disabled query.

3.4  Active entity badge

  When activeEntityId is set, a badge displays the entity ID being reviewed:
    [🔍 {activeEntityId}] — N events found (or "Loading…" / "Error…")

3.5  Hash mapping from raw API

  The forensics endpoint returns `hash` (full SHA-256 hex string).
  The page maps this to `hashPreview` as `hash.slice(0, 16) + "…"` before
  passing to the timeline component. The full hash is never displayed in the
  UI — only the truncated preview (sufficient for visual identification,
  insufficient for misuse as a collision target).

4.6  Empty state with filter awareness

SECTION 5 — INTEGRITY STATUS VISUAL LANGUAGE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Status → Tier mapping (INTEGRITY_STATUS_MAP):

  Status        Tier        Icon          Badge colours          Description
  ──────────────────────────────────────────────────────────────────────────────
  verified      healthy     ShieldCheck   emerald bg/text        "Hash verification passed — entry is intact."
  warning       attention   FileWarning   amber bg/text          "Minor integrity issue detected — review recommended."
  compromised   critical    ShieldX       red bg/text            "Hash mismatch detected — tamper evidence present."
  orphaned      critical    ShieldX       red bg/text            "Entry references a missing parent chain link."
  incomplete    attention   FileWarning   amber bg/text          "Entry is missing required fields or linkage."

Design principles:

  - critical (red) is reserved for compromised and orphaned — states with
    hard evidence of data integrity failure. These states are NEVER hidden,
    filtered out, or soft-rendered.
  - attention (amber) is used for warning and incomplete — states that
    need review but do not have confirmed tamper evidence.
  - healthy (emerald) is used for verified — the normal/expected state.
  - No "grey out" or "dim" of critical entries. Critical severity is always
    shown at full visual weight.
  - No labels imply legal compliance conclusions ("compliant", "legal",
    "regulatory", "verdict").
  - Left border colors on timeline events mirror the badge color:
    emerald for healthy, amber for attention, red for critical.

5.1  Status legend on the integrity status card

  The overall integrity status card shows a compact inline legend (hidden on
  mobile, visible on lg+ screens) mapping each status to its tier label.
  This helps operators understand the color coding without guessing.

SECTION 6 — RETENTION CLASSIFICATION VISIBILITY
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Retention classification → badge colours (RETENTION_CLASSIFICATION_MAP):

  Classification         Label                  Badge colour      Helper text
  ──────────────────────────────────────────────────────────────────────────────
  operational            Operational            secondary (grey)  "Routine platform operation record."
  governance             Governance             blue bg/text      "Governance oversight record."
  compliance_sensitive   Compliance Sensitive   orange bg/text    "Sensitive to compliance evaluation — handle with care."
  forensic_critical      Forensic Critical      violet bg/text    "Critical for investigation — highest retention priority."

Display locations:
  - IntegrityBadge + RetentionBadge both appear on every audit chain row
  - RetentionBadge appears on every forensic timeline event card
  - Helper text is surfaced as a `title` attribute on the badge (tooltip on hover)
  - Filter dropdown shows full label names for easy selection

Priority escalation from label colours:
  grey < blue < orange < violet (forensic_critical = highest urgency)
  This matches the intuitive priority ordering for investigative importance.

Semantic clarification:
  "forensic_critical" means the record is important for investigation purposes.
  It does NOT mean the underlying action was illegal or non-compliant.
  The helper text explicitly uses "investigation" language, not legal language.
