2. SYSTEM QUALITY ASSESSMENT
================================================================================

Scalable?
  MODERATE. In-process synchronous resume limits throughput. DB polling scheduler
  is multi-instance safe for delays. Rate limit 100 exec/min/workspace. Role fanout
  capped at 50. Monolithic workflows route file is maintenance risk.

Maintainable?
  MODERATE-LOW. Three approval systems + package drift (core-workflows vs api-server).
  Strong test suite helps. Excellent inline schema documentation.

Operationally meaningful?
  YES FOR ADMINS / OPS. Event-driven automation for tickets, forms, notifications,
  tasks, delays works. NO FOR EMPLOYEES seeking unified approvals/tracking.

================================================================================
3. RISKS & BOTTLENECKS
================================================================================

Hardcoded bottlenecks:
  - workflow.manage required to approve any execution
  - 24h execution TTL (not per-step)
  - 50 notification recipient cap
  - 7 fixed step types
  - Ticket-only legacy approvals table

Architectural risks:
  - Dual/triple approval models (workflow_approvals, approvals, leave_approval_steps)
  - Form submission status decoupled from workflow terminal state
  - In-process resume crash window (stuck running)
  - core-workflows / core-approvals not wired — future refactor debt
  - Naming collision: governance_workflow_actions vs workflow_executions

Runtime inconsistencies:
  - waiting_approval not shown in my-submissions enrichment
  - Multi/parallel approval types ≠ behavior
  - onTimeout escalate in seeds ≠ executor
  - No approvers → step skipped (success), not failed
  - department_head in types, not in resolver

Dangerous complexity:
  - Admins configure escalation/parallel modes believing they work
  - Silent auto-bypass when approver resolution fails
  - Any workflow.manage user can approve any pending execution

4. CAN THIS BECOME A REAL ENTERPRISE WORKFLOW PLATFORM?
================================================================================

Answer: YES — WITH FOCUSED STABILIZATION, NOT GREENFIELD REWRITE.

Existing assets worth keeping:
  - workflow_definitions + versions + executions + step audit
  - Event bus trigger model + form hint routing
  - Approval pause/resume with guarded transitions
  - Delay scheduler + TTL + publish validation
  - 30+ automated tests

Missing for enterprise grade:
  - Single approval abstraction (or clear deprecation of legacy)
  - Approver-scoped authorization + delegation + reassignment
  - Per-step timeout/escalation worker
  - Employee inbox + deep-linked notifications
  - Form/workflow status synchronization
  - Optional queue for long-running resume
  - Wire core-workflows / core-approvals packages
