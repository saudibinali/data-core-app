━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Convert point-in-time governance visibility into historical operational
intelligence via append-only snapshot persistence, trend extraction, and
alert frequency analysis.

Prior to P7-A:
  • evaluateTenantHealth() computed a fresh health state on every API call
    and discarded it after the HTTP response.
  • No historical record existed — trend analysis was impossible.
  • Alert codes were regenerated live; no frequency or recurrence data.
  • A workspace that was "critical" at 3am left no permanent record.

After P7-A:
  • Every governed health evaluation can be persisted as an immutable row.
  • Trend series (error rates, backlogs, stuck counts, severity evolution)
    are queryable across 1h / 24h / 7d / 30d windows.
  • Alert frequency intelligence detects "chronic" GOV-* codes.
  • Full observability event chain from capture through chronic detection.

Indexes:
  idx_gov_snap_workspace_time     → (workspace_id, captured_at)
    Primary trend query index. Covers: WHERE workspace_id=? AND captured_at>=?
  idx_gov_snap_workspace_severity → (workspace_id, severity, captured_at)
    For severity-filtered queries: "how often was this workspace critical?"

JSONB design rationale:
  • metricsSnapshot stores 11 numeric fields from OperationalMetricsSnapshot
    (excludes workspaceId, capturedAt, estimatedNotificationFanout which are
    first-class columns or derivable).
  • alert_codes as JSONB string[] enables fast frequency aggregation without
    joining to a separate alerts table.
  • schema_version enables future readers to detect and migrate legacy rows
    if JSONB shapes change in a breaking way.

captureGovernanceSnapshot(workspaceId, db?, now?)

  Step 1: evaluateTenantHealth()
          Reads workflow_executions (read-only).
          Runs computeMetricsFromRows, detectStuckFromRows,
          detectStormFromRows, generateGovernanceAlerts, classifyTenantHealth.

  Step 2: buildSnapshotPayload(summary)
          Pure serialization — extracts all storable fields from the summary.
          No I/O.

  Step 3: insertGovernanceSnapshot(payload)
          Executes INSERT INTO governance_snapshots ... RETURNING.
          Returns the hydrated StoredSnapshot with its assigned ID.

  Step 4: Emit governance_snapshot_captured structured log event
          Fields: workspaceId, snapshotId, snapshotSeverity, stuckCount,
                  stormSeverity, alertCodes, capturedAt

  Step 5: querySnapshotsByRange(workspaceId, "7d") → detectChronicAlerts()
          If any chronic alerts found → emit governance_chronic_alert_detected
          Non-fatal: failure here does not fail the capture.

  Returns: CaptureResult
    { snapshotId, workspaceId, capturedAt, severity, stuckCount,
      stormSeverity, alertCodes }

Failure handling:
  Any error in Step 1 or Step 3 → emit governance_snapshot_capture_failed
  structured log event with error message, then re-throw to the route handler.

Time ranges:
  "1h"  →   1 hour   →   60 snapshots at 1-min, 12 at 5-min
  "24h" →  24 hours  → 1440 snapshots at 1-min, 288 at 5-min
  "7d"  → 168 hours  → 2016 snapshots at 5-min
  "30d" → 720 hours  → 8640 snapshots at 5-min

Query semantics:
  • WHERE workspace_id = ? AND captured_at >= (now - range_hours) AND
          captured_at <= now
  • ORDER BY captured_at ASC
  • Future snapshots (captured_at > now) are always excluded

Trend series extracted (pure, from StoredSnapshot[]):
  severityHistory      → [{capturedAt, severity}]
                          Allows drawing a severity timeline / heat bar
  errorRateTrend       → [{capturedAt, value}]  (0–1 range)
                          Sparkline: error rate over time
  approvalBacklogTrend → [{capturedAt, value}]
                          Backlog pressure trend for approval queues
  delayBacklogTrend    → [{capturedAt, value}]
                          Backlog pressure trend for delayed executions
  stuckCountTrend      → [{capturedAt, value}]
                          Stuck execution count history

API endpoint: GET /governance/snapshots?range=24h
Response shape (GovernanceSnapshotTrend):
  { range, capturedAt, snapshotCount, firstAt, lastAt,
    severityHistory[], errorRateTrend[], approvalBacklogTrend[],
    delayBacklogTrend[], stuckCountTrend[] }

Empty result is valid (snapshotCount=0, empty arrays) — not an error.

ALERT FREQUENCY INTELLIGENCE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

computeAlertFrequency(snapshots[]) → AlertFrequencyEntry[]

  For each distinct GOV-* code found in any snapshot's alert_codes array:
    code           — the GOV-NN_* alert identifier
    count          — number of snapshots containing this code
    totalSnapshots — total snapshots in the query window
    frequencyPct   — (count / total) × 100, rounded to 2dp
    firstSeenAt    — earliest capturedAt where code appeared
    lastSeenAt     — latest capturedAt where code appeared
    isChronic      — frequencyPct >= CHRONIC_THRESHOLD_PCT (50%)

  Sorted: count descending, code alphabetical on ties (deterministic).

detectChronicAlerts(snapshots[], thresholdPct=50) → AlertFrequencyEntry[]
  Filters to only isChronic=true entries.

Chronic patterns detected without ML:
  • Persistent degradation: critical code appears in >50% of 7d snapshots
  • Recurring storms: GOV-storm codes appear repeatedly over 24h
  • Chronic backlog: GOV-approval or GOV-delay codes at >50% frequency
  • Systemic error concentration: GOV-error codes in majority of snapshots

API endpoint: GET /governance/snapshots/chronic-alerts?range=7d
Response shape (GovernanceChronicAlerts):
  { range, capturedAt, snapshotCount, chronicCount, items[] }

REMAINING HISTORICAL ANALYTICS GAPS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. AUTOMATED CAPTURE (not in P7-A):
   captureGovernanceSnapshot() must currently be called explicitly via
   POST /governance/snapshots/capture.  A background scheduler (cron-style
   or event-driven) is needed to capture automatically every 5 minutes.
   Without it, historical data only accumulates when an admin triggers manually.

2. SNAPSHOT PRUNING (not in P7-A):
   No pruning job exists.  The governance_snapshots table will grow
   indefinitely.  The retention tiers (30d raw → 90d hourly → 365d daily)
   are documented but not enforced.

3. PER-WORKSPACE ROLLUP (not in P7-A):
   No hourly/daily aggregation table exists yet.  For >30d trend queries,
   reading thousands of raw rows is inefficient.  A future rollup table
   (governance_snapshot_rollups) should aggregate by hour and day.

4. TREND COMPARISON (not in P7-A):
   The API does not support period-over-period comparisons (e.g., "this
   week vs last week").  The current range parameter is a single window only.

5. WORKSPACE-LEVEL RETENTION SETTINGS (not in P7-A):
   All workspaces share the same recommended retention policy.  Large
   enterprise tenants may want longer raw retention.  A future
   workspace_settings column could override the platform default.

6. CROSS-WORKSPACE TREND (super_admin only — not in P7-A):
   No endpoint exists for super_admin to see trend data across all
   workspaces simultaneously (e.g., "which workspaces are chronic-critical?").
