2. RELIABILITY HISTORY ARCHITECTURE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

2.1  Two-Layer Model

  Layer A — DB persistence (runtime, async)
    reliability_domain_snapshots  — append-only, never updated
    reliability_incidents         — lifecycle-mutable (status transitions only)

  Layer B — Pure engine (deterministic, sync, no DB)
    artifacts/api-server/src/lib/workflows/reliability-history.ts
    Handles: transition detection, incident reconstruction, SLO evaluation.
    No DB access, no async, no runtime mutations.

2.2  Capture Flow

  POST /platform/reliability/capture
    1. Fetch live platform partitions (P9-B) + active policies (P9-E)
    2. Evaluate P10-A FailureContainment per workspace
    3. Build ReliabilityDomainSnapshot value objects (pure, deterministic)
    4. Bulk-insert snapshots into DB (append-only)
    5. Load previous snapshot per workspace (DISTINCT ON query)
    6. Run trackReliabilityTransition(prev, current) per workspace
    7. Upsert reliability_incidents per workspace based on current severity
    8. Emit 4 observability events per snapshot
    9. Return { captureId, snapshotCount, overview }

  captureId links all workspace snapshots from one capture call:
    Format: "cap:<ms>-<seq>"
  snapshotId is unique per workspace per capture:
    Format: "snap:<ms>-<workspaceId>"
  incidentId is unique per workspace per incident:
    Format: "inc:<workspaceId>-<ms>-<seq>"

5. INCIDENT TIMELINE SEMANTICS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

buildIncidentTimelines(snapshots[]) → IncidentTimeline[]

Algorithm:
  1. Group snapshots by workspaceId; sort each group by capturedAt ASC.
  2. Walk each workspace's sorted snapshot sequence:
     a. No open incident + severity >= "severely_degraded" → open new incident.
     b. Open incident + severity >= "severely_degraded"    → update, stay "active".
     c. Open incident + severity == "degraded"             → status = "recovering".
     d. Open incident + severity == "healthy"              → status = "resolved",
        set resolvedAt, compute durationMinutes.
     e. Re-escalation from "recovering"                    → status = "active".
  3. All incidents open at end of snapshot sequence → status = "active".
  4. Results sorted by startedAt DESC (most-recent first).

IncidentTimeline fields:
  incidentId          — unique ("inc:<workspaceId>-<ms>-<seq>")
  workspaceId         — isolated per workspace
  startedAt           — ISO timestamp of first incident-level snapshot
  lastObservedAt      — ISO timestamp of most recent snapshot in incident
  resolvedAt          — ISO timestamp of resolution; null if still open
  highestSeverity     — worst DegradationStatus across all incident snapshots
  peakPropagationRisk — worst FailurePropagationRisk across incident snapshots
  incidentStatus      — "active" | "recovering" | "resolved"
  snapshots           — ordered sequence of snapshots in this incident
  transitions         — all ReliabilityTransitions within this incident
  escalationMoments   — ISO timestamps when severity got worse
  recoveryMoments     — ISO timestamps when severity improved
  advisoryStormCount  — cumulative advisory storm events in incident
  durationMinutes     — null if still active; minutes from start to resolution

6. TRANSITION TRACKING MODEL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

trackReliabilityTransition(prev, current) → ReliabilityTransition | null

Duplicate spam prevention:
  Returns null when:
    prev.degradationStatus === current.degradationStatus
    AND prev.propagationRisk === current.propagationRisk
  → Stable workspaces emit zero transitions per capture cycle.

TransitionType classification:
  DEGRADATION_INDEX[current] > DEGRADATION_INDEX[prev] → "escalation"
  DEGRADATION_INDEX[current] < DEGRADATION_INDEX[prev] → "recovery"
  Severity equal, propagation changed                   → "lateral"

DEGRADATION_INDEX ordering (ascending severity):
  healthy (0) < degraded (1) < severely_degraded (2) < containment_risk (3) < critical (4)

PROPAGATION_INDEX ordering:
  isolated (0) < bounded (1) < spreading (2) < cascading (3)

ReliabilityTransition fields:
  transitionId       — "trans:<workspaceId>-<ms>-<seq>"
  workspaceId        — always same workspace (cross-workspace → null defensively)
  fromDegradation    — previous DegradationStatus
  toDegradation      — current DegradationStatus
  fromPropagation    — previous FailurePropagationRisk
  toPropagation      — current FailurePropagationRisk
  transitionType     — "escalation" | "recovery" | "lateral"
  isDegradationChange — boolean
  isPropagationChange — boolean
  fromSnapshotId     — previous snapshotId
  toSnapshotId       — current snapshotId
  detectedAt         — ISO timestamp (current snapshot's capturedAt)

7. OPERATIONAL SLO MODEL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Four built-in platform SLOs:

SLO 1 — slo:healthy-workspace-ratio
  Metric:    healthy_workspace_ratio
  Target:    ≥ 0.80  (80% of workspaces healthy or degraded per capture)
  Window:    24h
  Condition: below_threshold
  Breach:    avg healthy ratio across captures in window < 0.80

SLO 2 — slo:critical-workspace-count
  Metric:    critical_workspace_count
  Target:    = 0  (zero critical workspaces)
  Window:    1h
  Condition: above_threshold
  Breach:    any capture in window has criticalCount > 0

SLO 3 — slo:advisory-storm-frequency
  Metric:    advisory_storm_count
  Target:    ≤ 2 storm events per 24h window
  Window:    24h
  Condition: above_threshold
  Breach:    total advisoryStormDetected=true snapshots in window > 2

SLO 4 — slo:cascading-risk-persistence
  Metric:    max_consecutive_cascading_captures
  Target:    ≤ 3 consecutive cascading captures for any workspace
  Window:    6h
  Condition: above_threshold
  Breach:    any workspace has cascading propagationRisk for > 3 consecutive captures

SLO Status classification:
  breached  — metric crosses threshold
  at_risk   — metric within 20% margin of threshold (early warning)
  compliant — metric comfortably within bounds

No-data handling:
  Zero snapshots in window → compliant (conservative, no data = no evidence of breach).

OperationalSLOReport fields:
  evaluatedAt    — ISO timestamp
  windowHours    — longest evaluation window across all SLOs (24h)
  captureCount   — distinct capture groups in window
  totalSLOs      — always 4
  compliantCount — SLOs in compliant state
  atRiskCount    — SLOs in at_risk state
  breachedCount  — SLOs in breached state
  overallStatus  — worst across all 4 evaluations
  sloEvaluations — per-SLO detail including currentValue, lastBreachAt, notes

9. PERSISTENCE GUARANTEES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

APPEND-ONLY (snapshots):
  reliability_domain_snapshots is INSERT-ONLY.
  No UPDATE, no DELETE (except cascade on workspace deletion).
  Historical audit integrity is preserved unconditionally.

LIFECYCLE-MUTABLE (incidents):
  reliability_incidents is updated only for status transitions:
    lastObservedAt, incidentStatus, highestSeverity, resolvedAt,
    advisoryStormCount, snapshotCount.
  No row is deleted outside of workspace cascade.
  One active/recovering incident per workspace at a time enforced in application.

FAIL-CLOSED:
  Ambiguous propagationRisk or degradationStatus values → treated as worst-case.
  Empty snapshot window → SLO status = compliant (conservative; no false breaches).

DETERMINISTIC:
  Same snapshot inputs → same transition outputs.
  Same snapshot inputs → same incident timelines.
  Same snapshot inputs → same SLO report.

Four structured events emitted per-capture:

A) reliability_snapshot_persisted
   Fields: snapshotId, captureId, workspaceId, degradationStatus,
           propagationRisk, action
   Fired:  after each snapshot is inserted into DB

B) reliability_transition_detected
   Fields: workspaceId, transitionType, fromDegradation, toDegradation, action
   Fired:  inside trackReliabilityTransition() when a real change is detected
   Suppressed: when transition is null (stable workspace)

C) incident_timeline_updated
   Fields: workspaceId, incidentId, incidentStatus, highestSeverity, action
   Fired:  when an incident opens, transitions to recovering, or resolves

D) operational_slo_breach_detected
   Fields: sloId, metricName, currentValue, targetThreshold, status, action
   Fired:  inside evaluateSLO() for every SLO evaluation
   action: "slo_breach_detected" when breached, "slo_evaluated" otherwise

All events use req.log / logger.info with structured JSON — no console.log.

11. GOVERNANCE INTEGRATION POINTS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Reliability history integrates with existing governance infrastructure:

  P10-A Reliability Domains:
    POST /capture runs full P10-A evaluation as its first step.
    captureId links live domain evaluation to persisted snapshot batch.

  P9-E Fairness Policies:
    activePolicyCount + enforcementStatus included in each snapshot's
    containment context via FailureContainmentInput.

  P9-B Workload Partitions:
    fetchPlatformPartitions() provides live pressureScore, containmentStatus,
    advisoryPressureLevel, backlogDepth, activeExecutionCount per workspace.

  Super Admin Platform Overview:
    GET /platform/stats — can be augmented with historical reliability count.
    GET /platform/activity — incident open/close events are observable via logs.

  Scheduler Oversight History:
    Snapshot history correlates scheduler weight changes (P9-E policies) with
    degradation status transitions — enabling post-hoc fairness analysis.

No automated remediation. No auto-recovery. Historical governance layer only.

13. REMAINING RESILIENCE GAPS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

P10-B intentionally excludes automated recovery and remediation.
The following gaps remain by design for future phases:

  1. RECOVERY RECOMMENDATIONS
     Captured incidents contain no operator guidance.
     Status: intentionally omitted per P10-B safety rules.
     Gap: no suggested remediation or weight-adjustment recommendation.

  2. SCHEDULED CAPTURE AUTOMATION
     POST /platform/reliability/capture must be called manually by a super admin.
     No periodic/scheduled capture mechanism exists.
     Gap: reliability gaps between captures may go undetected.

  3. SLO BURN-RATE ALERTING
     SLO evaluation is point-in-time (per-request).
     No persistent breach-tracking, SLO burn-rate windows, or alert dispatch.
     Gap: short breaches that resolve before next evaluation are invisible.

  4. SNAPSHOT RETENTION POLICY
     reliability_domain_snapshots is append-only with no TTL or pruning.
     Gap: unbounded table growth over time; no archival strategy defined.

  5. CROSS-WORKSPACE BLAST RADIUS CORRELATION
     Each workspace incident is tracked independently.
     Gap: cascading failures that span multiple workspaces simultaneously
          are not correlated into a "platform-level incident" object.

  6. INCIDENT NOTIFICATION DISPATCH
     Incident open/close events are logged but not dispatched to any
     notification channel (email, webhook, in-app notification).
     Gap: operators must poll GET /incidents to discover active incidents.

  7. RECOVERY VELOCITY MEASUREMENT
     durationMinutes is computed but not trended over time.
     Gap: no MTTR (mean time to recovery) or MTTD (detection) metrics.

  8. FRONTEND GOVERNANCE SURFACE
     All P10-B routes are backend-only.
     Gap: no UI in the Super Admin panel for reliability history,
          incident console, or SLO dashboard.
