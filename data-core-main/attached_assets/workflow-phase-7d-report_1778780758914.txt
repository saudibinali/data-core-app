OBJECTIVE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Convert the internal historical analytics infrastructure built in P7-A/B/C into
a safe, read-only API surface that enables:

  • Trend dashboards consuming governance health over time
  • Historical governance analysis (error rate, backlog, storm trends)
  • Operational history visualization (severity timeline, alerts storm)
  • Long-term tenant health insights (up to 365 days)

Prior to P7-D:
  • querySnapshotsByRange, queryRollupsByRange, selectQueryLayer existed
    internally in governance-history.ts and governance-rollup.ts but were
    not reachable via HTTP.
  • The only historical APIs were GET /governance/snapshots (range: 1h–30d raw)
    and GET /governance/snapshots/chronic-alerts (same raw-only range).
  • No API consumer could access rollup tiers (hourly/daily) for long-term data.
  • No stable DTO contract existed for trend chart consumption.

After P7-D:
  • Four new GET endpoints expose the cascade through a stable, versioned API.
  • All endpoints automatically route to raw / hourly / daily storage tier
    based on the requested range — transparent to the consumer.
  • Consistent TrendEnvelope<T> response shape across all tiers and endpoints.
  • MAX_TREND_POINTS cap prevents payload bloat for dense raw queries.
  • Full observability (4 structured log events per request).
  • 111 new pure model tests; 0 typecheck errors.

Purpose: Pure model and serialization layer for the four trend API surfaces.
Contains zero DB access — all query functions are imported from governance-

── Query cascade routing ─────────────────────────────────────────────────────

  selectTrendQueryLayer(rangeDays: number): QueryLayer

  Range parameter   │ Days    │ Storage tier │ Resolution
  ──────────────────┼─────────┼──────────────┼──────────────────────────
  "1h"              │ 0.042   │ raw          │ 5-min (1 point per snap)
  "24h"             │ 1       │ raw          │ 5-min
  "7d"              │ 7       │ raw          │ 5-min
  "30d"             │ 30      │ raw          │ 5-min (up to 8,640 pts)
  "90d"             │ 90      │ hourly       │ 1-hour (90 pts max)
  "180d"            │ 180     │ daily        │ 1-day  (180 pts max)
  "365d"            │ 365     │ daily        │ 1-day  (365 pts max)

  Cascade boundary alignment:
    RAW / HOURLY boundary: rangeDays ≤ 30 → raw, > 30 → hourly
    HOURLY / DAILY boundary: rangeDays ≤ 90 → hourly, > 90 → daily

  These boundaries are derived from the same constants used by the rollup
  scheduler (RETENTION_RAW_DAYS = 30, ROLLUP_HOURLY_RETENTION_DAYS = 90).
  If retention constants change, the cascade boundaries move automatically.

── Request lifecycle ─────────────────────────────────────────────────────────

  For every trend request:

  1. requireAuth + requireWorkspaceAdmin → workspaceId attached to request
  2. Parse ?range= query param (default "30d")
  3. validateTrendRange(rawRange, now):
     a. Check range is in EXTENDED_TREND_RANGES → 400 if not
     b. Compute since = now - hours, until = now
     c. Compute rangeDays, layer → return TrendQueryContext
  4. Emit governance_trend_api_requested (INFO)
  5. Dispatch by layer:
     a. "raw"    → querySnapshotsInBucket(workspaceId, since, until, db)
     b. "hourly" → queryRollupsByRange(workspaceId, "hourly", since, until, db)
     c. "daily"  → queryRollupsByRange(workspaceId, "daily",  since, until, db)
  6. Serialize raw rows to DTO array (pure, type-specific serializer)
  7. truncateTrendPoints(points) → apply MAX_TREND_POINTS cap
  8. If truncated → emit governance_trend_payload_truncated (WARN)
  9. buildTrendEnvelope(ctx, points, truncated) → consistent response shape
  10. Emit governance_trend_api_resolved (INFO)
  11. res.json(envelope)

── Safety invariants ─────────────────────────────────────────────────────────

  INVARIANT 1 — Read-only:
    governance-trends.ts exports zero write functions (INSERT/UPDATE/DELETE/capture).
    Route handlers import only read query functions from history/rollup modules.
    Verified: T10 test group checks that no prune*, insert*, delete*, capture*
    symbols are exported from governance-trends.ts.

  INVARIANT 2 — No rollup triggers:
    The four trend endpoints NEVER call buildWorkspaceRollups(), the rollup
    scheduler, or any write path in governance-rollup.ts.
    They only import queryRollupsByRange (SELECT-only).

  INVARIANT 3 — No pruning:
    The four trend endpoints NEVER call pruneSnapshotsOlderThan() or any
    delete operation.

  INVARIANT 4 — No runtime execution effect:
    governance-trends.ts imports ZERO symbols from workflow_executions,
    approval_requests, or any runtime operational table.

  INVARIANT 5 — Workspace isolation:
    workspaceId is always sourced from req.workspaceId (attached by requireAuth
    from the authenticated JWT).  It is NEVER read from query params or body.
    No cross-workspace data can be returned.

  INVARIANT 6 — Tenant data not leaked into response:
    All serializers strip workspaceId, id, schemaVersion, alertCodes,
    alertSummary, indicators, and all other internal DB fields.
    Each DTO contains only the analytics signal fields.
    Verified: T9 test group checks no id, workspaceId, schemaVersion,
    alertCodes fields appear in any trend point.

API ENDPOINTS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

All four endpoints share the same request/response contract structure:

  Method:  GET
  Auth:    requireAuth + requireWorkspaceAdmin
  Scope:   workspace-scoped (no workspaceId in URL/query — inferred from JWT)
  Param:   ?range= (optional, default "30d")
           Enum: "1h" | "24h" | "7d" | "30d" | "90d" | "180d" | "365d"
  Success: 200 with TrendEnvelope<T>
  Errors:  400 invalid range, 403 no workspace / insufficient role

── GET /governance/trends/severity ──────────────────────────────────────────

  operationId: getGovernanceTrendsSeverity
  Response DTO: SeverityTrendResponse (TrendEnvelope<SeverityTrendPoint>)

  Each SeverityTrendPoint:
    timestamp:   ISO 8601 (capturedAt for raw, bucketStart for rollups)
    severity:    "healthy" | "warning" | "degraded" | "critical"
    sourceLayer: "raw" | "hourly" | "daily"

  Semantic by tier:
    raw:    exact workspace severity at snapshot capture time
    hourly: dominantSeverity = worst severity in the hour's snapshots
    daily:  dominantSeverity = worst severity in the day's hourly rollups

  Use case: severity timeline chart showing workspace health degradation
  and recovery patterns over time.

── GET /governance/trends/error-rate ────────────────────────────────────────

  operationId: getGovernanceTrendsErrorRate
  Response DTO: NumericTrendResponse (TrendEnvelope<NumericTrendPoint>)

  Each NumericTrendPoint:
    timestamp:   ISO 8601
    value:       workflowErrorRate [0–1] (raw) or avgErrorRate (rollup)
    sourceLayer: "raw" | "hourly" | "daily"

  Semantic:
    raw:    exact workflowErrorRate at capture time (= failedExecutions / totalExecutions)
    hourly: arithmetic mean of workflowErrorRate across all snapshots in the hour
    daily:  arithmetic mean of avgErrorRate across all hourly rollups in the day

  Use case: error-rate sparkline showing operational degradation trends over
  weeks or months.  Consumers should format value as a percentage (× 100).

── GET /governance/trends/backlogs ──────────────────────────────────────────

  operationId: getGovernanceTrendsBacklogs
  Response DTO: BacklogTrendResponse (TrendEnvelope<BacklogTrendPoint>)

  Each BacklogTrendPoint:
    timestamp:       ISO 8601
    approvalBacklog: approvalBacklogCount (raw, integer) or avgApprovalBacklog (rollup, float)
    delayBacklog:    delayBacklogCount (raw, integer) or avgDelayBacklog (rollup, float)
    stuckCount:      stuckCount (raw, integer) or avgStuckCount (rollup, float)
    sourceLayer:     "raw" | "hourly" | "daily"

  Note: For rollup tiers, all three values are floating-point averages.
  The sourceLayer field signals this precision difference to consumers.

  Use case: stacked area chart showing queue pressure trends across approval,
  delay, and stuck dimensions.  Useful for identifying bottleneck patterns.

── GET /governance/trends/storms ────────────────────────────────────────────

  operationId: getGovernanceTrendsStorms
  Response DTO: StormTrendResponse (TrendEnvelope<StormTrendPoint>)

  Each StormTrendPoint:
    timestamp:        ISO 8601
    stormFrequency:   [0.0–1.0]
    dominantSeverity: "healthy" | "warning" | "degraded" | "critical"
    sourceLayer:      "raw" | "hourly" | "daily"

  stormFrequency semantic:
    raw:    binary — 1.0 if stormSeverity ≠ "none", 0.0 otherwise
            (each raw point IS a storm or IS NOT — no fractionality)
    hourly: fraction of the hour's snapshots with stormSeverity ≠ "none"
            (e.g. 0.25 = 3 out of 12 snapshots in the hour had storms)
    daily:  arithmetic mean of hourly stormFrequency values
            (e.g. 0.4 = on average, 40% of each hour in the day had storms)

  Use case: storm frequency bar chart showing alert storm intensity trends.
  Combined with dominantSeverity, enables "storm calendar" visualization.

QUERY CASCADE INTEGRATION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The P7-D APIs connect to the P7-C rollup infrastructure through two query
functions:

  Raw tier (0–30d):
    querySnapshotsInBucket(workspaceId, since, until, db)
    → SELECT from governance_snapshots WHERE workspaceId = ? AND capturedAt >= since AND capturedAt < until
    → Returns StoredSnapshot[] ordered by capturedAt ASC

  Rollup tiers (31–365d):
    queryRollupsByRange(workspaceId, granularity, since, until, db)
    → SELECT from governance_snapshot_rollups WHERE workspaceId = ? AND granularity = ?
         AND bucketStart >= since AND bucketStart < until
    → Returns StoredRollup[] ordered by bucketStart ASC

  Both functions are already scoped by workspaceId, ensuring tenant isolation
  at the DB query level.

  The route handler determines which function to call using the layer from
  TrendQueryContext:

    if (ctx.layer === "raw")   → querySnapshotsInBucket(...)
    else granularity = ctx.layer as "hourly" | "daily"
         queryRollupsByRange(workspaceId, granularity, ...)

  The response shape (TrendEnvelope<T>) is identical regardless of which
  query function was called.  The `sourceLayer` field tells the consumer
  which tier served the data, enabling them to display appropriate labels
  (e.g. "5-min resolution" vs "hourly average" vs "daily average").

RESPONSE MODELS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

── TrendEnvelope<T> (common wrapper for all four endpoints) ──────────────────

  {
    range:       string;      // range literal e.g. "30d"
    rangeDays:   number;      // numeric days in the query window (e.g. 30)
    sourceLayer: "raw" | "hourly" | "daily";
    pointCount:  number;      // length of points array (≤ MAX_TREND_POINTS)
    truncated:   boolean;     // true when original result > MAX_TREND_POINTS
    points:      T[];         // sorted ascending by timestamp
  }

  Design properties:
    • Pagination-ready: pointCount + truncated signal whether more data exists.
    • Layer-transparent: sourceLayer tells the consumer the resolution context.
    • Stable: same inputs always produce the same output (idempotent serialization).
    • Consumer-safe: no raw DB fields, no internal IDs, no JSONB internals.

── SeverityTrendPoint ────────────────────────────────────────────────────────

  { timestamp: string; severity: string; sourceLayer: QueryLayer }

  Exactly 3 fields. timestamp is always UTC ISO 8601.
  severity is always one of the 4 canonical severity strings.

── NumericTrendPoint ─────────────────────────────────────────────────────────

  { timestamp: string; value: number; sourceLayer: QueryLayer }

  Exactly 3 fields. value is always a float in [0, 1] for error-rate.
  For future numeric trends, the range of value is endpoint-specific.

── BacklogTrendPoint ─────────────────────────────────────────────────────────

  { timestamp: string; approvalBacklog: number; delayBacklog: number;
    stuckCount: number; sourceLayer: QueryLayer }

  Exactly 5 fields. For raw tier, all three counts are exact integers.
  For rollup tiers, all three are floating-point averages.

── StormTrendPoint ────────────────────────────────────────────────────────────

  { timestamp: string; stormFrequency: number; dominantSeverity: string;
    sourceLayer: QueryLayer }

  Exactly 4 fields. stormFrequency is always in [0.0, 1.0].
  dominantSeverity is always one of the 4 canonical severity strings.
  For raw tier, stormFrequency is binary (0.0 or 1.0).
  For rollup tiers, stormFrequency is fractional (proportion of period with storms).

RANGE VALIDATION RULES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Function: validateTrendRange(rawRange: string, now: Date): TrendRangeValidation

Validation is a discriminated union result:
  { ok: true,  context: TrendQueryContext }   — valid, ready to query
  { ok: false, reason: string, statusCode: 400 | 422 } — rejected

Validation rules applied in order:

  RULE 1 — Must be a declared range literal:
    EXTENDED_TREND_RANGES = ["1h", "24h", "7d", "30d", "90d", "180d", "365d"]
    Any other string (including "", "30", "1d", "weekly", "7D") → 400.
    Set-based O(1) check: VALID_EXTENDED_RANGES.has(rawRange).

  RULE 2 — Computed rangeDays must not exceed MAX_TREND_RANGE_DAYS (365):
    Since all declared range literals map to ≤ 365 days, this rule is
    currently unreachable through normal validation but protects against
    future additions of invalid ranges to the enum.

  RULE 3 — until = now is always used:
    This prevents future-window queries by construction.
    since = now - hours is always in the past for any positive hours value.
    There is no "since > now" path reachable through this function.

  RULE 4 — Default fallback in routes:
    If no ?range= query param is provided, the route defaults to "30d"
    (passes rule 1–3 automatically).

Safety properties:
  • No SQL injection risk: range is matched against a hard-coded Set before use.
  • No integer overflow: range hours are small constants (max 8,760 hours = 365d).
  • No unbounded historical queries: max 365d, enforced at validation time.
  • No future windows: since = now - hours guarantees since < now.

SERIALIZATION HARDENING STRATEGY
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The serialization layer in governance-trends.ts applies four hardening rules:

── 1. No raw DB row exposure ─────────────────────────────────────────────────

  Every serializer explicitly names only the fields needed for the DTO.
  Fields like `id`, `workspaceId`, `schemaVersion`, `alertCodes`,
  `alertSummary`, `chronicAlertCodes`, `indicators`, `createdAt` are never
  included in the output — not even undefined or null.

  Mechanism: Each serializer is a `.map(s => ({ field1: s.field1, ... }))` with
  an explicit object literal.  No spread operators are used that could
  accidentally forward internal DB fields.

── 2. Stable, deterministic ordering ────────────────────────────────────────

  All serializers preserve the input array order.  Callers (DB query functions)
  return data in capturedAt ASC / bucketStart ASC order.  The serializers
  never re-sort — they trust the ascending order from the DB layer.

  This ensures: same DB state → same API response → deterministic chart
  rendering on the frontend.  Re-running the same query always produces the
  same output.

── 3. Pagination-ready (truncation first, not sorting) ─────────────────────

  truncateTrendPoints() takes the FIRST MAX_TREND_POINTS items (oldest → newest).
  This preserves chronological order and ensures truncated payloads always
  contain the oldest available data, not a random subset.

  For chart use: consumers can detect `truncated=true` and optionally
  narrow the range (e.g. switch from "30d" to "7d") to get finer resolution
  within the available data.

── 4. No JSONB internals exposed ────────────────────────────────────────────

  Raw snapshots store `alertCodes` and `alertSummary` as JSONB.
  Rollup rows store `chronicAlertCodes` as JSONB.
  None of these JSONB columns appear in any trend DTO.

  stormFrequency for raw tier is computed from the `stormSeverity` string
  column — not from any JSONB field.

── 5. sourceLayer as consumer trust signal ───────────────────────────────────

  The sourceLayer field on every data point signals to consumers what
  precision they should expect:

    "raw"    → 5-min snapshot → exact integer counts, binary storm signal
    "hourly" → 1-hour average → floating-point averages, fractional storm
    "daily"  → 1-day average  → floating-point averages, fractional storm

  This prevents consumers from treating float averages as exact counts.

OBSERVABILITY ADDITIONS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Four structured log events per request, all carrying consistent fields.

A) governance_trend_api_requested
   Level:  INFO
   When:   Range validated successfully, DB query about to begin
   Fields: workspaceId, actorId, trendType, range, rangeDays, sourceLayer, action
   Purpose: Know what was requested before touching DB

B) governance_trend_api_resolved
   Level:  INFO
   When:   Response sent successfully
   Fields: workspaceId, actorId, trendType, range, rangeDays, sourceLayer,
           layerLabel (human-readable tier name), resultCount, truncated, action
   Purpose: Track response sizes and cascade routing outcomes for capacity planning

C) governance_trend_query_rejected
   Level:  WARN
   When:   Range validation failed → 400 response

1. NO FRONTEND TREND CHARTS (not in P7-D):
   The four new API endpoints are ready but no React chart components consume
   them.  A historical analytics dashboard page would need:
   • Recharts LineChart / AreaChart components for severity, error-rate, backlogs
   • BarChart for storm frequency
   • Range selector UI (1h / 24h / 7d / 30d / 90d / 180d / 365d)
   • sourceLayer-aware axis labels ("5-min", "hourly avg", "daily avg")
   • truncated=true warning banner ("Showing oldest 1000 points — narrow your range")

2. NO PAGINATION CURSOR (not in P7-D):
   The truncation mechanism is a safety cap, not a pagination interface.
   When truncated=true, there is no cursor token to fetch the next page.
   A future cursor-based pagination (e.g. ?before=<timestamp>&limit=500)
   would allow clients to page through high-density raw windows.

3. ROLLUP RETENTION PRUNING NOT YET IMPLEMENTED (not in P7-D):
   Hourly rollup rows older than 90 days and daily rollup rows older than 365 days
   are never pruned.  governance_snapshot_rollups grows indefinitely.
   A future pruning step needs to be gated on: prune hourly only after daily covers
   the same window; prune daily after 365-day threshold.

4. FIRST-RUN BACKFILL GAP (not in P7-D):
   The P7-C rollup scheduler only covers a ±2d overlap window.  On first
   deployment with existing raw history, 28 days of data before the overlap
   zone have no hourly rollups.  A one-time backfill job covering [0d, 30d ago]
   would fill this gap.  The APIs would then serve correct hourly data from day 1.

5. NO SUPER-ADMIN CROSS-WORKSPACE TREND (not in P7-D):
   Super admins cannot query trend data across all workspaces simultaneously.
   A platform-wide endpoint (e.g. GET /platform/trends/severity?range=90d)
   would aggregate across all workspace rollups.  This requires careful
   isolation to prevent data leakage.

6. CHRONIC ALERT TRENDS NOT SURFACED (not in P7-D):
   chronicAlertCodes on rollup rows are stored but not exposed by any
   P7-D endpoint.  A fifth endpoint — GET /governance/trends/chronic-alerts —
   could return the historically chronic codes per time bucket using the rollup
   tier.  Currently only GET /governance/snapshots/chronic-alerts (raw tier,
   ≤30d) exists.

7. NO RANGE DOWNSAMPLING FOR DENSE RAW WINDOWS (not in P7-D):
   A 30d raw query can return up to 8,640 points before truncation.  After
   truncation to 1,000 points, the oldest 7,640 points are silently discarded.
   Client-side downsampling (e.g. LTTB — Largest Triangle Three Buckets) would
   provide a better representation of the full window within the point limit.

8. ROLLUP SCHEDULER STATUS ENDPOINT (not in P7-D):
   No GET endpoint exposes rollup pipeline health (last run time, built counts,
   success/failure per workspace) to the super-admin dashboard.
