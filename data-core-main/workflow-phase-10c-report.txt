2. RECOVERY ADVISORY ARCHITECTURE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

2.1  Layer Design

  This phase adds a pure advisory intelligence layer on top of P10-B persistence:

  Layer C — Pure Advisory Engine (read-only, sync, no DB)
    artifacts/api-server/src/lib/workflows/recovery-recommendations.ts

    Reads:
      - RecommendationContext (derived from DB incident row + snapshot query)
      - WorkspaceIncidentHistory (derived from prior resolved incidents in DB)
      - IncidentSummary[] (derived from DB incident rows)

    Produces:
      - RecoveryRecommendation[]  (per-incident advisory guidance)
      - WorkspaceTrend            (per-workspace historical pattern)
      - ReliabilityTrendReport    (platform-wide trend summary)

  No new DB tables. All advisory outputs are computed on-demand, never persisted.
  Each API request produces fresh deterministic output from current DB state.

2.2  Data Flow

  GET /platform/reliability/recommendations:
    1. Fetch open/recovering incidents from reliability_incidents
    2. For each, fetch prior resolved incidents (same workspace)
    3. Fetch incident-era snapshot blast radius scores
    4. Build RecommendationContext + WorkspaceIncidentHistory (pure)
    5. Call generateRecoveryRecommendations() (pure, no DB)
    6. Return all recommendations flat-sorted by severity

  GET /platform/reliability/recommendations/:incidentId:
    1. Fetch target incident by incidentId
    2. Fetch all prior incidents for same workspace (including resolved)
    3. Fetch snapshot blast radius scores for incident window
    4. Generate recommendations + workspace trend summary
    5. Return { recommendations, incident, workspaceTrend }

  GET /platform/reliability/trends:
    1. Fetch all incidents within windowDays from reliability_incidents
    2. Map to IncidentSummary[] (pure value objects)
    3. Call buildPlatformTrendReport() (pure, groups by workspace)
    4. Return { trends: ReliabilityTrendReport }

3. RECOMMENDATION SEMANTICS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

3.1  RecoveryRecommendation Fields

  recommendationId    — "rec:<workspaceId>-<ms>-<seq>" (unique per call)
  workspaceId         — workspace this recommendation applies to
  incidentId          — incident that triggered this recommendation
  recommendationType  — one of 8 types (see 3.2)
  severity            — low | moderate | high | critical
  rationale           — human-readable explanation of why generated
  suggestedActions    — ordered list of specific operator investigation steps
  confidenceLevel     — low | moderate | high | strong (see 3.3)
  relatedSignals      — evidence codes that produced this recommendation
  generatedAt         — ISO 8601 timestamp

3.2  Eight Recommendation Types

  R1: monitor_closely
    Condition:  Always generated for active/recovering incidents.
    Purpose:    Ensures operators maintain observation cadence.
    Severity:   Maps from highestSeverity.
    Actions:    Continue monitoring; verify no new subsystems affected;
                do not reduce cadence until 3 consecutive healthy captures.

  R2: investigate_scheduler_pressure
    Condition:  maxBlastRadiusScore > 50.
    Severity:   high (>80), moderate (51–80).
    Purpose:    Signals elevated workload pressure on the scheduler.
    Actions:    Review execution count, backlog depth, capacity, scheduler weight.

  R3: isolate_noisy_tenant
    Condition:  propagationRisk in {spreading, cascading}
                AND highestSeverity >= "severely_degraded".
    Severity:   critical (cascading), high (spreading).
    Purpose:    Indicates possible cross-workspace contamination.
    Actions:    Investigate resource burst, apply fairness policy,
                verify isolation mechanisms.

  R4: review_fairness_policies
    Condition:  escalationCount > 0
                AND highestSeverity in {containment_risk, critical}.
    Severity:   high.
    Purpose:    Policy misconfiguration may be amplifying degradation.
    Actions:    Review weights, enforcement mode, expiry timestamps,
                compare weight before/after onset.

  R5: investigate_advisory_storm
    Condition:  advisoryStormCount > 0.
    Severity:   high (>2), moderate (1–2).
    Purpose:    Advisory storms indicate simultaneous multi-signal pressure.
    Actions:    Identify trigger patterns, review pressure composition,
                check correlated storms, recalibrate thresholds.

  R6: containment_boundary_review
    Condition:  peakPropagationRisk == "cascading"
                OR highestSeverity in {containment_risk, critical}.
    Severity:   critical (both cascading+critical), high (otherwise).
    Purpose:    Containment boundaries were stressed or breached.
    Actions:    Review boundary thresholds, investigate blast radius expansion,
                verify isolation, document for post-incident analysis.

  R7: escalation_watch
    Condition:  escalationCount >= 2.
    Severity:   high.
    Purpose:    Multiple escalations indicate workspace instability pattern.
    Actions:    Monitor next 5 captures, review escalation conditions,
                identify recurring triggers, consider pre-emptive policy.

  R8: recovery_stability_watch
    Condition:  incidentStatus == "recovering".
    Severity:   moderate.
    Purpose:    Guard against premature incident clearance.
    Actions:    Verify root cause addressed, maintain monitoring 3+ captures,
                watch for re-escalation, document recovery evidence.

3.3  Output Ordering

  Recommendations are sorted by severity: critical → high → moderate → low.
  At most one recommendation per type per call (no duplicates).
  Maximum 8 recommendations per incident.

4. TREND-ANALYSIS MODEL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

4.1  WorkspaceTrend Fields

  workspaceId             — workspace identifier
  totalIncidents          — all incidents in analysis window
  openIncidents           — active + recovering incidents
  mttrMinutes             — mean time to recovery (resolved incidents only)
  avgDurationMinutes      — same as mttrMinutes (alias for clarity)
  escalationFrequency     — none | low | moderate | high | chronic
  advisoryStormRecurrence — true when >1 incident had advisory storms
  cascadingRiskRecurrence — true when >1 incident had cascading propagation
  isChronicallyDegraded   — true when totalIncidents >= 3 in window
  lastIncidentAt          — ISO timestamp of most recent incident start
  incidentCount30d        — incidents started in last 30 days
  recurrenceInterval      — none | rare | occasional | frequent | chronic

4.2  escalationFrequency Classification

  none     — no escalation events across all incidents
  low      — avg escalations per incident < 1
  moderate — avg escalations per incident 1–2
  high     — avg escalations per incident > 2
  chronic  — totalIncidents >= 3 AND avg > 2

4.3  recurrenceInterval Classification

  none       — 0 incidents
  rare       — 1 incident
  occasional — 2 incidents
  frequent   — 3–4 incidents
  chronic    — 5+ incidents

4.4  PlatformTrendSummary Fields

  totalActiveIncidents     — workspaces with at least one open incident
  totalOpenIncidents       — total open incident count across platform
  chronicallyDegradedCount — workspaces with isChronicallyDegraded=true
  highEscalationCount      — workspaces with high/chronic escalation frequency
  cascadingRecurrenceCount — workspaces with cascadingRiskRecurrence=true
  avgMttrMinutes           — platform mean time to recovery
  platformHealthTrend      — improving | stable | degrading

4.5  platformHealthTrend Algorithm

  Window split in half (recent half vs earlier half):
    improving — recentHalf < earlierHalf × 0.80  (20% fewer incidents)
    degrading — recentHalf > earlierHalf × 1.20  (20% more incidents)
    stable    — change within ±20%

  Pure: no ML, no prediction, no extrapolation.

5. CONFIDENCE-CLASSIFICATION MODEL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Confidence is computed by computeConfidence(signalCount, priorMatchCount, history).

  Inputs:
    signalCount     — number of independent signals supporting this recommendation
    priorMatchCount — prior incidents where the same pattern was observed
    history         — WorkspaceIncidentHistory (total prior, recurrenceInterval)

  Rules (evaluated in descending priority):
    strong   — recurrenceInterval=chronic OR priorMatchCount >= 3
    high     — priorMatchCount >= 2
               OR (signalCount >= 3 AND totalPriorIncidents >= 1)
    moderate — priorMatchCount >= 1 OR signalCount >= 2
    low      — fallback (first occurrence, single signal)

  Ceiling invariant:
    If totalPriorIncidents == 0 → max confidence = "moderate".
    Insufficient history prevents false high-confidence recommendations.

  Confidence communicates CERTAINTY of pattern, not urgency.
  Severity communicates urgency. Both are independent.

6. GOVERNANCE API LAYER
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

All routes:
  - Require Authorization: Bearer <jwt> with role = super_admin
  - Protected by requireAuth + requireSuperAdmin middleware chain
  - Deterministic ordering: severity desc, then startedAt desc
  - Audit-safe serialization: all outputs are pure JSON, no functions
  - Read-only: no DB writes, no state mutations, no scheduler changes

GET /platform/reliability/recommendations
  Auth:    super_admin
  Params:  workspaceId? (filter), limit (1–50, default 20)
  Action:  Generates recommendations for all active/recovering incidents.
  Returns: { recommendations: RecoveryRecommendation[], incidentCount: number }

GET /platform/reliability/recommendations/:incidentId
  Auth:    super_admin
  Params:  incidentId (path parameter — text ID format "inc:<ws>-<ms>-<seq>")
  Action:  Full recommendation set for one incident with workspace trend context.
  Returns: {
    recommendations: RecoveryRecommendation[],
    incident: { incidentId, workspaceId, startedAt, lastObservedAt, resolvedAt,
                highestSeverity, peakPropagationRisk, incidentStatus,
                snapshotCount, advisoryStormCount },
    workspaceTrend: { totalIncidents, openIncidents, mttrMinutes,
                      advisoryStormRecurrence, cascadingRiskRecurrence,
                      isChronicallyDegraded, recurrenceInterval }
  }
  Returns 404 if incidentId not found.

GET /platform/reliability/trends
  Auth:    super_admin
  Params:  windowDays (1–90, default 30)
  Action:  Platform-wide reliability trend report from incident history.
  Returns: { trends: ReliabilityTrendReport }
  Note:    Reads up to 1000 incidents within the window. Groups by workspace.

7. OBSERVABILITY ADDITIONS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Four structured events emitted by the advisory engine:

A) recovery_recommendation_generated
   Fields: workspaceId, incidentId, recommendationType, confidenceLevel,
           propagationRisk, action
   Fired:  inside generateRecoveryRecommendations() for monitor_closely rule.
           Selective — emitted for the primary advisory event per incident.

B) reliability_trend_detected
   Fields: workspaceId, incidentId, patternType, occurrenceCount, action
   Fired:  inside buildWorkspaceTrend() when chronic_degradation is classified,
           and inside investigate_advisory_storm rule when storm recurrence found.
   patternType values: "chronic_degradation", "advisory_storm_recurrence",
                       "trend_computed"

C) incident_recurrence_detected
   Fields: workspaceId, incidentId, patternType, priorOccurrences, action
   Fired:  inside containment_boundary_review rule when priorWithCascadingRisk > 0.
   patternType: "cascading_risk_recurrence"

D) recovery_pattern_classified
   Fields: workspaceId, incidentId, patternType, escalationCount, action
   Fired:  inside escalation_watch rule when escalationCount >= 2.
   patternType: "repeated_escalation"

All events: logger.info() with structured JSON — no console.log.

10. REMAINING RESILIENCE GAPS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

P10-C is intentionally advisory-only. The following gaps remain by design:

  1. OPERATOR FEEDBACK LOOP
     No mechanism for operators to acknowledge, dismiss, or act on a
     recommendation. Recommendations cannot be marked as "addressed."
     Gap: no audit trail of which recommendations were followed.

  2. RECOMMENDATION PERSISTENCE
     Recommendations are computed on-demand and not stored.
     Gap: no history of which recommendations were generated for which
          incidents at what time. Each API call produces a fresh set.

  3. ESCALATION COUNT SOURCED FROM DB
     escalationCount is currently set to 0 for all DB-sourced incidents
     (the DB stores advisoryStormCount but not escalationCount separately).
     Gap: R4 (review_fairness_policies) and R7 (escalation_watch) rules
          cannot fire from DB-sourced incidents unless escalationCount
          is enriched from snapshot history.

  4. CROSS-WORKSPACE INCIDENT CORRELATION
     Recommendations are per-workspace in isolation.
     Gap: when multiple workspaces degrade simultaneously (platform-level
          event), no cross-workspace incident correlation recommendation
          is generated.

  5. FRONTEND ADVISORY SURFACE
     All P10-C routes are backend-only.
     Gap: no Recovery Center UI in the Super Admin panel showing active
          recommendations, incident triage queue, or trend dashboard.

  6. NOTIFICATION DISPATCH
     Recommendations are only accessible via polling.
     Gap: no push notification when a new high/critical recommendation
          is generated for an active incident.

  7. RECOMMENDATION DEDUPLICATION ACROSS REQUESTS
     Each GET call to /recommendations recomputes fresh IDs.
     Gap: if the caller fetches repeatedly, recommendation IDs differ,
          preventing idempotent integration with external systems.

  8. TIME-SERIES METRIC STORAGE
     Trend analysis reads incident-level aggregates, not time-series.
     Gap: fine-grained degradation trends (e.g., hourly MTTR, per-capture
          blast radius slope) are not tracked.
