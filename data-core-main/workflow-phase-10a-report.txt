━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

P10-A converts the platform from a governance-aware operational platform into a
reliability-aware enterprise runtime architecture. It adds a deterministic
failure-containment intelligence layer on top of the existing governance stack
(P9-A through P9-F) without adding any self-healing, autoscaling, or autonomous
remediation.

Before P10-A:
  • No formal model of failure boundaries — blast radius is undefined.
  • No classification of cross-subsystem propagation risk.
  • No degradation severity ladder for the governance subsystems.
  • No observability of containment boundary health.
  • Advisory storms (multiple noisy signals) are not recognized as a category.
  • No platform-wide reliability overview combining P9-B, P9-E, and P9-F signals.

After P10-A:
  • ReliabilityDomain — a workspace-scoped unit of failure containment with
    6 monitored subsystems, 5-level degradation status, 4-level propagation
    risk, 4-level containment level, recovery classification, and observability
    health.
  • evaluateFailureContainment() — deterministic engine composing P9-B/E/F
    signals into a complete reliability assessment per workspace.
  • buildPlatformReliabilityOverview() — platform-wide aggregate of all
    workspace reliability domains.
  • 5 containment boundaries per workspace with holding/stressed/at_risk/breached
    status for each protection point.
  • Advisory storm detection — identifies simultaneous multi-signal overloads.
  • BlastRadius model with 0–100 normalized score and 3-tier scope classification.
  • 4 structured observability events.
  • 1 new super-admin GET route: GET /platform/governance/reliability/domains.
  • All guarantees: read-only, no autonomous recovery, fail-closed, deterministic.

Explicitly excluded:
  Distributed failover, active-active replication, Kubernetes HA orchestration,
  multi-region recovery, autoscaling infra, self-healing runtime, automated
  restart, autonomous remediation.

1. FAILURE SURFACE ANALYSIS (pre-P10-A review)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Review of the existing governance stack for failure propagation gaps:

A) WORKFLOW RUNTIME — Unbounded backlog
   P9-B pressureScore accounts for delayedExecutionCount, but the platform has
   no formal model for "what happens if a backlog grows past 50 delayed jobs?"
   A saturated workspace backlog can starve the scheduler for other workspaces
   because the scheduler queue is shared. Blast radius: platform-wide.
   No containment boundary modeled before P10-A.

B) SCHEDULER — Pressure without propagation classification
   P9-B classifies containmentStatus as "contained/at_risk/pressured/saturated"
   and derives a schedulerWeight. But there is no model for:
     • When does scheduler pressure in workspace A affect workspace B?
     • What is the propagation path from "saturated" to cross-workspace failure?
   P10-A adds FailurePropagationRisk and the scheduler_limit boundary.

C) ADVISORY SYSTEM — Advisory storms undetected
   P9-B tracks noisyBehaviorCodes and advisoryPressureLevel independently.
   A workspace can simultaneously have 3 noisy categories + critical advisory
   pressure + 2 active conflicting policies. This compound state (advisory storm)
   amplifies cross-subsystem degradation but was not classified as a unified
   failure mode. P10-A adds detectAdvisoryStorm().

D) POLICY ENGINE — Conflict is catastrophic but uncontained
   P9-E prevents conflicts at creation time, and P9-F handles them defensively
   at resolution time (fail-closed to floor). But there was no model for:
     • What does a policy conflict mean for reliability?
     • Does it breach a containment boundary?
   P10-A models conflictDetected as a "breached" tenant_isolation boundary and
   escalates policy_engine to containment_risk.

E) ENFORCEMENT BRIDGE — Status not surfaced as reliability signal
   P9-F enforcementStatus ("resolved", "stale", "conflict", "floor_applied",
   "no_active_policy") was only visible in the enforcement snapshot.
   P10-A integrates enforcementStatus as an input to the
   enforcement_bridge subsystem classifier, making it a first-class
   reliability signal.

F) TENANT ISOLATION — Containment boundary not formally tracked
   P9-A defines a per-request isolation context, but there was no persistent
   model for "is this workspace's isolation boundary currently under stress?"
   P10-A adds the tenant_isolation ContainmentBoundary with status derived
   from containmentStatus + conflictDetected.

G) OBSERVABILITY — Degrades silently under critical load
   Under critical pressure, the observability layer itself becomes unreliable
   (log backlogs, dropped events). P10-A models this as ObservabilityHealth
   (full/partial/impaired/blind), degrading as overall domain severity increases.
   This gives operators a signal that under "critical" conditions, the events
   themselves may not be trustworthy.

2. RELIABILITY DOMAIN ARCHITECTURE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

A ReliabilityDomain is the fundamental unit of failure containment — one per
workspace per evaluation cycle.

interface ReliabilityDomain {
  domainId:               string;          // "rd:<workspaceId>-<ms>-<seq>"
  domainType:             DomainType;      // dominant (most-degraded) subsystem
  workspaceId:            number;
  affectedSubsystems:     DomainType[];    // all subsystems at "degraded" or worse
  containmentLevel:       ContainmentLevel;  // overall boundary health
  degradationStatus:      DegradationStatus; // worst across all 6 subsystems
  recoveryClassification: RecoveryClassification;
  observabilityHealth:    ObservabilityHealth;
  propagationRisk:        FailurePropagationRisk;
  evaluatedAt:            string;
}

DomainType — 6 monitored subsystems:

  "scheduler"          — execution scheduling pressure
                         Source: P9-B pressureScore.total, containmentStatus
  "advisory"           — governance advisory signal health
                         Source: P9-B noisyBehaviorCodes, advisoryPressureLevel
  "policy_engine"      — fairness policy lifecycle health
                         Source: P9-E activePolicyCount, conflictDetected
  "enforcement_bridge" — P9-F weight resolution health
                         Source: P9-F enforcementStatus
  "workflow_runtime"   — execution backlog health
                         Source: P9-B delayedExecutionCount, activeExecutionCount
  "tenant_isolation"   — workspace containment boundary health
                         Source: P9-B containmentStatus, P9-F conflictDetected

3. DEGRADATION CLASSIFICATION MODEL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

DegradationStatus — 5 severity levels (0 = nominal, 4 = critical):

  0  "healthy"           — nominal; all containment boundaries holding
  1  "degraded"          — mild signal; continue monitoring
  2  "severely_degraded" — elevated risk; operator review recommended
  3  "containment_risk"  — boundary under active stress; intervention recommended
  4  "critical"          — immediate operator action required

Per-subsystem classification rules:

─── scheduler ───────────────────────────────────────────────────────────────

  pressureScore >= 80 OR containmentStatus = "saturated"  → critical
  pressureScore >= 60 OR containmentStatus = "pressured"  → containment_risk
  pressureScore >= 40 OR containmentStatus = "at_risk"    → severely_degraded
  pressureScore > 0                                        → degraded
  else                                                     → healthy

  Constants: CRITICAL=80, RISK=60, SEVERE=40

─── advisory ────────────────────────────────────────────────────────────────

  advisoryPressure="critical" AND noisyCodes >= 3   → critical
  advisoryPressure="high" OR noisyCodes >= 2        → severely_degraded
  advisoryPressure="medium" OR noisyCodes >= 1      → degraded
  advisoryPressure="low"                            → degraded
  else                                              → healthy

  Constants: NOISY_CRITICAL=3, NOISY_SEVERE=2

─── policy_engine ───────────────────────────────────────────────────────────

  conflictDetected = true               → containment_risk
  activePolicyCount > 1                 → severely_degraded  (P9-E breach — defensive)
  activePolicyCount = 1                 → degraded           (governance in effect)
  else                                  → healthy

─── enforcement_bridge ──────────────────────────────────────────────────────

  enforcementStatus = "conflict"        → containment_risk
  enforcementStatus = "floor_applied"   → severely_degraded
  enforcementStatus = "stale"           → degraded
  enforcementStatus = "resolved"        → degraded           (policy in effect)
  enforcementStatus = "no_active_policy"→ healthy

─── workflow_runtime ────────────────────────────────────────────────────────

  backlogDepth >= 50 AND activeCount > 20   → critical
  backlogDepth >= 50                        → containment_risk
  backlogDepth >= 20                        → severely_degraded
  backlogDepth > 0                          → degraded
  else                                      → healthy

  Constants: BACKLOG_CRITICAL=50, BACKLOG_SEVERE=20

─── tenant_isolation ────────────────────────────────────────────────────────

  conflictDetected = true               → containment_risk
  containmentStatus = "saturated"       → critical
  containmentStatus = "pressured"       → severely_degraded
  containmentStatus = "at_risk"         → degraded
  else                                  → healthy

Overall domain degradationStatus = worstDegradation(all 6 subsystem statuses).
Tiebreaker for dominantSubsystem (affectedSubsystems): DOMAIN_TYPE_ORDER priority
  [tenant_isolation, scheduler, workflow_runtime, advisory, enforcement_bridge, policy_engine]

4. FAILURE PROPAGATION MODEL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

FailurePropagationRisk — 4 levels, derived deterministically from degradationStatus:

  "isolated"   (healthy)           — contained to one subsystem; no spread risk
  "bounded"    (degraded)          — could reach adjacent subsystems in workspace
  "spreading"  (severely_degraded) — cross-subsystem degradation already visible
  "cascading"  (containment_risk   — platform-wide propagation possible
               or critical)

This mapping is intentionally strict:
  • "isolated" requires a fully healthy domain — any signal degrades to "bounded".
  • "cascading" covers both containment_risk and critical — at containment_risk
    the propagation path is already active; at critical it is confirmed.

Propagation semantics:

  isolated  → failure is scoped to a single subsystem and has not been observed
              to cause downstream effects in other subsystems.

  bounded   → failure is visible in one subsystem (mild signal) and could reach
              adjacent subsystems (e.g., high backlogDepth in workflow_runtime
              could increase scheduler pressure). Operator should monitor.

  spreading → degradation is active in 2+ subsystems simultaneously (e.g.,
              scheduler pressure + advisory storm). The failure is crossing
              subsystem boundaries. Operator review recommended.

  cascading → cross-workspace impact is possible. Either containment boundaries
              are under active stress (containment_risk) or the workspace is
              in critical state. Immediate operator attention required.

Cross-workspace propagation paths (identified but not yet instrumented):

  saturated workspace backlog
    → scheduler shared queue overflow
    → delayed dispatch for other workspaces
    → increased pressureScore for unrelated workspaces

  policy conflict (two active policies for same workspace)
    → P9-F fail-closed to weight floor (0.25)
    → workspace scheduler share drops to minimum
    → other workspaces may see increased available share (uneven load)

  advisory storm in one workspace
    → high governance advisory signal volume
    → platform advisory health degrades for that workspace
    → cascades into policy_engine if operator creates multiple policies in response

5. CONTAINMENT BOUNDARY MODEL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Five ContainmentBoundary instances per workspace domain. Each represents a
specific isolation mechanism with four possible status values:

  "holding"  — boundary is intact; no stress detected
  "stressed" — boundary is under mild load; monitor
  "at_risk"  — boundary is under active stress; operator attention
  "breached" — boundary has failed; containment lost for this mechanism

─── 1. tenant_isolation (protects: tenant_isolation subsystem) ──────────────

  Status derivation:
    conflictDetected OR containmentStatus="saturated" → breached
    containmentStatus="pressured"                     → at_risk
    containmentStatus="at_risk"                       → stressed
    else                                              → holding

  What it means:
    This boundary represents the workspace-level isolation contract established
    by P9-A. "breached" means the workspace has escaped its containment envelope
    and is affecting (or being affected by) other workspaces.

─── 2. scheduler_limit (protects: scheduler subsystem) ──────────────────────

  Status derivation:
    effectiveWeight <= 0.25 AND pressureScore >= 80   → at_risk
    effectiveWeight < advisoryWeight                  → stressed
    else                                              → holding

  What it means:
    This boundary represents the P9-F weight enforcement. If the scheduler is
    at minimum weight (0.25 = floor) AND under maximum pressure, the weight
    limit is already at its minimum defensive value — no further reduction is
    possible. "stressed" means a policy has already reduced the effective weight
    below the advisory level, indicating active weight management in progress.

─── 3. advisory_gate (protects: advisory subsystem) ─────────────────────────

  Status derivation:
    noisyCodes.length >= 3   → at_risk
    noisyCodes.length >= 2   → stressed
    else                     → holding

  What it means:
    This boundary represents the advisory signal filter. When multiple noisy
    behavior categories are active simultaneously, the advisory system is
    operating above its normal signal-to-noise capacity. "at_risk" means
    the advisory layer may generate inaccurate governance recommendations
    because the signal volume is too high to distinguish true patterns.

─── 4. policy_gate (protects: policy_engine subsystem) ──────────────────────

  Status derivation:
    conflictDetected                    → breached
    enforcementStatus="conflict"        → at_risk
    activePolicyCount > 0               → stressed
    else                                → holding

  What it means:
    This boundary represents the P9-E single-policy-per-workspace invariant.
    "breached" means P9-E conflict prevention failed and two conflicting policies
    are active (the P9-F bridge handles this defensively). "stressed" means
    one active policy is in effect — the workspace is under governance adjustment
    and the policy gate is performing active weight management.

─── 5. runtime_gate (protects: workflow_runtime subsystem) ──────────────────

  Status derivation:
    backlogDepth >= 50   → at_risk
    backlogDepth >= 20   → stressed
    else                 → holding

  What it means:
    This boundary represents the workflow execution queue health. A large backlog
    means the runtime scheduler is unable to dispatch executions as fast as they
    are being created. "at_risk" means the backlog is large enough to cause
    queue-induced delays for users in this workspace.

6. BLAST RADIUS MODEL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

BlastRadius — estimated scope of failure impact if unchecked:

interface BlastRadius {
  workspaceId:             number;
  estimatedImpactScope:    "workspace_only" | "tenant_group" | "platform_wide";
  affectedSubsystemCount:  number;
  blastRadiusScore:        number;   // 0–100 normalized
  dominantSubsystem:       DomainType;
}

Score formula (clamped to [0, 100]):
  pressureScore × 0.40
  + noisyBehaviorCodes.length × 10
  + backlogDepth:  >50 → +20, >10 → +10, >0 → +5
  + conflictDetected → +20

Scope classification:
  score >= 60 OR overall=critical OR overall=containment_risk → "platform_wide"
  score >= 30                                                 → "tenant_group"
  else                                                        → "workspace_only"

Dominant subsystem selection (most to least influential for blast radius):
  1. tenant_isolation  (saturated/pressured containment)
  2. scheduler         (high pressureScore)
  3. workflow_runtime  (high backlogDepth)
  4. advisory          (multiple noisy behavior codes)
  5. enforcement_bridge (conflict detected)
  6. policy_engine     (fallback)

7. ADVISORY STORM DETECTION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

An advisory storm is a simultaneous overload of governance advisory signals
that amplifies cross-subsystem degradation and increases the probability of
operator misjudgment (creating conflicting policies under time pressure).

detectAdvisoryStorm(noisyCodes, advisoryPressureLevel, activePolicyCount):

Storm conditions (any one triggers storm=true):

  A) noisyCodes.length >= 3
     Three or more concurrent noisy behavior categories — the advisory system
     is receiving compound signal overload. Different categories can cancel
     each other out (false positive) or reinforce (real pattern), making
     automated governance difficult.

  B) advisoryPressureLevel is "high" OR "critical" AND noisyCodes.length >= 2
     High governance pressure combined with at least two noisy behavior
     categories. The advisory system is under compound stress.

  C) activePolicyCount >= 2
     Two or more active policies for the same workspace. This should not occur
     (P9-E prevents it), but if it does, the policy_engine is in a storm state —
     multiple simultaneous governance decisions create an advisory overload.

Storm impact:
  advisoryStormDetected is exposed in FailureContainmentResult and counted in
  PlatformReliabilityOverview.advisoryStormCount.
  An advisory storm independently causes the advisory subsystem to reach
  "severely_degraded" or worse, which in turn raises the workspace's overall
  degradationStatus and propagationRisk.

8. DERIVED STATE LADDER
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

All domain-level state is derived deterministically from degradationStatus:

DegradationStatus → PropagationRisk:
  healthy           → isolated
  degraded          → bounded
  severely_degraded → spreading
  containment_risk  → cascading
  critical          → cascading

DegradationStatus → ContainmentLevel (+ conflictDetected override):
  conflictDetected = true   → breached  (overrides all status values)
  healthy                   → contained
  degraded                  → contained
  severely_degraded         → partial
  containment_risk          → at_risk
  critical                  → at_risk

DegradationStatus → RecoveryClassification:
  healthy           → no_action_needed
  degraded          → monitor_closely
  severely_degraded → monitor_closely
  containment_risk  → operator_attention
  critical          → immediate_intervention

DegradationStatus → ObservabilityHealth:
  healthy           → full
  degraded          → full
  severely_degraded → partial
  containment_risk  → impaired
  critical          → blind

Note: ObservabilityHealth degrades under high pressure because under critical
conditions, the platform's logging subsystem may itself be under resource
contention. "blind" signals that the operator should not rely on the observability
layer for incident analysis — manual inspection is required.

GET /platform/governance/reliability/domains
  Auth:    requireAuth + requireSuperAdmin
  Status:  200 OK
  Returns: { reliability: PlatformReliabilityOverview }

Processing steps:
  1. fetchPlatformPartitions() + active policy GROUP BY query + active policy rows
     (parallel Promise.all — 3 DB queries)
  2. For each partition:
     a. resolveEffectiveSchedulerWeight() → bridge (P9-F)
     b. Build FailureContainmentInput from partition + bridge + policy count
     c. evaluateFailureContainment() → FailureContainmentResult (P10-A)
  3. buildPlatformReliabilityOverview() → PlatformReliabilityOverview
  4. Return { reliability: overview }

DB queries executed:
  1. SELECT id, name FROM workspaces WHERE status IN ('active', 'suspended')
  2. SELECT workspace_id, COUNT(*) FILTER active_executions, delayed_executions
     FROM workflow_executions GROUP BY workspace_id     (P9-B)
  3. SELECT workspace_id, COUNT(*) FROM scheduler_fairness_policies
     WHERE policy_status = 'active' GROUP BY workspace_id  (P9-E count)
  4. SELECT * FROM scheduler_fairness_policies
     WHERE policy_status = 'active'  (P9-E rows for P9-F bridge)

All four queries are read-only. No writes occur in this route.

4 structured log events emitted via logger.info():

All 4 events carry 6 common fields:
  domainId, degradationStatus, propagationRisk, containmentLevel,
  affectedSubsystems[], action

A) reliability_domain_evaluated
   Emitted by: evaluateFailureContainment() — always (every workspace evaluation)
   action: "domain_evaluated"
   Signals: a workspace reliability domain was computed and classified.

B) failure_containment_assessed
   Emitted by: evaluateFailureContainment() — always (every workspace evaluation)
   action: "containment_assessed"
   Signals: containment boundaries were evaluated for this workspace.

C) failure_propagation_risk_detected
   Emitted by: evaluateFailureContainment() — only when propagationRisk is
               "spreading" or "cascading" (non-trivial risk only)
   action: "propagation_risk_detected"
   Signals: cross-subsystem failure propagation risk is above bounded level.
   Note:  This event is intentionally filtered — "isolated" and "bounded" risks
          do not emit this event to reduce log noise during normal operation.

D) runtime_degradation_classified
   Emitted by: evaluateFailureContainment() — always (every workspace evaluation)
   action: "degradation_classified"
   Signals: runtime degradation status was classified from live signals.

Event design rationale:
  • Events A, B, D fire for every workspace evaluation (full observability).
  • Event C fires only for elevated risk (actionable signal filtering).
  • All events carry affectedSubsystems[] — searchable field for subsystem-
    specific alerting pipelines (e.g., alert if "scheduler" appears in
    affectedSubsystems AND propagationRisk="cascading").

15. REMAINING RESILIENCE GAPS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

A) NO CROSS-WORKSPACE PROPAGATION INSTRUMENTATION
   P10-A computes blast radius per workspace independently. There is no model
   for "workspace A's saturation is currently increasing workspace B's pressureScore."
   This cross-workspace causality chain requires:
     1. Correlation of pressureScore increases across workspaces with backlog
        growth in a noisy workspace.
     2. A shared-resource contention model (scheduler queue capacity).
   Currently: each workspace is evaluated in isolation. Cross-workspace
   propagation is classified at the platform level via overallPropagationRisk
   (worst across all) but not causally linked.

B) NO RELIABILITY HISTORY
   All domain evaluations are point-in-time. There is no historical record of:
     • When did workspace X enter "severely_degraded" state?
     • How long did the advisory storm last?
     • Was the "critical" event resolved after operator intervention?
   A reliability_domain_events table would enable incident timeline reconstruction.

C) NO RECOVERY TRACKING
   RecoveryClassification recommends operator action but there is no model for:
     • Was the recommended action taken?
     • Did the domain status improve after operator intervention?
     • MTTR (Mean Time To Recovery) per workspace.

D) NO SUBSYSTEM-LEVEL BLAST RADIUS
   BlastRadius is computed at the workspace level. A subsystem-specific blast
   radius would allow more precise containment recommendations:
     • "Scheduler subsystem blast radius: 60 (platform_wide)"
     • "Advisory subsystem blast radius: 20 (workspace_only)"
   Currently: a single blastRadiusScore aggregates all subsystem signals.

E) NO CONTAINMENT BOUNDARY HISTORY
   ContainmentBoundary status is computed live. There is no record of:
     • How long was the tenant_isolation boundary "stressed" before breaching?
     • Which boundary breaches precede critical-state incidents?
   This boundary transition history would enable predictive reliability modeling.

F) NO RELIABILITY SLO DEFINITIONS
   Reliability domains are classified but there are no formal SLO targets:
     • "Platform SLO: >= 90% of workspaces must be healthy or degraded"
     • "SLO breach: criticalCount > 0 for more than 15 minutes"
   Without SLO definitions, the overview provides no actionable threshold.

G) NO AUTOMATED ALERT ROUTING
   The failure_propagation_risk_detected event is emitted when risk >= spreading,
   but there is no routing to an alerting system. An operator must actively poll
   GET /platform/governance/reliability/domains to discover critical states.
   A future alerting pipeline would subscribe to the structured events and
   route to PagerDuty / Slack based on event fields.
