── OVERVIEW ──────────────────────────────────────────────────────────────────

Phase 4-E adds the ability to resume or reject workflow executions that are
paused at an approval step (status='waiting_approval').  It is the final
governance layer in Phase 4, completing the full approval lifecycle:

  Trigger → (P4-A context) → run steps → approval step → PAUSE (waiting_approval)
         ↓
  Authorized user approves or rejects
         ↓
  resumeExecution()  → guarded UPDATE → re-enter loop from step N+1  (P4-E)
  rejectExecution()  → guarded UPDATE → status='failed', audit record  (P4-E)

2. EXECUTOR REFACTOR: artifacts/api-server/src/lib/workflows/executor.ts
   ────────────────────────────────────────────────────────────────────────
   a) Extracted internal runStepLoop(executionId, workflowId, steps, ctx,
      timeoutAt, startFromIndex) — shared by both executeWorkflow and
      resumeExecution.  Zero behavior change to the initial execution path.

   b) executeWorkflow() unchanged semantics:
      • Guarded pending → running (P4-D WHERE clause + .returning())
      • Calls runStepLoop(..., startFromIndex=0)

   c) NEW: resumeExecution(executionId, resumedBy, notes?)
      • Fetches execution from DB (status, currentStepIndex, cancelRequested,
        timeoutAt, context, workflowId, workspaceId)
      • Pre-check: status must be 'waiting_approval'
      • Pre-check: cancelRequested must be false
      • Fetches workflow definition (steps)
      • Reconstructs ExecutionContext from DB:
          - triggerEvent/triggerData from execution.context JSONB
          - stepOutputs rebuilt from completed workflow_execution_steps rows
          - resolvedData recomputed via buildResolvedData(stepOutputs)
      • P4-D guarded UPDATE: waiting_approval → running
        WHERE status='waiting_approval' AND cancelRequested=false
        + .returning() race detection
      • On guard win: inserts workflowApprovalsTable record (action='approved')
      • Calls void runStepLoop(..., startFromIndex=currentStepIndex+1)
      • Returns immediately { success: true, approvalStepIndex }
      • On race lost: returns { success: false, code: 'TRANSITION_RACE_LOST' }

   d) NEW: rejectExecution(executionId, rejectedBy, notes?)
      • Fetches execution from DB
      • Pre-check: status must be 'waiting_approval'
      • P4-D guarded UPDATE: waiting_approval → failed
        WHERE status='waiting_approval'
        + .returning() race detection
      • On guard win: inserts workflowApprovalsTable record (action='rejected')
      • Returns { success: true, approvalStepIndex }

  ─────────────────   ─────────────   ───────────────────────────   ────────────
  pending             running          WHERE status='pending'         executeWorkflow
  running             waiting_approval WHERE status='running'         runStepLoop
  running             failed           WHERE status NOT terminal      runStepLoop

── KEY DESIGN DECISIONS ──────────────────────────────────────────────────────

WHY RE-ENTRY FROM NEXT STEP (currentStepIndex + 1):
  The approval step already completed (notifications sent, output persisted).
  Re-running it would re-send notifications, duplicate the step log row,
  and re-set status='waiting_approval'.

WHY IN-PROCESS (NOT QUEUED):
  resumeExecution() fires void runStepLoop() directly in the same Node process.
  The caller receives { success: true } immediately; the caller polls for status.
  This matches the pattern used by engine.ts for initial execution.

WHY 'failed' (NOT A NEW 'rejected' STATUS):
  'failed' already covers "could not complete" semantics.  No new terminal
  status, no downstream changes to TERMINAL_STATUSES, diagnostics, or the
  admin panel.  The rejection decision is auditable via workflow_approvals.

WHY APPROVAL RECORD INSERTED AFTER GUARDED UPDATE:
  Inserting before the UPDATE risks orphaned records when the guard fails
  (force-timeout, race).  Inserting after guarantees exactly one record per
  successful transition.

WHY cancelRequested IN THE WHERE GUARD (NOT JUST PRE-CHECK):
  The pre-check (before the UPDATE) can lose a race with a concurrent cancel
  request.  The WHERE clause enforces both conditions atomically in PostgreSQL.
