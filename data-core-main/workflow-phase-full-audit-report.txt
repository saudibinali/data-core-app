EXECUTIVE SUMMARY
================================================================================

The platform implements a genuine event-driven Workflow Engine in api-server
(artifacts/api-server/src/lib/workflows/) with database-backed definitions,
executions, step audit, approval pause/resume, delay scheduling, and publish
versioning. This is the CANONICAL automation path for workspace tenants.

However, "workflow" and "approval" are NOT single concepts in this codebase.
Parallel systems remain active: legacy ticket approvals (approvals table),
canonical leave approval steps (leave_approval_steps), domain policy gates
(procurement, inventory, payroll), and platform governance workflows
(governance_workflow_actions) — unrelated to tenant workflow_definitions.

Shared packages @workspace/core-workflows and @workspace/core-approvals are
foundation placeholders (types only) not connected to runtime.

Classification: PARTIALLY OPERATIONAL | SEMI-DYNAMIC | METADATA-HEAVY |
FAKE-DYNAMIC for enterprise approval/escalation features exposed in config/UI.

KEY FINDINGS (CROSS-PHASE)
================================================================================

WHAT IS REAL RUNTIME
--------------------
• WorkflowEngine listens on eventDispatcher, matches trigger_event + form hints
• Executor runs 7 step types with guarded status transitions
• Approval pause (waiting_approval) + resume/reject via API
• Delay pause (waiting_delay) + scheduler poll resume
• Execution TTL (24h default), cancellation flag, steps_snapshot immutability
• Publish pipeline with workflow_definition_versions
• In-app notifications from notification/approval steps + bus listeners
• Form submit → form.submitted → optional workflow start
• Extensive unit/integration tests under lib/workflows/__tests__/

WHAT IS UI / CONFIG ONLY
------------------------
• approvalType multi/sequential/parallel/conditional (single decision resumes)
• onTimeout auto_approve/reject/escalate (simulation.ts only)
• approverType department_head (not in approval step resolver)
• round_robin assignment (blocked at validation)
• Parallel "Governance Workflows" naming vs tenant Workflows

WHAT IS LEGACY / PARALLEL
-------------------------
• approvals table — ticket_id NOT NULL — routes/approvals.ts
• leave_approval_steps — canonical leave in routes/leave.ts
• hr_employee_leaves vs leave_requests (migration modes)
• PayrollRunWorkflow service class — not workflow_definitions
• governance_workflow_actions — platform violation remediation

WHAT IS MISSING
---------------
• Delegation, reassignment, fallback approvers
• Approval reminders and escalation workers
• Employee workflow approval inbox
• Form status sync with execution outcome
• Email as default workflow notification channel
• Unified process engine / BPMN
• core-workflows / core-approvals package wiring

================================================================================
RUNTIME FLOW (CONDENSED)
================================================================================

Create Request:
  Form → form_submissions | Leave → leave_requests | Ticket → tickets

Submit:
  Route emits bus event → workspace_event_logs → WorkflowEngine

Validate:
  Form fields (route) | Publish validator (admin) | Engine conditions

Assignment:
  task step → workflow_tasks | assignment step → ticket assignee

Approval:
  approval step → waiting_approval → POST /executions/:id/approve (workflow.manage)
  OR legacy PATCH /approvals OR leave step decision

Rejection:
  rejectExecution → failed + workflow_approvals OR domain-specific reject

Escalation:
  NOT RUNTIME for tenant automation (config fiction)

Delegation:
  NOT IMPLEMENTED

Completion:
  runStepLoop terminal → completed | failed | cancelled | timed_out

Archival:
  Soft-delete workflow definition; retain execution/version history

APPROVAL ENGINE VERDICT
================================================================================

Operational:

  • Sequential approval STEPS in workflow array
  • Legacy ticket queue
  • Canonical leave chain
  • Manager/role/specific approver resolution

  • Department head approver type
