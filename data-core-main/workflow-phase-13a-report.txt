━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Phase 13-A delivers a complete, read-only Platform Tenant Registry
accessible only by super-admin users. The registry provides a unified
view of every workspace/tenant registered on the platform — including
ownership metadata, operational status, subscription placeholders,
usage snapshots, and risk signal summaries.

The foundation strictly avoids: billing, payment, automatic suspension,
tenant deletion, workspace archiving, subscription mutation, or any
legal/enforcement inference. All subscription and plan fields are
nullable placeholders ready for future configuration.

2. TENANT REGISTRY ARCHITECTURE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Architecture decisions
──────────────────────

a) Tenant identity
   tenantId = String(workspace.id)
   The workspace IS the tenant. No separate tenant table was created.
   PlatformTenantProfile is assembled from: workspace row + first admin
   user row + DB count subqueries (users, tickets, departments).

b) Primary owner derivation
   SELECT id, email, fullName FROM users
   WHERE workspaceId = X AND role = 'admin'
   LIMIT 1
   If no admin exists: primaryOwnerUserId/Email/FullName are null.
   The registry gracefully displays "No workspace admin assigned."

c) Usage summary derivation
   activeUsers      = userCount (from subquery — count of users in workspace)
   ticketCount      = count of tickets in workspace
   departmentCount  = count of departments in workspace
   All limit fields (seatLimit, storageLimit, apiLimit, etc.) = null (placeholder)
   lastCalculatedAt = request timestamp (ISO string)
   No background jobs, no heavy aggregations.

d) Subscription / plan fields
   All null/unknown in P13-A. No payment provider connected.
   subscriptionStatus defaults to "unknown".
   planCode, billingPeriodStart/End, trialEndsAt, gracePeriodEndsAt = null.
   Displayed in UI as "not configured".

e) Risk signal derivation (read-only, no enforcement)
   workspace.status = "active" + userCount > 0  → riskLevel = "none"
   workspace.status = "active" + userCount = 0  → riskLevel = "low"
                                                   operationalWarnings = true
   workspace.status = "suspended"               → riskLevel = "high"
                                                   governanceWarnings = true
   workspace.status = "disabled"                → riskLevel = "critical"
                                                   subscriptionExpired = true
   All other risk signal boolean fields = false (placeholder).

f) No new DB migrations
   The workspaces table was not modified. No new columns were added.
   All enrichment is derived at query time from existing data.

Backend data flow
─────────────────

GET /platform/tenants:
  1. DB: SELECT all workspaces, ordered by name (with count subqueries)
  2. DB: SELECT first admin user per workspace (batch: WHERE workspaceId IN (...))
  3. Build ownerMap: Map<workspaceId, owner>
  4. Build profiles[]: buildTenantProfile(workspace, owner, now) for each
  5. applyTenantFilters(profiles, { status, subscriptionStatus, riskLevel, search })
  6. sortTenantsByName(filtered) — alphabetical, deterministic
  7. req.log.info({ action: "tenant_registry_listed", count, filtersApplied })
  8. res.json({ tenants: sorted, total: sorted.length })

GET /platform/tenants/:tenantId:
  1. DB: SELECT workspace by id (with count subqueries)
  2. DB: SELECT first admin user for this workspace
  3. buildTenantProfile(workspace, owner, now)
  4. req.log.info({ action: "tenant_profile_read", tenantId })
  5. res.json({ tenant: profile })

GET /platform/tenants/:tenantId/summary:
  1. DB: SELECT workspace by id (with count subqueries)
  2. buildTenantProfile(workspace, null, now) — no owner lookup (lightweight)
  3. req.log.info({ action: "tenant_summary_read", tenantId })
  4. res.json({ tenantId, workspaceName, tenantStatus, ..., riskSignalSummary, usageSummary })

  - No external payment provider in scope — platform manages its own

  - Renewal approaching logic: compute from billingPeriodEnd vs. now
  - Grace period active logic: compute from gracePeriodEndsAt vs. now

  - P13-A has an empty TenantPlanSummary structure ready for this

  - monthlyApiUsage: aggregate API call counts per workspace per month
  - documentsUsed: track document/form submissions per workspace
  - Requires: a new usage_snapshots table or an aggregation job
  - P13-A placeholder fields (null) will be populated from this

  - gracePeriodActive: compute from gracePeriodEndsAt (requires P13-C)

  - usageLimitExceeded: compute from usage vs. plan limit (requires P13-D/E)

  - Requires: GET /platform/tenants/:tenantId is already implemented ✓
