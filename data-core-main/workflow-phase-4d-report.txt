SECTION 0 — EXECUTIVE SUMMARY
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Phase 4-D closes the last class of race conditions in the workflow runtime by
converting every status-changing UPDATE from a blind write into a guarded write:

  BEFORE (P4-A/B/C):  UPDATE ... SET status='X' WHERE id = :id
  AFTER  (P4-D):      UPDATE ... SET status='X' WHERE id = :id AND <guard>
                      .returning() → empty means the race fired → log + return

The model is optimistic concurrency control without distributed locks:
  • PostgreSQL atomically serializes concurrent UPDATEs on the same row.
  • Exactly one UPDATE wins the WHERE guard in a concurrent race.
  • The losing UPDATE returns .returning() = [] — detected and handled.
  • No retries. No compensating transactions. No distributed mutexes.

The result: status transitions are now exactly-once. A terminal state,
once set, cannot be overwritten by any concurrent executor, route handler,
or admin endpoint — regardless of timing.

── T7: Force-timeout route (POST /executions/:id/timeout) ─────────────────────

  WHERE: status NOT IN TERMINAL
  GUARD: prevents overwriting a terminal state that the executor set between
         the pre-flight SELECT and this UPDATE.

  BEFORE:
    const [updated] = await db
      .update(...).set({ status: "timed_out", completedAt: now })
      .where(eq(...id, id))
      .returning();
    // always treated as successful

  AFTER:
    const [updated] = await db
      .update(...).set({ status: "timed_out", completedAt: now })
      .where(and(eq(...id, id), not(inArray(...status, [...TERMINAL_STATUSES]))))
      .returning();
    if (!updated) {
      req.log.warn({ ..., action: "transition_race_lost" }, "P4-D: force-timeout lost race");
      res.status(409).json({ code: "TRANSITION_RACE_LOST", executionId: id });
      return;
    }

  Race scenario this closes:
    t=0  GET existing → status='running' (pre-flight passes).
    t=1  Executor: last step completes, UPDATE status='completed'.
    t=2  POST /timeout UPDATE fires → .returning()=[] → 409 TRANSITION_RACE_LOST.
    WITHOUT GUARD: force-timeout would overwrite 'completed' → data corruption.

── T8: Cancel flag route (POST /executions/:id/cancel) ────────────────────────

  WHERE: cancelRequested=false AND status NOT IN TERMINAL
  GUARD: closes two races simultaneously with one atomic UPDATE:
    a) Duplicate cancel requests that bypass the pre-flight check.
    b) Execution becoming terminal between the pre-flight SELECT and UPDATE.

  BEFORE:
    await db.update(...).set({ cancelRequested: true })
            .where(eq(...id, id));
    // always succeeded

  AFTER:
    const [flagged] = await db
      .update(...).set({ cancelRequested: true })
      .where(and(
        eq(...id, id),
        eq(...cancelRequested, false),
        not(inArray(...status, [...TERMINAL_STATUSES])),
      ))
      .returning({ id });
    if (!flagged) {
      // Re-fetch to determine why the guard fired:
      const [current] = await db.select(...).where(eq(...id, id));
      if (!current || isTerminalStatus(current.status)) → 422 EXECUTION_ALREADY_TERMINAL
      else if (current.cancelRequested)                 → 409 CANCEL_ALREADY_REQUESTED
      else                                              → 409 TRANSITION_RACE_LOST
    }

  Why re-fetch on race:
    The 'existing' snapshot is stale by the time .returning() is empty.
    Re-fetching gives the caller accurate current-state information and
    the most specific error code.

  Error codes (post-P4-D):
    EXECUTION_ALREADY_TERMINAL — status became terminal between pre-flight and UPDATE.
    CANCEL_ALREADY_REQUESTED   — duplicate cancel (flag was set concurrently).
    TRANSITION_RACE_LOST       — state changed in an unexpected way (should be rare).

SECTION 5 — OPTIMISTIC CONCURRENCY MODEL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

WHY NO LOCKS:

  Option A (NOT chosen): Pessimistic locking.
    SELECT ... FOR UPDATE inside a transaction wrapping the entire step loop.
    Problem: locks the row for the entire duration of a step (seconds to minutes),
    blocking all readers and the force-timeout admin endpoint.  Unacceptable
    for an operations platform with real-time monitoring.

  Option B (NOT chosen): Advisory locks.
    pg_try_advisory_lock(executionId) held for the step loop lifetime.
    Problem: same duration issue + lock leaks on process crash.
    Requires background cleanup.

  Option C (CHOSEN): Optimistic concurrency via guarded UPDATE.
    PostgreSQL atomically evaluates the WHERE clause and runs the UPDATE
    in a single lock acquisition of the row.  Concurrent UPDATEs on the
    same row are serialized by the DB engine.  Exactly one UPDATE wins;
    all others see .returning()=[].  Total lock time: microseconds.
    Zero lock leaks.  Zero blocking of readers.

WHY THIS IS SUFFICIENT FOR CURRENT SCALE:

  The workflow engine is single-instance (one Node.js process).  Concurrent
  execution races occur ONLY between:
    1. The executor (async step loop) and an admin HTTP handler.
    2. Two concurrent admin HTTP requests (force-timeout + cancel).
  In both cases, the guarded UPDATE correctly serializes the winner.
  Multi-instance deployment (future): the guard model extends correctly —
  PostgreSQL serializes the guard across all instances simultaneously.

SECTION 6 — TRANSITION OWNERSHIP MODEL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  RULE: The executor owns all status transitions.
        Route handlers may request intent (cancel_requested flag) but
        must never write status directly.

  EXCEPTION: POST /timeout (force-timeout) is an irreversible admin override
  for stuck-execution recovery.  It bypasses the cooperative model by design.
  The T7 guard ensures even this exception cannot overwrite a terminal state.

  ENFORCEMENT MODEL (P4-D complete):
    All 8 status-changing UPDATEs now use WHERE guards.
    No blind UPDATEs remain in the execution status flow.

  WHY THIS IS SAFE WITHOUT TRANSACTIONS:
    Each individual status UPDATE is atomic at the PostgreSQL level.
    The transition correctness guarantee comes from the WHERE guard,
    not from wrapping multiple operations in a transaction.

    Between the WHERE guard and the status transition, no other code
    runs — it is a single SQL statement.  The window for a race is
    reduced to the microsecond-level lock acquisition time.

Every race detection emits a structured logger.warn (or req.log.warn) entry.

Guaranteed fields for executor races:
  executionId         number
  workflowId          number
  workspaceId         number
  attemptedTransition string     — e.g. "running→completed"
  action              string     — always "transition_race_lost"
  + transition-specific fields (stepIndex, stepName, timeoutAt, etc.)

Guaranteed fields for route races:
  executionId         number
  workflowId          number
  workspaceId         number
  previousStatus      string
  action              "transition_race_lost"
  + route-specific fields (adminId, cancelledBy, etc.)

Log level: WARN (all race detections).
  Race conditions are operational signals, not system errors.
  They indicate concurrent activity that was handled gracefully.

Search patterns (for log aggregation):
  action:transition_race_lost          — all races across all transitions
  attemptedTransition:running→completed — completion races specifically
  attemptedTransition:running→timed_out — TTL boundary races
  attemptedTransition:pending→running   — duplicate executor start races
  P4-D: (substring in message)         — all P4-D race log entries
