SECTION 0 — EXECUTIVE SUMMARY

Phase 4-B adds execution timeout protection to the workflow runtime.

Every new execution gets a deadline: timeout_at = now() + 24h.
The executor checks this deadline at every inter-step boundary.
If the deadline has passed, the execution transitions to status='timed_out'.

No schedulers. No cron jobs. No setTimeout per execution.
No step interruption. No distributed timers.

Two new diagnostic endpoints:
  GET  /api/workflows/executions/stuck        — list overdue executions
  POST /api/workflows/executions/:id/timeout  — admin force-timeout (super_admin only)

Exports:

  TERMINAL_STATUSES: readonly string[]
    ['completed', 'failed', 'error', 'timed_out', 'cancelled']
    Single source of truth for terminal status classification.

  isTerminalStatus(status: string): boolean
    Returns true if status is in TERMINAL_STATUSES.
    Used by POST /executions/:id/timeout to reject invalid requests.

  isExecutionTimedOut(timeoutAt: Date | null | undefined, now?: Date): boolean
    Core TTL guard — checks if the execution deadline has passed.
    Returns false for null/undefined timeoutAt (legacy rows — safe).
    Accepts injectable 'now' for deterministic unit testing.
    Called at every inter-step boundary in executor.ts.

  computeOverdueMs(timeoutAt: Date | null | undefined, now?: Date): number
    Returns how many milliseconds past the deadline an execution is.
    Returns 0 for null timeoutAt and for non-overdue executions.
    Used in structured logs and stuck endpoint response.

  computeTimeoutAt(ttlHours: number, from?: Date): Date
    Derives the absolute deadline from a reference time.
    Used by engine.ts: computeTimeoutAt(DEFAULT_EXECUTION_TTL_HOURS).

SECTION 3 — TIMEOUT LIFECYCLE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Full flow from event trigger to timed_out:

  1. Event fires → engine.handleEvent() matches workflow definition.

  2. engine.ts:
       const timeoutAt = computeTimeoutAt(24)  // now + 24h
       INSERT INTO workflow_executions (status='pending', timeout_at=timeoutAt)
       void executeWorkflow(executionId, steps, ctx, timeoutAt)

  3. executor.ts — for-loop over steps:
       For each step:
         a. INSERT step log (status='running')
         b. Execute step handler
         c. UPDATE step log (status='completed')
         d. UPDATE execution (currentStepIndex = i)
         e. P4-A: ctx.stepOutputs[step.index] = result.output
         f. P4-A: ctx.resolvedData = buildResolvedData(stepOutputs)

         ── P4-B: INTER-STEP BOUNDARY CHECK ──────────────────────
         g. if (isExecutionTimedOut(timeoutAt)):
              UPDATE workflow_executions SET status='timed_out', completed_at=now()
              logger.warn({ executionId, overdueMs, completedStepIndex, ... })
              return  ← stops execution cleanly

  4. Normal completion (no timeout):
       UPDATE workflow_executions SET status='completed'

  5. Admin force-timeout (POST /executions/:id/timeout):
       Checks isTerminalStatus(current) → 422 if already terminal
       UPDATE workflow_executions SET status='timed_out', completed_at=now()
       req.log.warn({ action: 'admin_force_timeout', adminId, overdueMs, ... })

Key invariant: A step that has started ALWAYS completes before timeout is checked.
The timeout fires at the BOUNDARY, not during the step.

SECTION 4 — TIMEOUT ENFORCEMENT BEHAVIOR
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

WHY COOPERATIVE (NOT PREEMPTIVE):
  Preemptive timeout (Promise.race + setTimeout) would cut a step mid-execution,
  potentially orphaning:
    - Notifications already sent
    - Tasks already created in DB
    - Status updates already applied
  No compensation/rollback path exists. Partial side effects would persist silently.

  The cooperative model guarantees:
    • Every step either FULLY completes (output written) or is NEVER STARTED.
    • No side effects from partial step execution.
    • Clean state transition: completed steps → timed_out with all prior outputs intact.

WHY TTL CHECKED BETWEEN STEPS ONLY:
  The inter-step boundary (after P4-A output lines, before next for-loop iteration)
  is the only point where:
    a) The previous step is fully settled (DB logs written, outputs stored).
    b) The next step has not been initiated.
    c) The executor can safely return without cleanup.

WHY NO SCHEDULER:
  setTimeout per execution:
    - Lost on process restart (no persistence).
    - Requires timer ID tracking across distributed instances.
    - Adds complexity with no immediate benefit for current use case.
  The lazy model: timeout checked at each inter-step boundary.
  Worst case: timeout fires one step late if a step runs longer than the TTL.
  For linear workflows with short-lived steps (< 1s each), this is negligible.

WHY timed_out ≠ failed:
  failed    = a step returned { success: false } or threw an unhandled error.
              The workflow ATTEMPTED the step and it DIDN'T WORK.
              Indicates a bug, integration failure, or data problem.

  timed_out = execution ran for too long regardless of step results.
              All steps may have SUCCEEDED; the workflow is just SLOW.
              Indicates a configuration problem, stuck approval, or scheduling issue.

  cancelled = (future P4-C) explicit admin/user intent to stop the execution.
              Active cancellation — different from passive deadline expiry.

GET /api/workflows/executions/stuck

  Auth:    requireAuth + workflow.view permission (admin/manager/member with perm)
  Purpose: Find all actively-stuck executions that have exceeded their timeout_at.

  Filter logic:
    WHERE status IN ('running', 'waiting_approval')
      AND timeout_at IS NOT NULL
      AND timeout_at < now()

  Optional query params:
    ?workflowId=<id>       — narrow to one workflow definition
    ?overdueMinutes=<n>    — only include executions overdue by ≥ N minutes
                             (adds: AND timeout_at <= now() - interval 'N minutes')

  Response:
    { data: [...], total: N, asOf: ISO8601 }

  Per row:
    id, workspaceId, workflowId, workflowName, workflowKey, module,
    triggerEvent, status, currentStepIndex, totalSteps,
    startedAt, timeoutAt, overdueMs (computed in SQL),
    triggeredBy, triggeredByName, error

  Ordering: timeout_at ASC (oldest deadline first — most urgent at top)

  Note: Legacy rows (timeout_at = NULL) are excluded by the IS NOT NULL filter.
  They cannot be "stuck" by the TTL definition and require manual assessment.

  Note: timed_out executions do NOT appear here — they are terminal. The stuck
  endpoint only shows executions that are STILL RUNNING past their deadline.

POST /api/workflows/executions/:id/timeout

  Auth:    requireAuth + requireSuperAdmin
  Purpose: Force-transition a stuck execution to status='timed_out'.
           Useful for executions where the inter-step TTL check cannot fire:
             - Execution stuck in waiting_approval (approval never provided).
             - Execution where the current step itself hung (rare).
             - Legacy executions (no timeout_at) that have been stuck for days.

  Validation:
    - 404 if execution not found.
    - 422 if execution is already terminal (completed, failed, error, timed_out, cancelled).
      Returns: { error, code: 'EXECUTION_ALREADY_TERMINAL', currentStatus, executionId }
    - Super admin can act across ALL workspaces (no workspace scope check).

  On success:
    UPDATE workflow_executions SET status='timed_out', completed_at=now()
    Returns: {
      executionId, workflowId, workspaceId,
      previousStatus, status: 'timed_out',
      completedAt, timeoutAt, overdueMs
    }

  Audit:
    Structured req.log.warn() entry containing:
      executionId, workflowId, workspaceId,
      adminId (req.userId), previousStatus,
      timeoutAt, overdueMs, forcedAt, action: 'admin_force_timeout'

  Idempotency note:
    Calling this on an already-timed-out execution returns 422 with
    EXECUTION_ALREADY_TERMINAL — not idempotent by design (the admin
    should only call this when the execution is genuinely stuck, not
    as a polling/retry mechanism).

If the DB update fails (unlikely), the process-level error catch in engine.ts
will log it as ERROR via the existing unhandled execution error path.

SECTION 10 — EDGE CASES DISCOVERED
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

EC-1: Single-step workflow never hits inter-step boundary (handled)
  A workflow with exactly 1 step completes in one iteration.
  The TTL check in the for-loop fires AFTER step 0 completes (i=0).
  If step 0 took longer than 24h, the execution is marked timed_out instead
  of completed, even though all steps ran. This is correct behavior:
  the execution exceeded its budget regardless of step outcomes.

EC-2: Zero-step workflow (no steps)
  The for-loop does not execute. The TTL check is never called.
  The execution transitions directly to completed after the loop.
  Zero-step workflows cannot time out via the inter-step check.
  They can still be force-timed-out by an admin via POST /timeout.
  Risk: NONE — zero-step workflows complete instantly in practice.

EC-3: Last step exceeded TTL (completed or timed_out?)
  If step N is the LAST step and it completes AFTER the deadline:
    - The TTL check fires at the inter-step boundary (i = N).
    - status = 'timed_out', NOT 'completed'.
    - The post-loop "completed" update is never reached (executor returned early).
  This is correct: the execution exceeded its budget.
  Trade-off: all N steps succeeded, but the execution is marked timed_out.
  Documented in ttl.ts and test T6.

EC-4: typecheck failure after schema change (resolved)
  After adding timeout_at to lib/db/src/schema/workflows.ts, the api-server
  typecheck failed because TypeScript types for @workspace/db were stale.
  Fix: pnpm run typecheck:libs (rebuilds composite lib declarations).
  Lesson: Always run typecheck:libs after editing lib/db/src/schema/*.ts.

EC-5: overdueMinutes filter edge cases
  overdueMinutes=0 → threshold = 0 → cutoff = now() → lte(timeout_at, now()).
  This is equivalent to the base lt(timeout_at, now()) already applied.
  The guard `if (!isNaN(thresholdMs) && thresholdMs > 0)` skips the redundant
  filter for overdueMinutes=0, preventing double application.
  overdueMinutes=NaN → parseInt returns NaN → guard skips it (safe).
  overdueMinutes=-1 → parseInt returns -1 → thresholdMs = -60000 → guard skips.

EC-6: Force-timeout on legacy execution (no timeout_at)
  Legacy executions have timeout_at = NULL.
  POST /timeout still works: it doesn't require timeout_at to be set.
  overdueMs in the response = 0 (computeOverdueMs(null) = 0).
  The audit log captures timeoutAt: null — visible for debugging.

EC-7: Concurrent timeout race (force-timeout vs. executor)
  If the executor is executing step N while an admin calls POST /timeout:
  - The executor's step N runs to completion (step handlers don't check status).
  - The executor then calls UPDATE SET status='timed_out' (via TTL check OR
    the normal completion path UPDATE SET status='completed').
  - The admin's POST /timeout already set status='timed_out'.
  - The executor's final UPDATE (either timed_out or completed) will succeed
    (no WHERE status='running' guard in P4-B — that's P4-D scope).
  Risk: LOW — the admin force-timeout and executor TTL check both set the same
  target status ('timed_out') so the race is benign in the timed_out case.
  The 'completed' overwrite race exists but is unlikely (< 1 step duration window).
  P4-D will add a WHERE status='running' guard to resolve this entirely.
