INVENTORY ARCHITECTURE SUMMARY
-------------------------------
- Greenfield domain: no inventory_* schema or APIs in repo.
- Twelve canonical models (design only): warehouses, locations, items, variants,
  stock balances (derived), stock movements (immutable ledger), receipts + lines,
  reservations, adjustments, count sessions + items, inventory policies.
- Principles: workspaceId on all rows, numeric(19,4) qty, policy snapshots,
  idempotency keys, audit via activity_logs/events, finance readiness fields
  without GL posting.
- Item master required to move from free-text PO lines to receivable stock.

WAREHOUSE STRATEGY
------------------
- Multi-warehouse per workspace; types: standard, virtual, project_site,
  consignment.
- Parent/child warehouse hierarchy; nested locations (bins, quarantine, damaged,
  receiving, shipping).
- Transfers: intra- and inter-warehouse with optional in-transit virtual warehouse;
  approval above thresholds.
- Quarantine: on-hand but not available; damaged location for write-down path.
- Permissions: inventory.warehouse.* ; module dependency inventory → procurement + hr.

MOVEMENT STRATEGY
-----------------
- inventory_stock_movements = source of truth; balances updated transactionally.
- Types: receipt, issue, transfer_out/in, adjustment, reservation_hold/release,
  count_variance.
- Immutability: no DELETE/UPDATE of posted rows; void via compensating movements.
- Corrections linked via correctionOfMovementId; receipt void reverses movements.
- Reservations decrease available qty; optional auto-reserve on PO approval (policy).

PROCUREMENT INTEGRATION SUMMARY
-------------------------------
- Receiving gated on PO statuses: approved, sent, acknowledged.
- inventory_receipts linked to procurement_purchase_orders and po_items.
- Partial receiving: multiple receipts; cumulative quantityReceived on PO lines (P25-A).
- Over-receive: block | warn | allow_with_approval via inventory policy.
- inventoryReadinessStatus on receipt (READY/WARNING/BLOCKED) — no AP integration.
- Vendor and document registry linkage for packing slips / GRN.

FINANCE READINESS SUMMARY
-------------------------
- Store costCenterId, dimensionJson, unitCost on movements/receipts; no journals.
- valuationReadinessStatus / glImpactReadiness mirrors procurement glReadinessStatus.
- Valuation methods: standard, moving_average, fifo (fifo layers deferred).
- finance_locks block adjustments/counts in closed periods (policy).
- finance_account_mappings prepared for future inventory asset / COGS — config only.
- Cost extract report for finance controllers — export gated, no posting.

SECURITY / GOVERNANCE
---------------------
- inventory.* permission namespace (view, warehouse, receipt, issue, transfer,
  adjustment, count, policy, export, confidential, break_glass).
- Workspace isolation mandatory; optional warehouse-level role segregation.
- Approval segregation via workflow + inventory_approval_links (P25-A).
- Exports via generated_reports + export_jobs; platform governance unchanged.
- Module key inventory depends on procurement and hr.

REPORTING
---------
- Fifteen proposed inventory.* report keys registered against generated_reports
  pattern (balance, movement ledger, receipt register, PO fulfillment, warehouse
  summary, reservations, variance, valuation readiness, cost extract).
- Ops center widgets aligned to same metrics (P25-C).

RISKS
-----
1. Free-text PO lines without itemId — receiving mapping friction until item master enforced.
2. Balance/ledger drift if balances updated outside movement transaction.
3. Dual approval paths (ticket vs workflow) — inventory must use workflow links only.
4. Over-receive and negative stock policy mistakes could corrupt trust — default block/warn.
5. Finance users expecting auto-GL on receipt — must communicate prepared-only scope.
