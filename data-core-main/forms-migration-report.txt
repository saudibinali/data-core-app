  appEventBus.emit() receives (EventInput<"form.submitted">):

    {
      type:      "form.submitted",           // EVENT_TYPES.FORM_SUBMITTED
      module:    "forms",
      workspace: { workspaceId: <number> },
      actor:     { userId: <number>, role: <string> },
      metadata:  { idempotencyKey: "form-submitted-<submissionId>" },
      data: {
        submissionId:      <number>,         // ID of the saved form submission
        formId:            <number>,         // form definition ID
        formName:          <string>,         // e.g. "Annual Leave Request"
        owningModule:      <string>,         // e.g. "hr", "approvals", "system"
        submittedByUserId: <number>,         // authenticated user ID
        answers:           <Record<string,   // raw form field values
                             unknown>>,
        workflowEventHint: <string|undefined> // form.workflowEvent column value
                                             // e.g. "hr.form.submitted"
                                             // undefined if column is NULL
      }
    }

  The bus assigns:
    id:        UUID v4 (auto-generated)
    timestamp: current UTC ISO-8601 time (auto-generated)

  Idempotency key format: "form-submitted-<submissionId>"
    Prevents duplicate processing if the bus handler is somehow called twice.

SECTION 3 — HOW WORKFLOWS NOW WORK
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Full event flow after migration:

    forms.ts POST /forms/:id/submissions
        │  (finalStatus = "submitted")
        │
        ▼
    appEventBus.emit({ type: "form.submitted", ..., data: { workflowEventHint: ... } })
        │
        ▼
    bridge.ts (subscribeToAll listener)
        │  toEventPayload() → sets payload.event = "form.submitted"
        │  payload.data includes workflowEventHint
        │
        ▼
    eventDispatcher.dispatch(payload)
        │
        ├──▶ workspace_event_logs INSERT  (audit trail, module = "forms")
        │
        └──▶ WorkflowEngine.handleEvent(payload)
                 │
                 ├─ TIER 1: query WHERE triggerEvent = "form.submitted"
                 │    matches: generic catch-all form workflows
                 │
                 └─ TIER 2: if workflowEventHint present (e.g. "hr.form.submitted")
                      query WHERE triggerEvent = "hr.form.submitted"
                      matches: module-scoped or service-specific workflows
                      dedup: by definition ID (no double execution)

  Workflow routing examples:

    Form with workflowEvent = "hr.form.submitted":
      TIER 1 matches:  workflows with trigger_event = "form.submitted"
      TIER 2 matches:  workflows with trigger_event = "hr.form.submitted"

    Form with workflowEvent = "approvals.form.submitted":
      TIER 1 matches:  workflows with trigger_event = "form.submitted"
      TIER 2 matches:  workflows with trigger_event = "approvals.form.submitted"

    Form with workflowEvent = NULL:
      TIER 1 matches:  workflows with trigger_event = "form.submitted"
      TIER 2:          skipped (hint is undefined, no second query)

    Non-form events (ticket.created, approval.created, etc.):
      TIER 1 matches: exact event type as before
      TIER 2:         never activates (payload.event ≠ "form.submitted")

SECTION 4 — BRIDGE BEHAVIOR
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  The bridge is unchanged — no code modifications were needed.

  What the bridge does for form.submitted events:
    1. Receives the typed event from appEventBus (subscribeToAll listener).
    2. Calls toEventPayload(event):
         payload.event    = event.type          → "form.submitted"
         payload.module   = event.module        → "forms"
         payload.workspaceId = event.workspace.workspaceId
         payload.triggeredBy = event.actor.userId
         payload.data     = event.data          → includes workflowEventHint
    3. Calls eventDispatcher.dispatch(payload):
         - Writes to workspace_event_logs
         - Triggers WorkflowEngine.handleEvent()
    4. Logs bridge and dispatch completion at debug/info level.

  No-loop guarantee (unchanged):
    appEventBus → bridge → eventDispatcher → WorkflowEngine
    eventDispatcher has no reference back to appEventBus.
    No circular event cycle is possible.

  workspace_event_logs entry for a form submission:
    event:       "form.submitted"
    module:      "forms"
    workspaceId: <req.workspaceId>
    triggeredBy: <req.userId>
    data:        full FormSubmittedPayload including workflowEventHint

  HR service submissions still call eventDispatcher.dispatch() directly with
  event = "hr.{slug}.submitted".  This is INTENTIONAL — HR services are
  a separate architecture (dynamic service forms with their own routing).
  No change was made to hr.ts.  TIER 1 of WorkflowEngine catches these exactly
  as before (payload.event = "hr.annual-leave.submitted", etc.).

  ── eventDispatcher is NOT deprecated ─────────────────────────────────────────
  eventDispatcher remains the persistence layer and workflow trigger mechanism.
  The bridge is now its only caller (from route perspective).
  It is NOT deleted, NOT marked @internal yet (that is Phase 3).

  ── workspace_event_logs event name change ────────────────────────────────────
  Before migration: workspace_event_logs stored "hr.form.submitted",
    "approvals.form.submitted", etc. (the raw form.workflowEvent value).
  After migration:  workspace_event_logs stores "form.submitted" (the canonical
    bus event type).  The workflowEventHint is still available in the data JSON.
  This is an intentional change — the audit trail now uses the canonical event
  type, and the hint is advisory metadata only.
  If this affects existing dashboards or analytics queries filtering on the
  old event names, those queries must be updated to filter on "form.submitted"
  and optionally join on data->>'workflowEventHint'.

  ── requestNumber assignment ──────────────────────────────────────────────────
  The requestNumber DB update (UPDATE form_submissions SET requestNumber = ...)
  runs BEFORE the emit call.  No change to this order.  The emitted payload
  does NOT include requestNumber — it was not part of FormSubmittedPayload.
  If requestNumber is needed downstream, listeners can look it up via submissionId.

  ── Confirmation email ────────────────────────────────────────────────────────
  sendSubmissionConfirmation() runs AFTER the emit call, in a separate void
  async block.  No change to email behavior.

  ── Draft submissions ─────────────────────────────────────────────────────────
  finalStatus !== "submitted" (i.e. draft saves) do NOT emit any event.
  Identical behavior to before migration.
