P9-D converts the per-workspace governance visibility produced by P9-A/B/C into
a platform-wide operational control plane visible only to super-admin users.
Where P9-C gave workspace admins a single-tenant view of their own workload
health, P9-D gives the platform operator a cross-workspace aggregated view:
containment distribution, fairness health classification, noisy-tenant roster,
and scheduler pressure statistics — all computed deterministically, all read-only.

Before P9-D:
  • Platform governance APIs limited to GET /platform/stats and /platform/activity
    (workspace/user counts only — no scheduler visibility).
  • No cross-workspace workload aggregation.
  • No platform-wide fairness health signal.
  • No noisy-tenant oversight surface.
  • Super-admin lacked operational scheduler context.

After P9-D:
  • 4 new super-admin-only governance endpoints under /platform/governance/*.
  • PlatformGovernanceOverview — single-model cross-workspace snapshot.
  • PlatformFairnessHealth — tri-dimensional health classifier (healthy/stressed/
    degraded/critical) aggregating containment, pressure fraction, and noisy fraction.
  • ContainmentDistribution + AdvisoryDistribution — precise workspace counts
    by containmentStatus and advisoryPressureLevel.
  • SchedulerPressureSummary — platform-wide totalActive, totalDelayed, avg/max
    pressureScore, pressureDensity.
  • NoisyTenantRecord list — workspaces with detected EXECUTION_MONOPOLY,
    SCHEDULER_BACKLOG_FLOOD, ADVISORY_STORM, or CHRONIC_HOTSPOT_FLOOD behavior.
  • 4 structured observability events for all control-plane operations.

Explicitly excluded (will not build):
  Autoscaling, tenant throttling, dynamic rate-limiting, realtime streaming,
  distributed control planes, autonomous scheduler balancing, cross-region
  federation.

1. PLATFORM GOVERNANCE ARCHITECTURE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Data flow (per request):

  [route handler]
       │
       ├─① DB: SELECT id, name FROM workspaces WHERE status IN ('active','suspended')
       │        ORDER BY id
       │
       ├─② DB: SELECT workspace_id,
       │              COUNT(*) FILTER (WHERE status='running')       AS activeExecutionCount,
       │              COUNT(*) FILTER (WHERE status='waiting_delay') AS delayedExecutionCount
       │        FROM workflow_executions
       │        GROUP BY workspace_id
       │
       ├─③ Compute platformActiveExecutions = SUM(active) across all workspaces
       │        (used for EXECUTION_MONOPOLY detection in evaluateWorkloadContainment)
       │
       ├─④ FOR EACH workspace:
       │        evaluateWorkloadContainment({ workspaceId, activeCount, delayedCount,
       │                                     platformActiveExecutions })
       │        → TenantWorkloadPartition (P9-B engine)
       │
       └─⑤ [platform-governance engine]
                 computeContainmentDistribution(partitions)
                 computeAdvisoryDistribution(partitions)
                 computeSchedulerPressureSummary(partitions)
                 detectNoisyTenants(partitions, workspaceNames)
                 classifyPlatformFairnessHealth(distribution, noisyCount, total)
                 computeTopPressureWorkspaces(partitions, workspaceNames, 10)
                 → PlatformGovernanceOverview

Two DB queries for any number of workspaces — no N+1 query pattern.

3. FAIRNESS HEALTH AGGREGATION SEMANTICS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

classifyPlatformFairnessHealth takes the maximum severity across 3 independent
dimensions. This ensures any single dimension at critical level is surfaced even
if the others are healthy.

Dimension 1 — Saturated partition count:
  ≥ 3  saturated workspaces → critical   (starvation floor active for 3+ tenants)
  ≥ 1  saturated workspace  → degraded
  0    saturated             → healthy

Dimension 2 — Pressured+saturated fraction of total workspaces:
  ≥ 50% (pressured + saturated) → critical
  ≥ 25% (pressured + saturated) → degraded
  ≥ 10% (pressured + saturated) → stressed
  < 10%                         → healthy

Dimension 3 — Noisy tenant fraction:
  ≥ 50% workspaces noisy → critical   (majority of platform under noisy load)
  ≥ 25% workspaces noisy → degraded
  ≥ 10% workspaces noisy → stressed
  < 10%                  → healthy

Result: max("healthy", dim1, dim2, dim3) via FAIRNESS_SEVERITY table.
Special case: 0 partitions → "healthy" (empty platform guard).

Design rationale:
  The tri-dimensional model prevents a single saturated workspace on a large
  platform from being drowned by a "healthy average" — the saturated-count
  dimension is absolute (not fractional) for exactly this reason.
  Conversely, a platform with many slightly-elevated workspaces is caught by
  the pressured-fraction dimension even if no single workspace is saturated.

4. NOISY TENANT OVERSIGHT MODEL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Noisy tenant detection is performed per-workspace by the P9-B engine
(evaluateWorkloadContainment → detectNoisyBehavior). The platform layer
aggregates the results and exposes them via GET /platform/governance/noisy-tenants.

Four noisy behavior categories (P9-B definitions):

  EXECUTION_MONOPOLY
    Workspace holds > 60% of platform-wide active executions.
    Detected when: workspace.active / platformActiveExecutions > 0.60.
    Requires platformActiveExecutions to be supplied to evaluateWorkloadContainment.
    This is the main reason the route computes the platform total first.

  SCHEDULER_BACKLOG_FLOOD
    Workspace's delayed queue exceeds schedulerBatchSize × 5 (default: 50).
    Indicates a workspace is saturating the delayed-execution wakeup queue.

  ADVISORY_STORM
    Active governance signal count exceeds threshold AND advisory level
    is "urgent" or "critical". Indicates governance alert volume beyond
    operational response capacity.

  CHRONIC_HOTSPOT_FLOOD
    ≥ 3 workflows at urgent/critical priority AND concentration ratio > 50%.
    Indicates persistent systemic degradation within the workspace.

The platform layer:
  • Filters partitions by noisyBehaviorDetected = true.
  • Exposes noisyBehaviorCodes (array of category codes) — not the full
    human-readable noisyBehaviorReasons, to avoid leaking internal counts.
  • Sorts by pressureScore DESC for actionability.
  • Advisory-only: no automatic throttling, no scheduler weight mutation.

5. SCHEDULER PRESSURE VISIBILITY MODEL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

SchedulerPressureSummary fields and their interpretation:

  totalActiveExecutions   — sum of running executions across all workspaces.
                            Scheduler throughput indicator. Spikes here suggest
                            a sudden burst workload across the platform.

  totalDelayedExecutions  — sum of waiting_delay executions across all workspaces.
                            Scheduler queue depth indicator. A large value with
                            low active means many workflows are waiting for wakeup.

  avgPressureScore        — mean of all workspace pressureScore.total values.
                            Rounded to 2 decimal places. A rising average indicates
                            general platform loading, not a single hotspot.

  maxPressureScore        — highest individual workspace pressureScore.total.
                            Identifies the single worst-case workspace.
                            Cross-referenced with topPressureWorkspaces for name.

  pressureDensity         — fraction of workspaces with pressureScore > 30.
                            The threshold (ELEVATED_PRESSURE_THRESHOLD=30) is the
                            P9-B "elevated" boundary (normal vs elevated pressure).
                            pressureDensity = 0    → all workspaces normal.
                            pressureDensity = 1    → all workspaces elevated+.
                            pressureDensity = 0.25 → 1 in 4 workspaces elevated+.

11. REMAINING CONTROL-PLANE GAPS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The following gaps are intentionally deferred (per P9-D scope constraints):

A) EXECUTION_MONOPOLY detection requires platformActiveExecutions to be non-zero.
   The current route correctly computes this total, but the value is only as fresh
   as the DB snapshot at request time. There is no continuous platform-execution
   counter — this is acceptable for advisory purposes.

B) platformDelayedBacklog is not passed to evaluateWorkloadContainment in the P9-D
   routes. It is an optional field in WorkloadContainmentInput; when absent, per-
   workspace SCHEDULER_BACKLOG_FLOOD detection uses only the workspace's own delayed
   count, not a platform-wide baseline. Adding it would improve BACKLOG_FLOOD signal
   accuracy but is low priority at current platform scales.

C) Historical trending (fairnessHealth over time) is not stored. Each request
   produces a fresh snapshot. Trend analysis would require periodic snapshots
   written to a governance_snapshots table — deferred to a future phase.

D) P8-E hotspotConcentrationRatio and P8-F advisoryLevel are not fetched per-workspace
   in the P9-D routes. They are optional fields in WorkloadContainmentInput. The current
   implementation uses only activeExecutionCount and delayedExecutionCount from DB.
   Enriching with P8-E/F data would increase noisy-tenant signal quality (CHRONIC_HOTSPOT
   detection, ADVISORY_STORM) but requires additional DB queries per workspace.

E) The /workloads endpoint does not support filtering by containmentStatus or
   fairnessHealth. Filtering is a future enhancement once the UI surface is defined.
