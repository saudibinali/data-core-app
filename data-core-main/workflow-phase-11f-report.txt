1. OBJECTIVE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Convert the governance intelligence architecture (P11-A through P11-E) into an
audit-ready evidence packaging and controlled export preparation layer with:

  ✓ Governance evidence packaging
    buildGovernanceEvidencePackage() assembles a scope-filtered, integrity-hashed
    bundle of governance evidence from all five P11-* layers into a single
    GovernanceEvidencePackage artifact. Six scopes (platform → readiness) control
    which sections are included. All section summaries are derived from runtime
    data passed in from the route layer — no I/O inside the engine.

  ✓ Audit bundle preparation
    Every package includes: packageId, includedSections, per-section summaries,
    evidenceReferences (cross-layer), structured warnings, integrityStatus, and
    a SHA-256 packageIntegrityHash over the stable canonical content.

  ✓ Forensic evidence export readiness
    buildEvidenceReferences() generates typed cross-layer references (P11-A
    through P11-E) for each section of evidence included in the bundle.
    These references carry sourceLayer, entityType, entityId, and a human-readable
    description for auditor consumption.

  ✓ Compliance review packet generation
    GovernanceEvidencePackage is fully JSON-serializable — suitable for storage,
    transfer, or frontend rendering. The packageIntegrityHash allows a downstream
    consumer to verify the package content was not altered in transit.

  ✓ Governance snapshot serialization
    buildTopologySnapshotPayload() produces a GovernanceTopologySnapshotPayload
    with its own snapshotHash (SHA-256 over stable content). No DB persistence —
    the payload is structured for future storage in a governance_topology_snapshots
    table when required.

  ✓ Topology diff and change detection
    diffGovernanceTopologySnapshots(prev, next, now) compares two snapshot payloads
    and returns a GovernanceTopologyDiff with all detected changes: boundary status
    changes per layer, coverage score changes, readiness status changes, layer
    additions/removals, observability coverage changes, and critical gap additions/
    removals.

  ✓ Structured warning codes
    All advisory signals now carry a GovernanceWarningCode (machine-readable) in
    addition to a human-readable message. Codes:
      P11A_NO_RUNTIME_DATA, P11C_NO_RUNTIME_DATA, GOVERNANCE_BOUNDARY_LEAK,
      GOVERNANCE_READINESS_PARTIAL, GOVERNANCE_CRITICAL_UNRESOLVED,
      GOVERNANCE_OBSERVABILITY_INCOMPLETE, EVIDENCE_REFERENCE_MISSING,
      EVIDENCE_SECTION_INCOMPLETE.

Safety constraints upheld:
  ✓ READ-ONLY: packaging engine never writes to DB
  ✓ NO EXTERNAL SUBMISSION: no HTTP calls, no regulator endpoints
  ✓ NO PDF GENERATION: readiness layer only — format is structured JSON
  ✓ NO LEGAL CONCLUSIONS: package is evidence assembly, not legal analysis
  ✓ NO ENFORCEMENT: package content never triggers policy or workflow changes
  ✓ NO AI: all classification and hashing is deterministic rule-based logic
  ✓ FAIL-CLOSED: package integrity degrades (compromised → incomplete → warning)
    rather than silently succeeding when evidence is missing

2. GOVERNANCE EVIDENCE PACKAGE ARCHITECTURE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

2.1  Design Decision: Route-Layer I/O + Pure Engine Assembly

  The packaging engine (governance-evidence-packaging.ts) is a pure function
  library — no DB reads inside the engine. The route handler is responsible for:
    1. Fetching counts and aggregate data from compliance_audit_chains and
       governance_workflow_actions
    2. Building the topology + boundary + readiness artifacts (via P11-E functions)
    3. Passing everything to buildGovernanceEvidencePackage() as a typed input object

  This follows the same separation-of-concerns pattern used by P11-D and P11-E:
  pure functions receive data, compute results, return structured artifacts.

2.2  Architectural Position

  P11-A  integrity_layer    record      — immutable hash-linked audit chains
  P11-B  policy_layer       detect      — 8-policy violation evaluation engine
  P11-C  workflow_layer     orchestrate — 6-state human-governed lifecycle
  P11-D  analytics_layer    observe     — 13-metric effectiveness intelligence
  P11-E  consolidation      topology    — cross-layer boundary + readiness map
  ─────────────────────────────────────────────────────────────────────────────
  P11-F  packaging          export_prep — audit-ready evidence bundle assembly

  P11-F sits above all five prior layers and reads from all of them (through
  data passed in from the route handler). It synthesizes their outputs into
  a single structured artifact suitable for auditor review, future storage,
  or controlled export.

3.1  GovernanceEvidencePackageScope — 6 scope types

  type GovernanceEvidencePackageScope =
    | "platform"    // full cross-workspace view (all 7 sections)
    | "workspace"   // single workspace evidence bundle (5 sections)
    | "entity"      // specific entity (user/resource) evidence (3 sections)
    | "violation"   // evidence around one policy violation (2 sections)
    | "workflow"    // single governance workflow lifecycle (1 section)
    | "readiness";  // topology + boundary + readiness only (2 sections)

3.2  GovernanceEvidenceSection — 7 section types

  type GovernanceEvidenceSection =
    | "audit_integrity"      // P11-A: chain records, hash verification, orphan status
    | "policy_violations"    // P11-B: violation profiles with evidence references
    | "workflow_lifecycle"   // P11-C: workflow action history with transitions
    | "governance_analytics" // P11-D: analytics profile (metrics, stability score)
    | "topology_readiness"   // P11-E: topology + readiness profile
    | "forensic_timeline"    // P11-A: chronological event reconstruction
    | "boundary_summary";    // P11-E: boundary verification results

3.4  Section Summary Models (per-section typed summaries)

  AuditChainSectionSummary    — totalRecords, integrityStatus, orphanCount,
                                 retentionClassifications (Record<string,number>)

  ViolationSectionSummary     — totalViolations, bySeverity, byPolicy,
                                 criticalCount

  WorkflowSectionSummary      — totalWorkflows, activeWorkflows, resolvedWorkflows,
                                 escalatedWorkflows, criticalUnresolved,
                                 escalationRate, throughputRate

  AnalyticsSectionSummary     — workflowStabilityScore, escalationTrend,
                                 policyBreachFrequency, unresolvedCriticalCount

  TopologyReadinessSectionSummary — overallBoundaryStatus, coverageScore,
                                     readinessStatus, totalLayers, activeLayers,
                                     observabilityComplete

  A section summary is only populated if its section is included in the scope
  AND the corresponding data was provided in GovernanceEvidencePackageInput.
  If data is not provided for an included section, the summary is null and a
  EVIDENCE_SECTION_INCOMPLETE warning is generated.

5. PACKAGE INTEGRITY SEMANTICS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

5.1  GovernancePackageIntegrityStatus — 4-tier fail-closed result

  type GovernancePackageIntegrityStatus =
    | "verified"    // no warnings → all expected evidence present, no leaks
    | "warning"     // advisory concerns (e.g., critical unresolved, partial readiness)
    | "incomplete"  // EVIDENCE_SECTION_INCOMPLETE or EVIDENCE_REFERENCE_MISSING
    | "compromised"; // GOVERNANCE_BOUNDARY_LEAK — package cannot be trusted as complete

5.2  classifyPackageIntegrity() — fail-closed algorithm

  ┌─────────────────┬────────────────────────────────────────────────────────┐
  │ Status          │ Condition (checked in order, first match wins)         │
  ├─────────────────┼────────────────────────────────────────────────────────┤
  │ compromised     │ warnings contains GOVERNANCE_BOUNDARY_LEAK             │
  │ incomplete      │ warnings contains EVIDENCE_SECTION_INCOMPLETE          │
  │                 │ OR EVIDENCE_REFERENCE_MISSING                          │
  │ warning         │ any other warnings present                             │
  │ verified        │ warnings list is empty                                 │
  └─────────────────┴────────────────────────────────────────────────────────┘

  compromised takes absolute precedence — a package with a boundary leak is
  never classified as incomplete or warning.

5.3  packageIntegrityHash — SHA-256 over stable canonical content

  computePackageIntegrityHash(content: object): string

  Algorithm:
    1. JSON.stringify(content, Object.keys(content).sort())
       — sorts keys alphabetically for determinism
    2. SHA-256 of the resulting string
    3. Returns 64-character lowercase hex

  The hash covers all package fields EXCEPT packageIntegrityHash itself and
  generatedAt. This means:
    - The same content always produces the same hash (deterministic)
    - Changing scope, summaries, warnings, or references changes the hash
    - A downstream consumer can recompute the hash over the content fields
      to verify package integrity

  Note: This is NOT cryptographic certification, blockchain notarization,
  or a legal seal. It is a content integrity check for transit verification.

5.4  buildGovernanceEvidencePackage() — full build pipeline

  Input:   GovernanceEvidencePackageInput (all optional data fields)
  Steps:
    1. sectionsForScope(scope) → includedSections
    2. buildAuditChainSummary(input) → auditChainSummary (null if not applicable)
    3. buildViolationSummary(input)  → violationSummary
    4. buildWorkflowSummary(input)   → workflowSummary
    5. buildAnalyticsSummary(input)  → analyticsSummary
    6. buildTopologySummary(input)   → topologySummary
    7. input.boundarySummary         → boundaryDetails (direct pass-through)
    8. buildEvidenceReferences(input, includedSections) → evidenceReferences
    9. gatherWarnings(input, includedSections, refs) → warnings
   10. classifyPackageIntegrity(warnings) → integrityStatus
   11. computePackageIntegrityHash(hashableContent) → packageIntegrityHash
  Output:  GovernanceEvidencePackage (pure, no I/O, no side effects)

6.4  Backward Compatibility

  Structured warning codes are additive. Prior warning strings from P11-E
  (plain text in GovernanceBoundaryVerification.warnings[]) are not affected.
  GovernanceScopedWarning is a new type in P11-F — it does not replace or
  modify existing P11-E warning arrays.

7.1  GovernanceTopologySnapshotPayload

  interface GovernanceTopologySnapshotPayload {
    snapshotId:       string;    // "gtsnap:<nowMs>"
    topology:         GovernanceTopologyProfile;
    boundarySummary:  GovernanceBoundarySummary;
    readinessProfile: GovernanceReadinessProfile;
    generatedAt:      Date;
    snapshotHash:     string;    // SHA-256 over stable content (excluding hash)
  }

7.2  buildTopologySnapshotPayload() — pure, no I/O

  Arguments:
    topology:        GovernanceTopologyProfile   (from buildGovernanceTopology)
    boundarySummary: GovernanceBoundarySummary   (from buildBoundarySummary)
    readiness:       GovernanceReadinessProfile  (from buildGovernanceReadiness)
    now:             Date

  Hashed content (snapshotHash):
    SHA-256 of { snapshotId, topology, boundarySummary, readinessProfile }
    with keys sorted alphabetically. excludes snapshotHash and generatedAt.

7.3  No DB Persistence

  buildTopologySnapshotPayload() returns a plain object. It does NOT write to
  any database table. The payload is:
    - Returned directly from GET /platform/governance/topology/snapshot
    - Accepted as POST body to POST /platform/governance/topology/diff
    - Ready for storage in a future governance_topology_snapshots table

8. TOPOLOGY DIFF MODEL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

8.1  GovernanceTopologyDiff — change detection between two snapshots

  interface GovernanceTopologyDiff {
    diffId:                      string;    // "gtdiff:<nowMs>"
    prevSnapshotId:              string;
    nextSnapshotId:              string;
    boundaryStatusChanges:       GovernanceBoundaryStatusChange[];
    coverageScoreChange:         { prev: ...; next: ... } | null;
    readinessStatusChange:       { prev: ...; next: ... } | null;
    layerChanges:                { added: string[]; removed: string[] };
    observabilityCoverageChange: { prev: number; next: number } | null;
    criticalGapChanges:          { added: string[]; removed: string[] };
    hasChanges:                  boolean;
    computedAt:                  Date;
  }

  interface GovernanceBoundaryStatusChange {
    layerId: string;
    prev:    GovernanceBoundaryStatus;
    next:    GovernanceBoundaryStatus;
  }

8.2  diffGovernanceTopologySnapshots(prev, next, now) — pure function

  Arguments:
    prev: GovernanceTopologySnapshotPayload
    next: GovernanceTopologySnapshotPayload
    now:  Date

  Algorithm:
    1. Build prevByLayer / nextByLayer maps from boundarySummary.byLayer arrays
    2. For each layerId in the union of both maps — emit change if status differs
    3. Compare coverageScore from lifecycleCoverage in each topology
    4. Compare overallStatus from each readinessProfile
    5. Diff the set of governanceLayer IDs for additions/removals
    6. Compare totalEventTypes from observabilityCoverage
    7. Diff the criticalGaps arrays from each readinessProfile
    8. hasChanges = true if ANY of 2–7 produced a change

  Ordering: addedLayers, removedLayers, addedGaps, removedGaps are all sorted
  alphabetically for determinism.

  Immutability: neither prev nor next is mutated.

8.3  API Semantics — POST (read-only body comparison)

  POST /platform/governance/topology/diff accepts two snapshot payloads in the
  request body as JSON: { "prev": <snapshot>, "next": <snapshot> }.

  It is POST (not GET) because the comparison inputs are too large for query
  parameters or URL encoding. The handler does NOT write to the database — it
  reads from the request body only, computes the diff, and returns the result.

  Required body fields:
    prev.snapshotId — validated before processing
    next.snapshotId — validated before processing
  Error 400 if either is absent.

GET /platform/governance/evidence-packages/readiness
  Auth:    super_admin
  Params:  (none)
  Process: SELECT count(*) from both tables (no row-level data needed)
           → computeLifecycleCoverage() → buildGovernanceTopology() →
           buildBoundarySummary() → buildGovernanceReadiness() →
           buildGovernanceEvidencePackage(scope="readiness")
           → emit A event
  Returns: { package: GovernanceEvidencePackage }
  Note:    Lightweight — only topology + boundary + readiness, count(*) only.

GET /platform/governance/topology/snapshot
  Auth:    super_admin
  Params:  (none)
  Process: SELECT count(*) + workflowStatus + escalationLevel rows
           → computeLifecycleCoverage() → buildGovernanceTopology() →
           buildBoundarySummary() → buildGovernanceReadiness() →
           buildTopologySnapshotPayload()
           → emit C event
  Returns: { snapshot: GovernanceTopologySnapshotPayload }
  Note:    Snapshot payload includes snapshotHash for integrity verification.
           Designed for clients that want to store snapshots and diff them later.

POST /platform/governance/topology/diff
  Auth:    super_admin
  Body:    { prev: GovernanceTopologySnapshotPayload,
             next: GovernanceTopologySnapshotPayload }
  Process: validate prev.snapshotId + next.snapshotId → 400 if missing
           → diffGovernanceTopologySnapshots(prev, next, now)
           → emit D event
  Returns: { diff: GovernanceTopologyDiff }
  Note:    Read-only body comparison. No DB access. Large snapshots fit in body.
           400 if prev or next are absent or missing snapshotId.

12. REMAINING GOVERNANCE EXPORT GAPS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

P11-F is intentionally a pure evidence assembly layer — no PDF export, no
external submission, no automated enforcement.

  1. NO PERSISTENT TOPOLOGY SNAPSHOTS
     buildTopologySnapshotPayload() returns a payload but does NOT persist it.
     Every call to GET /platform/governance/topology/snapshot generates a fresh
     payload computed from current DB state.
     Gap: a governance_topology_snapshots table (id, snapshotHash, snapshotData JSONB,
     generatedAt) would enable "compare current posture to last week" queries
     via the diff endpoint without the client needing to store the snapshots.

  2. NO WORKSPACE-SCOPED EVIDENCE PACKAGES
     GET /platform/governance/evidence-packages always builds a platform-wide
     package (or the selected scope without workspace filtering). There is no
     workspaceId query parameter to produce a workspace-specific bundle.
     Gap: adding workspaceId filtering to the route + passing workspaceId to
     buildGovernanceEvidencePackage() would enable per-workspace audit bundles.

  3. NO ENTITY-SCOPED EVIDENCE
     The "entity" and "violation" scopes in sectionsForScope() are defined and
     work at the packaging level, but the route always uses "platform" scope
     (or the query param scope). There are no entity-specific DB queries to
     populate violation/entity data for those scopes.
     Gap: dedicated routes GET /platform/governance/evidence-packages/entity/:id
     and /violation/:id with entity-specific DB filtering would populate those
     scopes properly.

  4. PACKAGE NOT STORED — NO PACKAGE REGISTRY
     Generated evidence packages are returned in the HTTP response but not stored.
     An auditor who needs the same package again must re-request it (which may
     produce a different packageIntegrityHash if the DB state changed).
     Gap: a governance_evidence_packages table (id, packageId, packageScope,
     packageData JSONB, integrityHash, generatedAt) would provide an immutable
     package registry — once generated, a package can be retrieved by its
     packageId regardless of subsequent DB changes.

  5. NO PDF/XLSX EXPORT FORMAT
     The package is JSON-only. External auditors typically require PDF or Excel.
     Gap: a separate export service (or a future route) could accept a packageId
     and render the JSON package into a PDF report using a templating library
     (e.g., pdfkit, puppeteer). This is intentionally deferred — the JSON
     structure is designed to be renderer-agnostic.

  6. VIOLATION SECTION HAS NO ACTUAL VIOLATION DATA
     violationSummary in the package is populated from totalViolations,
     violationsBySeverity, violationsByPolicy passed in from the route handler.
     The route handler computes these from governance_workflow_actions (by policyId
     frequency), NOT from an actual P11-B policy evaluation run.
     Gap: the evidence package route could call evaluateGovernancePolicies() with
     current audit chain data to produce real-time violation intelligence for the
     violationSummary section. Currently the route approximates violation counts
     from workflow row counts only.

  7. FORENSIC_TIMELINE SECTION HAS NO ACTUAL CONTENT
     The forensic_timeline section is included in platform and entity scope packages,
     and a P11-A evidence reference is generated for it. However, no actual timeline
     events (chronological audit chain events for a specific entity) are fetched
     from compliance_audit_chains and included in the package.
     Gap: calling reconstructForensicTimeline() (P11-A engine function) from the
     route handler and including the resulting timeline in the package payload
     would give the section real content.

  8. INTEGRITY HASH USES SORTED-KEY JSON (NOT CANONICAL JSON RFC 8785)
     computePackageIntegrityHash() uses JSON.stringify(content, Object.keys(content).sort())
     which sorts top-level keys only. Nested objects are not recursively sorted.
     This means deep objects with differently-ordered keys could produce different
     hashes even for logically equivalent content.
     Gap: using a proper canonical JSON serialization (RFC 8785 / JSON-LD) library
     would guarantee full recursive determinism regardless of how objects are
     constructed internally.

  9. EVIDENCE REFERENCES CARRY COUNT DESCRIPTORS (NOT ACTUAL IDs)
     referenceId and entityId in GovernanceEvidenceReference carry count
     descriptors ("count:400") rather than actual database primary keys.
     Gap: for entity- and violation-scoped packages, the referenceId should
     carry actual DB PKs so an auditor can query the specific records.

  10. NO PACKAGE SIGNATURE / AUDIT STAMP
      The packageIntegrityHash is a content hash — it is NOT signed by any key.
      A malicious actor with access to both the package JSON and the hash function
      could re-hash a modified package to match.
      Gap: signing the packageIntegrityHash with a server private key (stored as
      a platform secret) would make the integrity check non-repudiable. This is
      out of scope for P11-F (which was explicitly asked not to include notarization)
      but is the natural next step for a forensic-grade export pipeline.
