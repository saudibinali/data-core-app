FINAL ARCHITECTURE DECISIONS (A–K)
================================================================================
A. Workspace boundary
   - workspace_id is client data isolation; no tenant_id on HR tables
   - tenantId in platform APIs = workspaces.id (alias); no relationship change now

B. Person model
   - employees = HR source of truth; users = login/RBAC only
   - employees.user_id optional; no employee-in-users; no silent user-from-employee

C. Organization model
   - hr_org_units = canonical HR org; departments = legacy (users/RBAC)
   - no new HR on departments; no delete departments now

D. Leave model
   - leave_requests = future canonical; hr_employee_leaves = legacy
   - no new APIs on legacy; no production leave_requests until migration drift fixed

E. Employee status
   - hr_employee_statuses = catalog; employees.status = text (unenforced)
   - no new status logic until FK/enforced code phase

F. Contracts / types
   - hr_contract_types = catalog; contract_type on contracts = text
   - no new contract-type logic until alignment phase

G. Payroll / attendance
   - extend existing; no rebuild; no duplicate tables

H. HR services / forms / workflows
   - hr_services + forms + workflows = integration path; no parallel self-service

I. Permissions
   - no HR fine-grained permissions now; stabilize models first
   - self_service.view gap = known issue only (no fix in P18-A)

J. API structure
   - hr.ts monolith acknowledged; no split until tests + refactor plan

K. Future ERP
   - HR first domain under workspace_id; Finance/etc. reuse same boundary
   - no new tenant/workspace model per domain

DO-NOT-DUPLICATE RULES (SUMMARY)
================================================================================
See p18-a-do-not-duplicate-list.md. Highlights:

  - No new users/workspace/tenant models for HR
  - No new leave APIs on hr_employee_leaves
  - No parallel self-service or permission system
  - No hr.ts refactor in P18-A
