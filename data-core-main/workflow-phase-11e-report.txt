    All four governance layers described in a single static registry

  ✓ Compliance-readiness closure
    buildGovernanceReadiness() derives a GovernanceReadinessProfile with
    production_ready / ready / partial / not_ready classification from the
    combined topology + boundary summary. Fail-closed: any leak → not_ready.

2. GOVERNANCE TOPOLOGY ARCHITECTURE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

2.1  Design Decision: Static + Runtime Consolidation (No New Tables)

  P11-E introduces NO new database tables. The topology is built from:

    STATIC:  GOVERNANCE_LAYERS registry — 4 GovernanceLayerDescriptor entries
             defined at module load time. These describe what each layer IS
             (its type, role, db tables, declared boundary properties).

    RUNTIME: DB counts passed in by the route handler — count(*) from
             compliance_audit_chains and governance_workflow_actions, plus
             per-row status + escalationLevel for the readiness route.
             These describe what each layer HAS (runtime activity evidence).

  The pure engine receives the runtime data as parameters and returns a
  fully self-contained topology profile. No DB access inside the engine.

2.2  Architectural Position

  P11-E sits ABOVE P11-D as a meta-observability layer:

  P11-A  integrity_layer   record     — immutable hash-linked audit chains
  P11-B  policy_layer      detect     — 8-policy violation evaluation engine
  P11-C  workflow_layer    orchestrate — 6-state human-governed lifecycle
  P11-D  analytics_layer   observe    — 13-metric effectiveness intelligence
  ─────────────────────────────────────────────────────────────────────────
  P11-E  consolidation     topology   — cross-layer boundary + readiness map

  P11-E does not re-evaluate policies (P11-B), does not read workflow action
  content (P11-D does that), and does not access the audit chain content (P11-A
  does that). It reads only:
    - Row COUNTS (not content) from P11-A and P11-C tables
    - workflowStatus + escalationLevel columns for critical-unresolved count
  Everything else is derived from static layer descriptors.

3. GOVERNANCE LAYER CLASSIFICATION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

3.1  GovernanceLayerType — 4 canonical types

  type GovernanceLayerType =
    | "integrity_layer"    // P11-A: immutable record keeper
    | "policy_layer"       // P11-B: violation detector
    | "workflow_layer"     // P11-C: human-governed lifecycle orchestrator
    | "analytics_layer";   // P11-D: read-only effectiveness observer

3.2  GovernanceLifecycleRole — what each layer does

  type GovernanceLifecycleRole =
    | "record"       // P11-A: inserts immutable audit records
    | "detect"       // P11-B: evaluates rules and emits violations
    | "orchestrate"  // P11-C: manages human-triggered state transitions
    | "observe";     // P11-D: reads and computes analytics

3.3  GovernanceBoundaryProperty — 7 verifiable invariants

  type GovernanceBoundaryProperty =
    | "append_only"     // inserts only; never UPDATE or DELETE
    | "read_only"       // SELECT only; no writes at all
    | "human_governed"  // every state change requires a human operatorId
    | "deterministic"   // same inputs → same outputs always
    | "fail_closed"     // ambiguity defaults to strictest result
    | "no_enforcement"  // outputs are advisory only
    | "no_ai";          // classification is rule-based, no ML

3.4  GOVERNANCE_LAYERS Static Registry

  ┌───────┬─────────────────────┬─────────────────┬──────────────────────────┐
  │ Layer │ Type                │ Role            │ Key Boundary Properties   │
  ├───────┼─────────────────────┼─────────────────┼──────────────────────────┤
  │ P11-A │ integrity_layer     │ record          │ append_only, deterministic│
  │       │                     │                 │ fail_closed, no_enforcement│
  │       │ DB: compliance_     │                 │ no_ai                    │
  │       │     audit_chains    │                 │                          │

  │       │                     │                 │ fail_closed, no_enforcement│
  │       │ DB: (none)          │                 │ no_ai                    │
  │       │ pure engine         │                 │                          │
  ├───────┼─────────────────────┼─────────────────┼──────────────────────────┤
  │ P11-C │ workflow_layer      │ orchestrate     │ human_governed,           │
  │       │                     │                 │ deterministic, fail_closed│
  │       │ DB: governance_     │                 │ no_enforcement, no_ai    │
  │       │     workflow_actions│                 │                          │
  ├───────┼─────────────────────┼─────────────────┼──────────────────────────┤
  │ P11-D │ analytics_layer     │ observe         │ read_only, deterministic  │
  │       │                     │                 │ fail_closed, no_enforcement│

  All 4 layers declare: no_enforcement, no_ai, deterministic, fail_closed.
  These 4 properties are universal across the entire governance stack.

  Differences between layers:
    P11-A alone declares append_only  — it is the only persistent INSERT layer.
    P11-C alone declares human_governed — state changes require a human actor.
    P11-B and P11-D declare read_only — they never touch the DB.
    P11-A and P11-C do NOT declare read_only (they write to the DB).
    No layer declares both append_only and read_only simultaneously.

4. BOUNDARY VERIFICATION SEMANTICS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

4.1  GovernanceBoundaryStatus — 4-state result

  type GovernanceBoundaryStatus =
    | "verified"               // all expected properties confirmed
    | "warning"                // advisory concerns, not blocking
    | "boundary_leak_detected" // one or more expected properties missing
    | "incomplete";            // insufficient data to verify

4.2  verifyLayerBoundary() — fail-closed algorithm

  function verifyLayerBoundary(
    layer:              GovernanceLayerDescriptor,
    expectedProperties: GovernanceBoundaryProperty[],
    additionalWarnings: string[] = [],
  ): GovernanceBoundaryVerification

  Evaluation order (first match wins):

  ┌───────────────────────┬──────────────────────────────────────────────────┐
  │ Result                │ Condition                                        │
  ├───────────────────────┼──────────────────────────────────────────────────┤
  │ boundary_leak_detected│ Any expected property is NOT declared by layer   │
  │ warning               │ No missing properties BUT warnings list is set   │
  │ verified              │ All expected properties present, no warnings     │
  └───────────────────────┴──────────────────────────────────────────────────┘

  Fail-closed: boundary_leak_detected takes precedence over warnings.
  A layer can have warnings AND a leak — the result is always boundary_leak_detected.

4.3  GovernanceBoundaryVerification fields

  interface GovernanceBoundaryVerification {
    layerId:            string;
    layerType:          GovernanceLayerType;
    boundaryStatus:     GovernanceBoundaryStatus;
    verifiedProperties: GovernanceBoundaryProperty[];  // expected + present
    missingProperties:  GovernanceBoundaryProperty[];  // expected but absent
    warnings:           string[];                      // advisory messages
  }

4.4  verifiedProperties vs missingProperties

  These two arrays are always mutually exclusive and always sum to
  expectedProperties.length. They allow callers to see exactly which properties
  were verified and which caused a leak without re-running the check.

4.5  Warning generation rules used in buildGovernanceTopology()

  P11-A: emits a warning when auditRecordsTotal === 0:
    "No audit records yet — P11-A table is empty. Layer boundary verified but
     no runtime activity."

  P11-C: emits a warning when workflowsTotal === 0:
    "No workflow rows yet — P11-C table is empty. Layer boundary verified but
     no runtime activity."

  These are structural warnings (the layer exists and is correctly configured)
  not boundary leaks (the layer's declared properties still match expectations).
  A freshly deployed platform with no data should be "warning" not "verified",
  because the runtime evidence of correct operation is not yet present.

4.6  Cross-Layer Enforcement Check

  A synthetic CROSS descriptor is built at runtime with boundaryProperties
  populated only from properties that ALL 4 real layers declare:
    - no_enforcement (if all 4 layers declare it)
    - no_ai         (if all 4 layers declare it)

  verifyLayerBoundary(CROSS, ["no_enforcement", "no_ai"]) is then called.
  Since all current layers declare both properties, this check returns "verified".
  If any future layer were added without no_enforcement, the check would return
  boundary_leak_detected, triggering a governance_readiness_confirmed WARN event.

5. LIFECYCLE CONTINUITY MODEL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

5.1  GovernanceLifecycleCoverage

  interface GovernanceLifecycleCoverage {
    auditRecordsTotal:    number;   // count(*) from compliance_audit_chains
    workflowsTotal:       number;   // count(*) from governance_workflow_actions
    activeWorkflows:      number;   // non-terminal workflows
    resolvedWorkflows:    number;   // resolved + dismissed
    criticalUnresolved:   number;   // escalationLevel=critical AND non-terminal
    policyLayerActive:    boolean;  // always true (pure engine, no data gate)
    analyticsLayerActive: boolean;  // always true (pure engine, no data gate)
    totalLayers:          number;   // always 4
    activeLayers:         number;   // 2–4 depending on DB state
    coverageScore:        GovernanceCoverageScore;
  }

5.2  Active Layer Counting

  Base: P11-B (pure engine) + P11-D (pure engine) = 2 always active.
  +1 if auditRecordsTotal > 0  → P11-A has runtime activity
  +1 if workflowsTotal > 0     → P11-C has runtime activity

  activeLayers ∈ {2, 3, 4}.

5.3  GovernanceCoverageScore — 4-tier classification

  type GovernanceCoverageScore =
    | "minimal"       // activeLayers / 4 < 0.25    → impossible in current model (floor is 0.5)
    | "partial"       // 0.25 ≤ fraction < 0.75     → activeLayers ∈ {2} → 0.5 → "partial"
    | "substantial"   // 0.75 ≤ fraction < 0.90     → activeLayers = 3 → 0.75 → "substantial"
    | "comprehensive"; // fraction ≥ 0.90            → activeLayers = 4 → 1.0 → "comprehensive"

  In practice with the current 4-layer stack:
    activeLayers = 2 → fraction = 0.50 → "partial"
    activeLayers = 3 → fraction = 0.75 → "substantial"
    activeLayers = 4 → fraction = 1.00 → "comprehensive"

5.4  classifyCoverageScore() — deterministic, fail-closed

  Ties round DOWN (e.g., exactly 0.75 = "substantial", not "comprehensive").
  Thresholds checked in descending order so no condition is ever skipped.

5.5  computeLifecycleCoverage() — pure function, no I/O

  Takes 5 integer parameters (all non-negative) and returns a fully computed
  GovernanceLifecycleCoverage object. Never reads from DB. Never emits events.
  Called by each of the 3 P11-E routes with counts fetched from the DB.

6.1  GovernanceTopologyProfile

  interface GovernanceTopologyProfile {
    topologyId:            string;          // "gtopo:<nowMs>"
    governanceLayers:      GovernanceLayerDescriptor[];    // 4 entries
    integrityBoundaries:   GovernanceBoundaryVerification; // P11-A check
    policyBoundaries:      GovernanceBoundaryVerification; // P11-B check
    workflowBoundaries:    GovernanceBoundaryVerification; // P11-C check
    analyticsBoundaries:   GovernanceBoundaryVerification; // P11-D check

    observabilityCoverage: ObservabilityCoverage;
    lifecycleCoverage:     GovernanceLifecycleCoverage;
    evaluatedAt:           Date;

6.2  topologyId Format

  "gtopo:<evaluatedAt.getTime()>"
  Example: "gtopo:1747310400000"

6.3  buildGovernanceTopology() — pure function

  Arguments:
    lifecycleCoverage: GovernanceLifecycleCoverage   (from computeLifecycleCoverage)
    now:               Date                           (reference timestamp)

  Steps:
    1. Look up P11-A, P11-B, P11-C, P11-D from GOVERNANCE_LAYERS by layerId
    2. verifyLayerBoundary(layerA, expected_A, runtime_warnings_for_A)
    3. verifyLayerBoundary(layerB, expected_B)
    4. verifyLayerBoundary(layerC, expected_C, runtime_warnings_for_C)
    5. verifyLayerBoundary(layerD, expected_D)
    6. Build synthetic CROSS descriptor and run enforcement check
    7. buildObservabilityCoverage() — static map of 16 event types
    8. Assemble GovernanceTopologyProfile from all of the above

  All 8 steps are synchronous. The function performs no I/O.

7. GOVERNANCE READINESS PROFILE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

7.1  GovernanceReadinessStatus — 4-tier result

  type GovernanceReadinessStatus =
    | "not_ready"       // boundary_leak_detected in any layer → critical gap
    | "partial"         // incomplete verifications present
    | "ready"           // all boundaries verified or warning, coverage may be partial
    | "production_ready"; // all verified + substantial/comprehensive coverage
                          //   + observability complete

7.2  buildGovernanceReadiness() — fail-closed algorithm

  ┌──────────────────┬──────────────────────────────────────────────────────┐
  │ Status           │ Condition                                            │
  ├──────────────────┼──────────────────────────────────────────────────────┤
  │ not_ready        │ Any boundary_leak_detected OR criticalGaps.length > 0│
  │ partial          │ Any incomplete boundaries                            │
  │ production_ready │ All verified + (substantial OR comprehensive)        │
  │                  │   + observabilityComplete == true                    │
  │ ready            │ All verified or warning (no leaks, no incomplete)    │
  └──────────────────┴──────────────────────────────────────────────────────┘

7.3  GovernanceReadinessProfile fields

  interface GovernanceReadinessProfile {
    readinessId:           string;                            // "gready:<nowMs>"
    overallStatus:         GovernanceReadinessStatus;
    layerReadiness:        Record<string, GovernanceBoundaryStatus>;
    criticalGaps:          string[];   // one entry per boundary_leak_detected layer
    readinessNotes:        string[];   // warnings, runtime notes, coverage gaps
    coverageScore:         GovernanceCoverageScore;
    lifecycleCoverage:     GovernanceLifecycleCoverage;
    observabilityComplete: boolean;
    evaluatedAt:           Date;
  }

7.4  readinessNotes Population

  Sources (all advisory, not blocking):
    - Per-boundary warnings from GovernanceBoundarySummary.byLayer
    - auditRecordsTotal === 0 → P11-A has no runtime data yet
    - workflowsTotal === 0    → P11-C has no runtime data yet
    - criticalUnresolved > 0  → N critical violations remain open (from P11-D)
    - observabilityComplete = false (impossible with current 4-layer stack)

7.5  GovernanceBoundarySummary

  interface GovernanceBoundarySummary {
    summaryId:           string;    // "gbsum:<nowMs>"
    totalLayers:         number;    // always 5 (4 layers + 1 cross-layer check)
    verifiedLayers:      number;
    warningLayers:       number;
    leakDetectedLayers:  number;
    incompleteLayers:    number;
    overallStatus:       GovernanceBoundaryStatus;  // worst across all 5
    byLayer:             GovernanceBoundaryVerification[];
    evaluatedAt:         Date;
  }

  overallStatus is the worst across all 5 verifications:
    boundary_leak_detected > incomplete > warning > verified

8. GOVERNANCE OBSERVABILITY ADDITIONS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

           → buildBoundarySummary()

GET /platform/governance/readiness
  Auth:    super_admin

           SELECT workflowStatus + escalationLevel from workflow_actions

           → buildGovernanceTopology()
           → buildBoundarySummary()
           → buildGovernanceReadiness()
           → emit governance_readiness_confirmed (×1, WARN if any leak)
  Returns: { readiness: GovernanceReadinessProfile,
             topology:  GovernanceTopologyProfile,
             summary:   GovernanceBoundarySummary }
  Note:    Most complete route — returns all three artifacts in one response.
           Suitable for a governance dashboard that needs the full picture.

12. REMAINING GOVERNANCE GAPS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

P11-E is intentionally a pure consolidation visibility layer — no forecasting,
no enforcement, no automated policy changes.

  1. NO HISTORICAL TOPOLOGY SNAPSHOTS
     buildGovernanceTopology() always reflects current DB state. There is no
     persistent record of prior topology evaluations.
     Gap: a governance_topology_snapshots table (written periodically via cron
     or on each readiness call) would enable "was our governance health better
     last week?" comparisons.

  2. COVERAGE SCORE FLOOR IS 50% (NEVER "MINIMAL")
     With 2 pure-engine layers (P11-B, P11-D) always active, activeLayers is
     always at least 2/4 = 0.50 = "partial". The "minimal" tier (< 0.25) is
     unreachable in the current 4-layer model.
     Implication: the score cannot signal a truly minimal governance state even
     if both persisted layers have no data. Coverage "partial" includes both
     "no DB data yet" and "audit chains exist but no workflows".
     Gap: weighting layers by operational significance (e.g., P11-C as heavier
     than P11-D since workflow data proves the system is in active use) would
     produce a more meaningful coverage signal.

  3. BOUNDARY PROPERTY DECLARATIONS ARE SELF-REPORTED
     GOVERNANCE_LAYERS is a static registry maintained by code. A layer's
     declared boundary properties (e.g., read_only, append_only) are not
     verified against the actual DB or engine code — they are assertions, not
     proofs.
     Gap: integration-level tests that actually attempt a write against P11-D's
     engine functions, or a mutation against P11-A's table via the engine, would
     provide structural proof of the declared boundary properties rather than
     trust in the declarations alone.

  4. CRITICAL_UNRESOLVED COUNT IN TOPOLOGY ROUTE IS AN APPROXIMATION
     GET /platform/governance/topology uses total active workflows as the
     criticalUnresolved proxy (it doesn't filter by escalationLevel because
     it only SELECTs workflowStatus). The readiness route correctly filters by
     both workflowStatus AND escalationLevel.
     Gap: the topology route could also SELECT escalationLevel to give an accurate
     criticalUnresolved count instead of using total active as a conservative proxy.

  5. NO WORKSPACE-SCOPED TOPOLOGY
     All three P11-E routes are platform-wide (no workspaceId filter).
     A workspace admin cannot see the governance topology for their specific
     workspace — only the super_admin can see the platform view.
     Gap: workspace-scoped topology could be useful for workspace admins who
     want to know which governance layers have data for their workspace.

  6. GOVERNANCE_LAYERS REGISTRY IS NOT EXTENSIBLE AT RUNTIME
     Adding a P12-A or future governance layer requires a code change to
     GOVERNANCE_LAYERS. There is no mechanism for a super_admin to register
     a custom governance layer through the API.
     Gap: a governance_layer_registry DB table would allow runtime registration
     of new layers with their declared properties, enabling the topology engine
     to be data-driven rather than code-driven.

  7. NO FRONTEND GOVERNANCE INTELLIGENCE DASHBOARD
     All three P11-E routes are live but the ops-platform React frontend has
     no super admin pages for:
       - Governance topology visualization (layer graph with boundary status badges)
       - Boundary verification table (per-layer status, verified/missing properties)
       - Readiness profile (overallStatus, criticalGaps, readinessNotes)
     Gap: super admin UI at /super-admin/governance/topology would complete the
     human-facing governance intelligence surface.

  8. OBSERVABILITY COVERAGE IS COMPUTED FROM STATIC MAP (P11-A THROUGH P11-D)
     buildObservabilityCoverage() returns a hardcoded map of 16 event types.
     P11-E's own 4 events are NOT included in this map (the map covers P11-A to
     P11-D, not P11-E).
     Gap: adding P11-E's events to the map would give 20 total event types and
     a more complete observability picture. The current "16 events across 4
     layers" is technically accurate but does not count P11-E's contribution.

  9. BOUNDARY WARNINGS DO NOT HAVE STRUCTURED CODES
     Warnings in GovernanceBoundaryVerification are plain strings:
     "No audit records yet — P11-A table is empty…"
     This makes programmatic processing of warnings difficult.
     Gap: adding a warning code (e.g., "P11A_NO_RUNTIME_DATA") alongside the
     human-readable message would enable frontend UI to render localized warning
     labels without string parsing.

  10. NO TOPOLOGY DIFF / CHANGE DETECTION
      There is no mechanism to compare two topology profiles (current vs. prior
      snapshot) and surface what changed between them.
      Gap: a diffTopologies(prev, next) function that returns changed boundary
      statuses, added/removed layers, and coverage score changes would enable
      "what changed in our governance posture since last audit" queries.

SAFETY BOUNDARIES (FINAL — MUST REMAIN RESPECTED IN ALL FUTURE PHASES)
─────────────────────────────────────────────────────────────────────────────
⚠ READ-ONLY ANALYTICS (P11-D, P11-E):
    All analytics and topology functions must SELECT only.
    No INSERT/UPDATE/DELETE permitted in the analytics or topology engines.

⚠ APPEND-ONLY INTEGRITY (P11-A):
    compliance_audit_chains accepts INSERT only.
    No UPDATE or DELETE of audit chain records is ever permitted.
    Any future feature that modifies an audit record violates P11-A.

⚠ HUMAN-GOVERNED LIFECYCLE (P11-C):
    Every governance_workflow_actions state transition must carry an explicit
    operatorId. No automated state change is permitted.
    P11-E must never call any P11-C transition function.

⚠ NO AUTO-ENFORCEMENT (ALL LAYERS):
    No analytics result, boundary status, or readiness profile may trigger:
      - automatic workflow escalation (P11-C)
      - automatic policy modification (P11-B)
      - automatic audit record mutation (P11-A)
      - automatic external notification or regulator integration
    The governance_readiness_confirmed WARN event is a log signal only.

⚠ NO AI (ALL LAYERS):
    All classification is rule-based and deterministic.
    No ML models, probabilistic scoring, or external AI services.
    "fail_closed" and "deterministic" are inviolable properties of the stack.

⚠ FAIL-CLOSED (ALL LAYERS):
    When classifying boundary status, coverage, readiness, or effectiveness:
    worst condition wins; conservative result is always chosen on ambiguity.
    This invariant must be maintained if thresholds are ever adjusted.

⚠ DETERMINISM (ALL LAYERS):
    same inputs + same `now` → same JSON output.
    No environment-dependent branching, no randomness, no stale-read caching
    inside pure engine functions.
