CHANGE 2 — seed/forms.ts: fix Leave Request workflowEvent
  File: artifacts/api-server/src/seed/forms.ts
  What:
    Changed workflowEvent for the "Leave Request" seed template from:
      "leave.requested"   → "hr.form.submitted"
  Why "leave.requested" was wrong:
    "leave.requested" (EVENT_TYPES.LEAVE_REQUESTED) is a canonical domain event
    with a typed payload (LeaveRequestedPayload) that requires:
      leaveRequestId:  number   (FK to leave_requests table — doesn't exist)
      employeeUserId:  number   (camelCase, from req.userId)
      leaveType:       string   (validated enum, camelCase)
      startDate:       string   (ISO date, camelCase)
      endDate:         string   (ISO date, camelCase)
      daysRequested:   number   (computed: endDate - startDate)
    When forms.ts fires eventDispatcher.dispatch() with event="leave.requested",
    the actual payload is { leave_type, start_date, end_date, ... } (snake_case,
    uncomputed, no leaveRequestId).  This is a silent contract violation invisible
    to TypeScript because eventDispatcher.dispatch() takes Record<string, unknown>.
  Why "hr.form.submitted" is correct:
    It is a WorkflowEngine HINT string, not a bus event type.  It routes the
    WorkflowEngine to the right workflow_definitions row for HR leave flows.
    It carries no false semantic promise about payload shape.
  Comment added explaining the full rationale with forward pointer to the
  future dedicated /hr/leave-requests route.


CHANGE 3 — seed/forms.ts: fix Expense Reimbursement workflowEvent
  File: artifacts/api-server/src/seed/forms.ts
  What:
    Changed workflowEvent for the "Expense Reimbursement" seed template from:
      "approval.requested"   → "approvals.form.submitted"
  Why "approval.requested" was wrong:
    "approval.requested" is a legacy name for the canonical domain event

      approvalId:         number  (FK to approvals table)

      2. Admin or workflow creates POST /approvals → approval.created (domain)
    This form must never shortcut to "approval.requested" directly.
  Comment added explaining the two-step saga pattern.

    private async handleEvent(payload: EventPayload): Promise<void> {
      // Primary match: exact event type
      let definitions = await db.select()...
        .where(eq(triggerEvent, payload.event));

      // Secondary match for form.submitted: match on workflowEventHint
      if (payload.event === "form.submitted") {
        const hint = (payload.data as Record<string, unknown>)?.workflowEventHint;
        if (hint && typeof hint === "string") {
          const hintDefs = await db.select()...
            .where(eq(triggerEvent, hint));
          // deduplicate by id
          const existingIds = new Set(definitions.map(d => d.id));
          for (const d of hintDefs) {
            if (!existingIds.has(d.id)) definitions.push(d);
          }
        }
      }
      // rest of method unchanged...
    }

SECTION 5 — WORKFLOWEVENTHINT SEMANTICS — FINAL DEFINITION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Source:  formDefinitionsTable.workflowEvent column (free text, set by admin/seed)
  Carrier: FormSubmittedPayload.workflowEventHint (optional string)
  Purpose: WorkflowEngine secondary trigger key — enables per-form workflow routing

  Correct values (naming convention established):
    Module prefix + ".form.submitted" for generic module forms:
      "hr.form.submitted"         — HR module forms (leave, transfers, etc.)
      "approvals.form.submitted"  — Approval-routed forms (expense, etc.)
      "system.form.submitted"     — System/IT/general forms
    Service slug pattern for HR service forms (generated by hr.ts):
      "hr.{slug}.submitted"       — e.g. "hr.annual-leave.submitted"

  Incorrect values (now documented and prevented in seed data):
    "leave.requested"             — is a domain event type, NOT a hint
    "approval.requested"          — is a legacy domain event type, NOT a hint

  Bus listeners (activity.ts, notifications-bus.ts):
    Must NOT branch on workflowEventHint for routing logic.
    Use owningModule and answers fields instead.
    workflowEventHint is WorkflowEngine-only routing metadata.

  Future: When the dedicated /hr/leave-requests route is built, it will emit
  EVENT_TYPES.LEAVE_REQUESTED with a proper LeaveRequestedPayload — completely
  independent of this hint system.  The Leave Request form (hr.form.submitted)
  and the leave.requested domain event are separate tracks.
