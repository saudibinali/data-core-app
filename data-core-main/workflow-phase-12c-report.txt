SECTION 2 — VIOLATION SEVERITY VISUAL MODEL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Source: VIOLATION_SEVERITY_MAP (governance-console-config.ts)

  Severity       Tier          Order  Badge colour    Icon             Description
  ─────────────────────────────────────────────────────────────────────────────────
  informational  info          0      slate            Info             "Observation-level deviation — no immediate action needed."
  low            low           1      blue             Info             "Minor policy deviation — log and monitor."
  medium         medium        2      amber            AlertTriangle    "Moderate deviation — warrants investigation."
  high           high          3      orange           AlertTriangle    "Significant deviation — prompt review recommended."
  critical       critical      4      red              ShieldX          "Severe deviation — immediate review required."

Design principles:
  - Order index 0–4 drives deterministic sort (highest = 4, lowest = 0)
  - Display order is always highest-first (critical → informational)
  - Left border on violation rows mirrors badge colour
  - No label uses guilt, fault, legal, illegal, disciplinary wording
  - severity = "critical" never triggers a UI that suggests legal exposure
  - SeverityBadge includes an icon so colour is not the only signal

2.1  VIOLATION_SEVERITY_ORDER_DESC

  ["critical", "high", "medium", "low", "informational"]

  Used by:
  - The violation list sort (highest first, then detectedAt desc)
  - The policy registry table sort (highest defaultSeverity first)
  - The severity summary card row ordering

2.2  Severity summary cards (interactive)

  5 cards rendered in VIOLATION_SEVERITY_ORDER_DESC sequence.
  Each shows: severity label + count for that severity.
  Clicking a card toggles the severity filter (ring highlight on active).
  This is the fastest path from overview → filtered list.
  No cards are hidden when count = 0.

SECTION 5 — POLICY REGISTRY VISIBILITY
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The policy registry table shows all governance policies from GET /policies.

5.1  Columns (POLICY_REGISTRY_COLUMNS)

  Policy ID       Mono, 28-char width
  Name            Full policy name, flex-1, with description as tooltip/sub-text
  Severity        defaultSeverity SeverityBadge
  Status          Active (emerald) or Inactive (secondary) badge
    — derived from policy.enabled !== false && policy.active !== false
    — no enable/disable toggle
  Violations      Count of violations from the current violation list (derived client-side)
    — red if count > 0, muted if 0
  Last Detected   lastDetectedAt formatted date, hidden on mobile

5.2  Sort order

  Policies are sorted by defaultSeverity descending (using VIOLATION_SEVERITY_MAP.order).
  Critical-default policies appear first, informational last.
  Ties broken by original API order (stable sort).

5.3  Violation count derivation

  A Map<policyId, count> is computed from the currently loaded violation list
  using useMemo. This reflects the active workspace filter — when a workspace
  filter is active, only violations for that workspace contribute to the counts.

5.4  No policy controls

  No enable/disable switch.
  No edit form.
  No delete button.
  No "create policy" trigger.
  Policies are read-only metadata for review only.

SECTION 6 — FILTER BEHAVIOR
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Three filter axes, two mechanisms:

6.1  Severity filter (client-side)
  Select: All / Critical / High / Medium / Low / Informational
  Filters the in-memory violation list (filteredViolations useMemo).
  Also togglable by clicking severity summary cards.
  data-testid: filter-severity

6.2  Violation type filter (client-side)
  Select: All Types + 8 violation type labels
  Filters by violationType field on each violation.
  data-testid: filter-violation-type

6.3  Workspace ID filter (server-side via hook param)
  Text input + Enter/Search button.
  applyWorkspaceFilter() sets activeWorkspaceId state.
  useGovernanceViolations(activeWorkspaceId) is called with the new value.
  This triggers a fresh API call: GET /api/platform/governance/violations/:workspaceId
  (when set) or GET /api/platform/governance/violations (when cleared).
  data-testid: filter-workspace-id

  Active workspace indicator shown below filters when a workspace ID is set:
  "Scoped to workspace: {activeWorkspaceId}"

6.4  Clear all filters
  "Clear (N)" button appears when any filter is active.
  Clears all three filter states simultaneously.
  data-testid: clear-violation-filters

6.5  No hidden critical violations
  The two client-side filters (severity, type) are ANDed, never ORed.
  A critical violation is only excluded from the list if its violationType
  does not match the active type filter — this is intentional narrowing, not hiding.
  When severity filter = "critical", ONLY critical violations are shown (positive filter).
  There is no filter combination that can positively exclude ALL critical violations
  from an operator-initiated narrow — the operator would have to choose a type filter
  that excludes them, which is a legitimate investigation scoping action.

SECTION 7 — FORENSIC CONTEXT LINKING
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Every violation that carries an entityId gets a "Forensic Context" panel
inside its expanded detail view.

7.1  What the panel shows

  - "Forensic Context" heading with blue Read-Only badge
  - FORENSIC_CONTEXT_GUIDANCE.description:
    "Use the entity ID below to investigate the full audit trail on the Audit Integrity page."
  - Entity ID in a code block (font-mono, bg-muted)
  - "Copy Entity ID" button (navigator.clipboard, 1.5s feedback)
  - "Review Forensic Timeline" link to /super-admin/governance/audit-integrity
    (wouter <Link> — internal navigation only, no external href)

7.2  What the panel does NOT do

  - Does NOT auto-fetch the forensic timeline (avoids one API call per expanded row)
  - Does NOT auto-navigate — the link is a manual action by the operator
  - Does NOT pass the entityId as a query param (the Audit Integrity page has its own
    entity ID input — the operator copies and pastes; copy button is provided)
  - Does NOT create, repair, or modify any audit records

7.3  Why manual copy rather than auto-link with query param

  wouter's navigation model and the current Audit Integrity page use local React state
  (useState) for the entity ID input, not URL search params. Implementing query-param
  based pre-fill would require modifying the Audit Integrity page router integration.
  The copy + navigate pattern achieves the same result with zero additional complexity
  and zero risk of URL injection. This can be upgraded to a query-param pre-fill in P12-D
  or when the Audit Integrity page migrates its entity ID state to the URL.

─────────────────────────────────────────────────────────────────────────────
  The "Review Forensic Timeline" link navigates to the Audit Integrity page
  but does not pre-fill the entity ID input. The operator must copy and paste.
  Future: migrate the Audit Integrity entity ID state to a URL search param
  (?entityId=...) and update the forensic context link accordingly.

─────────────────────────────────────────────────────────────────────────────
C)  PAGINATION
─────────────────────────────────────────────────────────────────────────────
  No pagination on the violations list or policy registry.
  The API is called with default limits. Future: offset/limit controls.

─────────────────────────────────────────────────────────────────────────────
D)  VIOLATION TIMELINE MINI-VIEW
─────────────────────────────────────────────────────────────────────────────
  The expanded violation row shows linked audit chain IDs but does not
  embed a GovernanceForensicTimeline component inline (to avoid one API
  call per expanded violation row). Future: a lazy-loaded "Show Audit Trail"
  toggle inside the expanded row that fires the forensic timeline query
  only when explicitly opened.

─────────────────────────────────────────────────────────────────────────────
E)  VIOLATION HISTORY / RECURRENCE
─────────────────────────────────────────────────────────────────────────────
  No display of how many times the same violation has recurred across
  detection cycles. Future: violation recurrence count + first seen date.

─────────────────────────────────────────────────────────────────────────────
F)  WORKSPACE SELECTION DROPDOWN
─────────────────────────────────────────────────────────────────────────────
  The workspace filter is a free-text input for the ID only. A workspace
  name dropdown (populated from GET /workspaces) would be more usable.
