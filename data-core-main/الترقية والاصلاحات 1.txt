0. مبادئ الترقية بدون انهيار (Non-Disruptive Upgrade Principles)
================================================================================

يجب أن يلتزم كل فيز بهذه القواعد لضمان سير العمل الحالي:

  1. **Expand → Migrate → Contract**
     • أضف المسار/الجدول/الحقل الجديد أولاً
     • انقل البيانات/الحركة تدريجياً
     • أزل القديم فقط بعد cutover flag + فترة مراقبة

  2. **Backward-compatible API**
     • لا تغيّر response shape لمسارات production دون versioning
     • مسارات جديدة تحت /api/v2 أو feature flag — القديم يبقى حتى deprecation

  3. **Feature flags / cutover flags**
     • استخدم workspace settings أو platform-runtime registry الموجود:
       artifacts/api-server/src/routes/platform-runtime.ts
       lib/db/src/schema/hr-import-runtime.ts (rollout patterns)
       workforce-health cutover-readiness endpoints

  4. **Database migrations آمنة**
     • additive migrations فقط في production (ADD COLUMN nullable)
     • destructive changes في فيز لاحق بعد backfill
     • backup PG إلزامي قبل كل deploy
     • مرجع: artifacts/api-server/src/seed/migrate.ts

  5. **Blue/Green أو Rolling deploy**
     • Docker: api service جديد + healthz قبل switch
     • PM2: cluster reload graceful
     • مرجع: deploy/docker-compose.yml، deploy/ecosystem.config.cjs

  6. **Rollback plan لكل فيز**
     • image/tag سابق محفوظ
     • migrations reversible أو restore من snapshot

  7. **Smoke checklist بعد كل deploy**
     • login → ticket → HR employee → leave request → approval inbox
     • provisioning user (إن مُفعّل)

  8. **لا Big Bang refactor**
     • hr.ts (~161 handler) يُقسّم module-by-module وليس دفعة واحدة

9. الخلاصة
================================================================================

  منصة Data Core تمتلك **أساساً enterprise حقيقياً** (multi-tenant، HCM breadth،
  workflow engine، on-prem wizard، platform RBAC). الفجوات الرئيسية **تشغيلية
  وهندسية** وليست في الرؤية: CI، migrations، OpenAPI drift، dual domains،

    • فيزات صغيرة قابلة للrollback

    • contract-first قبل توسعة product
    • اختبارات عزل tenant إلزامية
    • observability قبل scale

  **الخطوة التالية الموصى بها:** تنفيذ F0 بالكامل قبل أي feature جديد —
  ثم F1 + F4 (أمان + provisioning) لأنهما الأكثر تأثيراً على المستخدمين الحاليين.
