Convert manual governance snapshot capture (POST /governance/snapshots/capture)
into a self-sustaining operational history lifecycle with automatic capture,
deterministic workspace iteration, and bounded retention pruning.

Prior to P7-B:
  • captureGovernanceSnapshot() existed (P7-A) but required an explicit API call.
  • governance_snapshots grew without bound — no pruning was ever executed.
  • Historical continuity depended entirely on human or automation triggering
    POST /governance/snapshots/capture for each workspace.

After P7-B:
  • GovernanceSnapshotScheduler polls every 5 minutes, captures all active
    workspaces automatically, and prunes raw rows older than 30 days.
  • Zero manual intervention required after server startup.
  • Retention lifecycle is deterministic and bounded — no unbounded growth.
  • The governance scheduler started at the same lifecycle stage as the
    workflow delay scheduler (P6-A), via runInitSequence().

Class: GovernanceSnapshotScheduler
File:  artifacts/api-server/src/lib/workflows/governance-scheduler.ts

Follows the WorkflowScheduler (P6-A) design exactly:

  SETTIMEOUT CHAIN (NOT setInterval):
    setTimeout(fn, CAPTURE_INTERVAL_MS).finally(() => _schedule())
    → Next cycle starts only AFTER the current one completes.
    → If a cycle takes 6 minutes and the interval is 5 minutes, the next
      cycle fires 5 minutes after the previous finished — not 1 minute later.

  NON-REENTRANT GATE (isCapturing flag):
    Prevents overlapping capture passes even if a cycle runs long.
    When the gate is held, a new cycle returns buildOverlapSkipResult()
    immediately with skippedOverlap=true and zero counts.

  IDEMPOTENT START:
    start() checks `if (this.running) return` — second call is a no-op.
    Safe to call from init-sequence.ts without checking server state.

  NON-DESTRUCTIVE STOP:
    stop() cancels the next scheduled timer and clears running=false.
    Any in-flight cycle completes naturally.
    Restart-safe: all state is in the DB — no in-memory timer IDs to recover.

  SINGLETON EXPORT:
    export const governanceScheduler = new GovernanceSnapshotScheduler();
    Mirror of workflowScheduler singleton from P6-A.

Key constants:
  CAPTURE_INTERVAL_MS        = 300,000 ms (5 min, derived from RECOMMENDED_CAPTURE_INTERVAL_MINUTES)
  RETENTION_RAW_DAYS         = 30         (aligned with RECOMMENDED_RETENTION_RAW_DAYS)
  PRUNE_MIN_AGE_HOURS        = 1          (min-age floor — see Retention Safety Rules)
  PRUNE_MAX_DELETE_PER_CYCLE = 500        (bounded DELETE per workspace per cycle)
  WORKSPACE_BATCH_SIZE       = 50         (max active workspaces per cycle)

WORKSPACE ITERATION STRATEGY
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

queryActiveWorkspaceIds(database, limit = 50):
  SELECT id FROM workspaces WHERE status = 'active' ORDER BY id ASC LIMIT 50

Design rationale:

  ONLY ACTIVE WORKSPACES:
    status='suspended' and status='disabled' workspaces are skipped.
    Suspended tenants are not generating workflow activity → no new governance
    data to capture.  Disabled workspaces should not accrue new history.

  DETERMINISTIC ORDER:
    ORDER BY id ASC ensures the scheduler always visits workspaces in the
    same sequence across restarts.  After process restart, the first cycle
    visits workspace 1 first — no randomness, no starvation.

    sortWorkspaceIds() is applied in-memory after the DB query as an
    additional guarantee (pure, injectable, testable).

  PAGINATION-READY BOUNDED BATCH:
    LIMIT 50 via WORKSPACE_BATCH_SIZE.  Remaining workspaces past position 50
    would be captured in a future multi-page implementation.
    At current scale (< 50 active workspaces per deployment), this covers all
    workspaces in a single cycle.

  SKIP DELETED WORKSPACES:
    The governanceSnapshotsTable has ON DELETE CASCADE on workspaceId FK →
    workspaces(id).  If a workspace is deleted, its snapshots are automatically
    removed.  The scheduler never encounters or processes deleted workspace IDs.

  NO DISTRIBUTED SHARDING:
    Single-process model.  No partition key, no worker affinity, no distributed
    coordinator.  Future multi-process deployments can use a guarded
    acquisition model (similar to P6-A execution resumption) if needed.

RETENTION ENFORCEMENT BEHAVIOR
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Pruning runs after every capture pass in the same cycle.  The same workspace
batch that received new snapshots is also pruned.

pruneSnapshotsOlderThan(workspaceId, cutoff, now, maxDelete, dryRun, database):

  Step 1 — SELECT eligible IDs:
    SELECT id FROM governance_snapshots
    WHERE workspace_id = ? AND captured_at < cutoff AND captured_at <= now
    LIMIT 500

  Step 2 — Dry-run check:
    If dryRun=true → return { deleted: 0, eligible: count } immediately.

  Step 3 — Bounded DELETE by primary key:
    DELETE FROM governance_snapshots WHERE id IN (...)
    Bounded to the SELECT result set — prevents unbounded single statement.

Why SELECT + DELETE by PK instead of DELETE WHERE ... LIMIT:
  Drizzle ORM does not support LIMIT on DELETE for PostgreSQL.
  The SELECT-then-DELETE-by-PK pattern is equivalent in correctness and
  achieves the same safety guarantee with an extra round-trip.

Cutoff computation (computePruneCutoff):
  retentionDate = now - 30 days
  minAgeDate    = now - 1 hour
  cutoff = min(retentionDate, minAgeDate)   ← more conservative / earlier

  Normal case (30d retention): cutoff = 30 days ago ✓
  Edge case (0d retention):    cutoff = 1 hour ago ✓ (min-age floor kicks in)

APPEND-ONLY GUARANTEES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The P7-A append-only contract is fully preserved through P7-B:

  ✓ captureGovernanceSnapshot() in the scheduler loop uses INSERT (P7-A Layer 3).
  ✓ pruneSnapshotsOlderThan() is the ONLY DELETE path in the codebase.
    It never deletes rows inside the retention window.
    It never issues UPDATE statements.
  ✓ Surviving rows remain immutable after pruning — no UPDATE is ever run.
  ✓ The scheduler does not modify workflow_executions or any other runtime table.
  ✓ Structural proof: the pruning WHERE clause requires capturedAt < cutoff
    (strictly less than) — rows at the boundary are preserved.
  ✓ isEligibleForPruning() uses strict less-than (< cutoff) — boundary row
    survives, consistent with the DB query.

What "append-only" means in practice under the P7-B retention lifecycle:
  • Rows 0–30 days old: immutable, never touched after INSERT.
  • Rows > 30 days old: eligible for pruning (deleted in bounded batches).
  • No row is ever mutated — only created or deleted.
  • A snapshot that existed at T=0 and is still within 30 days is bit-identical
    at T=30d.  No background job ever modifies its JSONB columns, severity,
    or any other field.

PRUNING SAFETY MODEL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Six safety rules enforced:

  RULE 1 — Never delete recent snapshots:
    PRUNE_MIN_AGE_HOURS = 1.  computePruneCutoff() is always at most 1 hour
    before now, regardless of the retentionDays value.  Even if retentionDays
    is accidentally set to 0, snapshots from the last hour are protected.

  RULE 2 — Never delete all history:
    Since PRUNE_MIN_AGE_HOURS > 0, the pruning cutoff is always strictly before
    now.  At any given moment, all snapshots captured within the last hour are
    guaranteed to survive.  Combined with the 5-minute capture interval,
    at least 12 snapshots are always preserved per active workspace.

  RULE 3 — Never prune future-dated rows:
    WHERE captured_at <= now in the SELECT query.  Snapshots with capturedAt
    in the future (clock skew, test data) are never returned by the
    eligibility query and cannot be deleted.

  RULE 4 — Bounded delete queries:
    LIMIT 500 (PRUNE_MAX_DELETE_PER_CYCLE) in the SELECT phase.
    The DELETE is then bounded to those IDs.  Prevents:
      • Lock contention from large in-flight DELETE statements.
      • Statement timeouts from runaway DELETE scans.
      • Backlog-burst DELETE after scheduler downtime.
    Residual rows beyond 500 are pruned in subsequent cycles.

  RULE 5 — Dry-run mode:
    dryRun=true: queries eligible count, returns PruneSingleWorkspaceResult
    with deleted=0 and eligible=N.  Zero DB mutations.
    Available via GovernanceSnapshotScheduler.runCaptureOnce(now, { dryRun: true }).

  RULE 6 — Per-workspace error isolation:
    If pruning fails for one workspace, the error is logged and the cycle
    continues with the next workspace.  One workspace's prune failure does
    not affect other workspaces or the capture pass.

Storage pressure thresholds:
  "low"      < 50,000  rows (~5 workspaces at 30d capacity)
  "medium"   < 150,000 rows (~17 workspaces at capacity)
  "high"     < 300,000 rows (~34 workspaces at capacity)
  "critical" ≥ 300,000 rows

Capacity reference:
  1 workspace × 5-min intervals × 30 days = 8,640 rows at steady state.
  50 workspaces at full capacity = 432,000 rows → "critical" pressure signal.
  With daily pruning, raw rows per workspace stays bounded at ≤ 8,640.

estimatedSnapshotsAtCapacity(intervalMin=5, retentionDays=30) = 8,640
  — pure helper for capacity planning displays.

REMAINING STORAGE GOVERNANCE GAPS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. WORKSPACE PAGINATION (not in P7-B):
   WORKSPACE_BATCH_SIZE = 50 covers current scale.  Deployments with > 50
   active workspaces will silently miss workspaces ranked > 50 by ID.
   A future multi-page iteration (page cursor in DB or memory) would fix this.

2. HOURLY / DAILY ROLLUP (not in P7-B):
   Raw rows older than 30 days are deleted, not aggregated.  There is no
   hourly or daily rollup table.  Long-term trends (> 30 days) are not available.
   The retention recommendation model (P7-A) defined 90-day hourly and 365-day
   daily tiers but no schema or migration implements them yet.

3. PRUNE CALLBACK / QUOTA ENFORCEMENT (not in P7-B):
   No mechanism notifies the platform owner when a workspace crosses a
   storage pressure threshold ("critical" = 300k+ rows).  A future webhook
   or notification event would close this gap.

4. PER-WORKSPACE RETENTION OVERRIDES (not in P7-B):
   All workspaces share RETENTION_RAW_DAYS = 30.  A workspace_settings column
   could allow enterprise tenants to configure longer raw retention.

5. SCHEDULER HEALTH ENDPOINT (not in P7-B):
   No API endpoint exposes scheduler status (last cycle time, failure count,
   storage metrics).  A future GET /governance/scheduler/status endpoint would
   surface CycleStats and StorageGovernanceMetrics to the super-admin dashboard.

6. CROSS-WORKSPACE STORAGE VIEW (super_admin — not in P7-B):
   No super_admin endpoint shows platform-wide snapshot volume or which
   workspaces are at/near storage capacity.

7. CONFIGURABLE INTERVAL (not in P7-B):
   CAPTURE_INTERVAL_MS is a compile-time constant.  A platform setting (or
   env var) for the capture interval would allow ops teams to tune the
   scheduler without redeploying.

    Every snapshot is immutable and timestamped.  Historical comparisons
    (e.g., "this week vs last week") can trust that past rows were not mutated.

    GOV-* codes are stored in alert_codes JSONB on every snapshot row.
