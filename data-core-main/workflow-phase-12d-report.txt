SECTION 4 — RESOLUTION CLASSIFICATION MODEL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Source: RESOLUTION_CLASSIFICATION_MAP (5 entries)

  Classification              Tier       Badge colour   Description
  ──────────────────────────────────────────────────────────────────────────────
  confirmed_violation         finding    red            "Investigation confirmed a policy deviation occurred."
  false_positive              cleared    emerald        "Investigation found no actual policy deviation."
  operational_exception       exception  amber          "Deviation was an approved operational exception."
  policy_gap                  gap        violet         "Investigation revealed an uncovered policy area."
  unresolved_pending_review   pending    slate          "Investigation is ongoing — resolution not yet recorded."

Design principles:
  - "confirmed_violation" uses tier "finding" — not "violation" — to avoid
    implying legal findings. It confirms a policy deviation, not legal fault.
  - "false_positive" uses tier "cleared" — visually positive (emerald) to
    signal this is a good outcome, not a problem.
  - No label contains "guilty", "illegal", "law", "verdict", or "fault".

ResolutionBadge is shown in the collapsed row header when a classification
is present, and again in the expanded detail view with its description.

SECTION 6 — LIFECYCLE TIMELINE COMPONENT
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

File: artifacts/ops-platform/src/components/governance/governance-workflow-lifecycle-timeline.tsx

6.1  Props

  interface GovernanceWorkflowLifecycleTimelineProps {
    events:          WorkflowLifecycleEvent[];
    compact?:        boolean;
    data-testid?:    string;
  }

  interface WorkflowLifecycleEvent {
    key:           WorkflowLifecycleEventKey;  // one of 7 lifecycle event keys
    label:         string;
    timestamp?:    string;                     // ISO date string or undefined
    actorId?:      string;
    note?:         string;
    evidenceRef?:  string;
  }

6.2  Lifecycle event slot ordering (WORKFLOW_LIFECYCLE_EVENT_ORDER)

  0. createdAt               → Play icon (blue)
  1. acknowledgedAt          → Eye icon (sky)
  2. investigationStartedAt  → Search icon (violet)
  3. escalatedAt             → ArrowUp icon (orange)
  4. resolvedAt              → CheckCircle2 icon (emerald)
  5. closedAt                → Archive icon (zinc)
  6. updatedAt               → Clock icon (muted)

  Order is fixed by the tuple index. buildLifecycleEvents() constructs the
  event array from the raw workflow object; the timeline sorts by index.

6.3  Spine visualisation

  Each event gets a dot + optional connecting line.
  Dot colour:
    recorded  → tier-specific colour (blue/sky/violet/orange/emerald/zinc/primary)
    not yet   → muted/bg-muted (dimmed)

  In full mode:
    All 7 slots rendered. Unrecorded = dimmed dot + "not recorded" italic text.
    Actor, note, evidence ref shown if present on recorded events.
    Footer: "N / 7 lifecycle events recorded — read-only"

  In compact mode (for future embedding):
    Only recorded events shown. No footer. No detail expansion.

6.4  buildLifecycleEvents(workflow) utility

  Exported function that maps a raw workflow Record<string, unknown> to
  WorkflowLifecycleEvent[]. Reads:
    createdAt, acknowledgedAt, investigationStartedAt, escalatedAt,
    resolvedAt, closedAt, updatedAt,
    initiatedBy, assignedOperatorId, escalatedBy, resolvedBy,
    escalationReason, resolutionNotes

  Deduplication: if updatedAt equals createdAt (nothing changed since init),
  the updatedAt slot is suppressed (timestamp → undefined) to avoid false
  "Last Updated" noise.

6.5  Empty state

  In full mode when zero events are recorded:
    Clock icon + "No lifecycle events recorded for this workflow."
    data-testid: {testId}-empty

  In compact mode: empty array produces an empty container (no icon).

6.6  Safety guarantees

  No acknowledge/escalate/resolve buttons.
  No mutation of any workflow state.
  Component is purely presentational (no internal query calls).
  Stable chronological ordering regardless of input order.

7.1  Evidence references

  GovernanceEvidenceReferenceList is reused from P12-C.
  Receives evidenceReferences[] from the workflow object (mapped to the
  standard EvidenceReference interface). All P12-C safety rules apply.
  No download, no export, no external links.

  If evidenceReferences is absent or empty, the empty state renders:
  "No evidence references attached to this violation."
  data-testid: workflow-evidence-{index}-empty

7.2  Forensic context linking

  Identical pattern to P12-C violations:
  - Shown when workflow.entityId is present
  - FORENSIC_CONTEXT_GUIDANCE.description explains how to use the entity ID
  - Copy Entity ID button (navigator.clipboard, 1.5s feedback)
  - "Review Forensic Timeline" link to /super-admin/governance/audit-integrity
  - No auto-fetch of forensic timeline within the workflow row

7.3  Linked violation context

  Shown when violationId or policyId is present:
  - Dashed-border panel with amber badge (Read-Only)
  - "Use the identifiers below to cross-reference on the Policy Violations page."
  - violationId + copy button
  - policyId + copy button
  - "View Policy Violations" link to /super-admin/governance/violations

7.4  No auto-fetch of violation details

  The page does not call useGovernanceViolations() from within each expanded
  workflow row. This avoids N+1 query patterns. The operator copies the
  violationId and navigates to the violations page for full details.

11.4  Test fix note (T7 precision)

  Initial T7 used "create" and "escalate" in a simple substring check.
  These words appear legitimately in status descriptions ("has been created")
  and status labels ("Escalated"). The test was refined to check:
  - descriptionForbidden: UI-export and legal-verdict terms only
    (trigger, export, pdf, xlsx, ai summary, legal verdict, guilty, etc.)
  - labelActionPhrases: button-phrasing patterns only
    (click to escalate, click to resolve, export, pdf, ai summary, etc.)
  This correctly distinguishes state labels from button labels.
