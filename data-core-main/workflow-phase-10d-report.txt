━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

2.1  Layer Design

  This phase adds a persistent, human-controlled orchestration tracking layer
  on top of P10-C advisory intelligence:

  Layer D — Pure Orchestration Engine (read-only, sync, no DB)
    artifacts/api-server/src/lib/workflows/recovery-orchestration.ts

    Pure functions:
      buildOrchestrationAction(input)          → RecoveryOrchestrationAction
      validateOrchestrationTransition(c, n)    → TransitionValidation
      detectDuplicateOrchestration(ws, t, all) → { isDuplicate }
      canAcknowledge / canBeginReview /
        canResolve / canRollBack / canCancel   → boolean status guards

    Constants:
      VALID_TRANSITIONS         — state machine table
      TERMINAL_ORCHESTRATION_STATUSES
      ACTIVE_ORCHESTRATION_STATUSES
      ALL_ORCHESTRATION_TYPES
      ORCHESTRATION_RECOMMENDATION_MAP

  DB Layer — Persistent Audit Log
    lib/db/src/schema/recovery-orchestrations.ts
    Table: recovery_orchestration_actions
    Semantics: append-only (records never deleted; only status updated)

2.2  Full Intelligence Chain (P10-A → P10-D)

  P10-A: Live reliability domain evaluation (per-capture snapshot)
  P10-B: Append-only snapshot persistence + incident lifecycle tracking
  P10-C: Operator advisory recommendations + trend analysis
  P10-D: Human-in-the-loop orchestration actions + remediation lifecycle

  Advisory (P10-C) feeds Orchestration (P10-D):
    RecoveryRecommendation.recommendationId → RecoveryOrchestrationAction.recommendationId
    RecoveryRecommendation.recommendationType → maps to orchestrationType via
                                                ORCHESTRATION_RECOMMENDATION_MAP

2.3  Data Flow

  POST /platform/reliability/orchestrations:
    1. Validate input via buildOrchestrationAction() (pure engine)
    2. Check for duplicate active orchestration (same workspace + type)
    3. If duplicate → 409 ORCH_DUPLICATE with conflictingActionId
    4. Persist to recovery_orchestration_actions
    5. Return 201 with { action: OrchestrationDto }

  POST /platform/reliability/orchestrations/:id/acknowledge:
    1. Fetch action by actionId
    2. Validate transition initiated → acknowledged via validateOrchestrationTransition()
    3. Update: orchestrationStatus, acknowledgedBy, acknowledgedAt
    4. Emit recovery_orchestration_acknowledged event
    5. Return { action: OrchestrationDto }

  POST /platform/reliability/orchestrations/:id/resolve:
    1. Fetch action by actionId
    2. Accept acknowledged or in_review — acknowledged auto-advances to in_review
    3. Validate transition → resolved
    4. Update: orchestrationStatus, resolvedBy, resolvedAt, rollbackEligible=false
    5. Emit recovery_orchestration_resolved event
    6. Return { action: OrchestrationDto }

  POST /platform/reliability/orchestrations/:id/rollback:
    1. Fetch action by actionId
    2. Validate transition in_review → rolled_back (rollbackEligible=true required)
    3. Update: orchestrationStatus, rolledBackBy, rolledBackAt, rollbackEligible=false
    4. Emit recovery_orchestration_rolled_back event
    5. Return { action: OrchestrationDto }

  GET /platform/reliability/orchestrations:
    1. Apply filters: workspaceId?, status?, orchestrationType?, activeOnly?
    2. Order: initiatedAt DESC
    3. Paginate: limit (1–100, default 50), offset (default 0)
    4. Return { actions[], total, limit, offset }

3.1  RecoveryOrchestrationAction Fields

  actionId            — "orch:<workspaceId>-<ms>-<seq>" (unique, stable)
  workspaceId         — target workspace
  incidentId          — soft-ref to reliability_incidents.incident_id
  recommendationId    — soft-ref to advisory recommendation (optional)
  orchestrationType   — one of 8 types (see 3.2)
  initiatedBy         — operator identifier (required; blank → 400)
  initiatedAt         — ISO 8601 creation timestamp
  orchestrationStatus — current lifecycle state
  acknowledgedBy      — operator who acknowledged (populated on transition)
  acknowledgedAt      — timestamp of acknowledgement
  resolvedBy          — operator who resolved
  resolvedAt          — timestamp of resolution
  rollbackEligible    — false after rollback or resolution (double-rollback guard)
  rolledBackBy        — operator who rolled back
  rolledBackAt        — timestamp of rollback
  cancelledBy         — operator who cancelled
  cancelledAt         — timestamp of cancellation
  relatedSignals      — evidence codes from advisory engine (JSON string array)
  executionNotes      — append-only operator notes (each transition appends)
  createdAt / updatedAt

3.2  Eight Orchestration Types

  scheduler_pressure_review      — investigate elevated scheduler workload pressure
  fairness_policy_review         — review active fairness policies for misconfiguration
  containment_audit              — audit containment boundaries and blast radius metrics
  noisy_tenant_investigation     — investigate cross-workspace resource contamination
  advisory_threshold_review      — review advisory storm thresholds and weight composition
  recovery_stability_validation  — validate stability before declaring incident resolved
  escalation_monitoring          — monitor workspace for repeated escalation patterns
  operational_watch              — maintain observation cadence for degraded workspace

3.3  Type ↔ Recommendation Mapping

  scheduler_pressure_review      ← investigate_scheduler_pressure
  fairness_policy_review         ← review_fairness_policies

4. ROLLBACK GUARANTEES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

4.1  Rollback Eligibility Rules

  Initial state:        rollbackEligible = true on every new action
  After rollback:       rollbackEligible = false (prevents double-rollback)
  After resolution:     rollbackEligible = false (resolved = concluded positively)
  After cancellation:   never rollbackable (terminal state, no VALID_TRANSITION)

4.2  Rollback Gate (pure engine enforcement)

  validateOrchestrationTransition("in_review", "rolled_back", rollbackEligible):
    If rollbackEligible=false → { valid: false, errorCode: "ORCH_ROLLBACK_INELIGIBLE" }

  canRollBack(status, rollbackEligible):
    Returns false for any terminal status regardless of rollbackEligible value.
    Returns false if rollbackEligible=false even for in_review.
    Returns true only when: status="in_review" AND rollbackEligible=true.

4.3  Rollback vs Cancellation

  rolled_back:  Operator explicitly undoes the action taken during in_review.
                Implies an action was taken and then undone — used for operational
                record-keeping to document that recovery steps were reversed.

  cancelled:    Operator abandons the orchestration at any pre-terminal state
                without committing to a recovery action or undo.
                Lighter-weight exit — no implication of executed-then-undone steps.

4.4  No Cascading Rollbacks

  Rolling back an orchestration action does NOT:
    - Undo scheduler state changes (the engine never made any)
    - Remove the action from DB history
    - Revert any fairness policy changes (the engine never changed any)
    - Cancel sibling orchestration actions

  Rollback is an AUDIT RECORD only — it documents that the operator explicitly
  chose to reverse the investigation steps they took during in_review.

5. AUDITABILITY MODEL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

5.1  Operator Attribution

  Every transition requires a named operator:
    initiatedBy    — required at creation (blank string → 400 ORCH_VALIDATION_INITIATED_BY)
    acknowledgedBy — required to acknowledge
    resolvedBy     — required to resolve
    rolledBackBy   — required to roll back

  No anonymous orchestration is permitted. Every lifecycle event has a named
  human operator permanently recorded in the DB row.

5.2  Append-Only Semantics

  Records in recovery_orchestration_actions are NEVER deleted.
  Status transitions update the existing row (status + attribution fields).
  executionNotes accumulate: each transition appends its notes to the existing
  value with a newline separator, preserving the full note history.

5.3  Immutable After Terminal

  Once a record reaches resolved, rolled_back, or cancelled:
    - No further status transitions are accepted (ORCH_TERMINAL error)
    - The DB row remains readable in perpetuity
    - Soft-references (incidentId, recommendationId) are never modified

5.4  Audit Serialization

  Every DTO field is JSON-safe (no functions, no Date objects — all ISO strings).
  relatedSignals is a string[] (serialized as JSON in DB, deserialized by DTO).
  All timestamps are UTC ISO 8601.

5.5  Indexes for Audit Queries

  recovery_orch_workspace_status_idx  — workspace × status (duplicate check, active view)
  recovery_orch_incident_idx          — incidentId (per-incident orchestration history)
  recovery_orch_type_status_idx       — orchestrationType × status (type-level monitoring)
  recovery_orch_initiated_idx         — initiatedAt (chronological ordering)

6. ORCHESTRATION LIFECYCLE STATE MACHINE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

States and transitions:

  initiated ──→ acknowledged ──→ in_review ──→ resolved     (terminal)
      │               │              │
      └──→ cancelled  └──→ cancelled └──→ rolled_back  (terminal)
                                     └──→ cancelled    (terminal)

  VALID_TRANSITIONS table (enforced deterministically by the pure engine):
    initiated:    [acknowledged, cancelled]
    acknowledged: [in_review, cancelled]
    in_review:    [resolved, rolled_back, cancelled]
    resolved:     []  (terminal)
    rolled_back:  []  (terminal)
    cancelled:    []  (terminal)

  Error codes:
    ORCH_TERMINAL          — attempted transition from a terminal status
    ORCH_INVALID_NEXT      — target status is not a known status string
    ORCH_TRANSITION_DENIED — transition not listed in VALID_TRANSITIONS
    ORCH_ROLLBACK_INELIGIBLE — rollback attempted with rollbackEligible=false
    ORCH_DUPLICATE         — active conflict for same workspace+type (HTTP 409)
    ORCH_VALIDATION_*      — input validation failures (HTTP 400)

  Convenience: resolve from acknowledged (skips in_review):
    Route allows resolve from acknowledged by internally auto-advancing
    to in_review before resolving. Validated as two sequential pure-engine
    checks before DB write. Audit record shows directly resolved — no
    phantom in_review timestamp is created.

           orchestrationStatus="initiated", initiatedBy, action

           orchestrationStatus="acknowledged", initiatedBy, action

   Fired: inside POST /:id/resolve route, after successful DB update.
   Fields: actionId, workspaceId, incidentId, orchestrationType,

   Fired: inside POST /:id/rollback route, after successful DB update.
   Fields: actionId, workspaceId, incidentId, orchestrationType,
           orchestrationStatus="rolled_back", initiatedBy, action

All events: logger.info() with structured JSON — no console.log.
Event A emitted from pure engine; events B/C/D emitted from route handlers
after DB persistence (ensuring events only fire on committed transitions).

12. REMAINING RESILIENCE GAPS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

P10-D is intentionally minimal and human-controlled. Remaining gaps:

  1. ESCALATION COUNT NOT STORED IN DB
     escalationCount is derived from snapshot sequences, not stored as an
     aggregate in reliability_incidents. R4 and R7 P10-C recommendation rules
     cannot fire from DB incidents without enrichment.
     Gap: requires a schema migration to add escalation_count to
          reliability_incidents (P10-E candidate).

  2. NO FRONTEND ORCHESTRATION QUEUE
     All P10-D routes are backend only.
     Gap: no Recovery Center UI in the Super Admin panel — no "Initiate Action"
          button, no active queue view, no lifecycle progress tracker.

  3. NO NOTIFICATION DISPATCH
     Orchestration state changes are only visible via API polling.
     Gap: no push notification when an operator initiates, resolves, or
          rolls back an orchestration action.

  4. NO CANCEL ROUTE
     VALID_TRANSITIONS includes "cancelled" from all non-terminal states,
     and canCancel() is implemented in the pure engine. However, no
     POST /:id/cancel route was added (spec did not require it).
     Gap: operators can only cancel via the DB directly or by resolving/rolling back.

  5. NO ORCHESTRATION HISTORY FOR AN INCIDENT (dedicated route)
     The GET /orchestrations route supports incidentId filter via query param
     but there is no dedicated GET /recommendations/:incidentId/orchestrations
     route that surfaces the full orchestration chain for a specific incident.
     Gap: linked recommendation → orchestration traceability requires two
          separate API calls.

  6. NO CROSS-ORCHESTRATION DEPENDENCY TRACKING
     Each action is independent. If workspace W has both a containment_audit
     and an operational_watch running in parallel, there is no dependency link
     or ordering enforcement between them.
     Gap: parallel orchestration coordination not modelled.

  7. NO SLA / TIMEOUT TRACKING
     There is no maximum age enforcement on active orchestrations.
     An initiated orchestration can remain active indefinitely.
     Gap: stale orchestrations (e.g. initiated 7+ days ago with no progress)
          are not flagged or escalated.

  8. EXECUTION NOTES NOT VERSIONED
     Notes accumulate via string append (newline separator) — there is no
     per-transition structured note record. History is reconstructable by
     reading the full notes string but not queriable per transition.
     Gap: structured per-transition audit notes would require a separate
          orchestration_notes table or JSONB array column.
