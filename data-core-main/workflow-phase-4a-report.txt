  buildResolvedData(stepOutputs)
    Computes the backward-compatible flat resolvedData from the namespaced

    Undefined/null per-step outputs handled safely (treated as {}).

SECTION 2 — HOW stepOutputs WORKS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

stepOutputs is a Record<number, Record<string, unknown>> on ExecutionContext.
The number key is step.index (not array position — matches step.index from config).

Lifecycle:
  • Context created: stepOutputs = {}
  • Step 0 completes: stepOutputs[0] = { notified: 3, recipientIds: [1,2,3] }
  • Step 1 completes: stepOutputs[1] = { taskId: 42, assigneeId: 7 }
  • Step 2 completes: stepOutputs[2] = { entity: "ticket", newStatus: "resolved" }

The executor stores output AFTER the step completes successfully (i.e., after
the step log has already been written to DB). Failed steps do not populate
stepOutputs (the execution halts at the failure handling block before reaching
the P4-A line).

Isolation guarantee: stepOutputs[0].status and stepOutputs[2].status are
independent values. A key "status" in step 0's output CANNOT silently overwrite
the same key from step 2 within the isolated slots. This is the WG-15 fix.

SECTION 3 — HOW resolvedData BACKWARD COMPATIBILITY WORKS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Step handlers (notification, task, assignment, condition, status_update, approval)
all read from:
  data = { ...ctx.triggerData, ...ctx.resolvedData }

ctx.resolvedData is now a computed flat merge of all stepOutputs.
It is recomputed by buildResolvedData() after every step completion.

buildResolvedData():
  1. Takes all entries from stepOutputs.
  2. Sorts them by step index ascending (0, 1, 2, ...).
  3. Reduces them left-to-right with object spread.
  4. Result: flat map where later steps shadow earlier steps on key collision.

This is IDENTICAL to the old behavior:
  Old: each step's output was merged into a single flat object in sequence.
  New: each step's output is stored namespaced, then merged in the same order.

The only difference: the OLD merge was irreversible (once merged, you could
not tell which step produced which key). The NEW merge is reversible — you can
always inspect stepOutputs[i] to see exactly what step i produced.

Merge order example (non-sequential indices 0, 5, 10):
  stepOutputs = { 10: {c:3}, 0: {a:1}, 5: {b:2, a:99} }
  After sort:  [(0,{a:1}), (5,{b:2,a:99}), (10,{c:3})]
  Merge result: { a: 99, b: 2, c: 3 }
  Step 5's a=99 shadows step 0's a=1 — correct (later step wins).

SECTION 4 — HOW triggerData BECAME IMMUTABLE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Problem (before P4-A):
  engine.ts assigned: triggerData: payload.data
  payload.data is the original EventPayload object passed through the bridge.
  Step handlers that call: data = { ...ctx.triggerData, ...ctx.resolvedData }
  spread the top-level keys safely.
  BUT: if a handler mutates a NESTED object (e.g., data.metadata.priority = "low"),
  it mutates the original payload.data object, which could affect:
    - The original event log entry (if the object is shared).
    - Other concurrent executions triggered by the same event
      (all receive the same payload reference from the loop in handleEvent()).

Fix (P4-A):
  createExecutionContext() calls: triggerData: structuredClone(triggerData)
  structuredClone() is a Node.js v17+ built-in (available on Node 24).
  It performs a deep clone of the entire payload, including nested objects and arrays.
  Each execution receives its own independent copy of the event data.

Result:
  - Mutations to triggerData inside a step handler do not affect the original payload.
  - Multiple concurrent executions for the same event each get an independent clone.
  - The original payload.data passed to handleEvent() remains intact.

Cost: one structuredClone() per execution at trigger time.
Typical trigger payload: < 1KB. Clone cost: < 1ms. Negligible.

SECTION 5 — EDGE CASES DISCOVERED
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

EC-1: step.index vs. array position (pre-existing, documented)
  buildResolvedData uses step.index (the numeric key), not array position.
  If steps are stored with non-sequential indices (e.g., 0, 5, 10), the merge
  order is by step.index ascending — correct and deterministic.
  The current executor still uses array position i for stepIndex logging (DB),
  but uses step.index for stepOutputs storage. Both are internally consistent:
  stepOutputs[step.index] and the DB log at position i refer to the same step.
  This inconsistency between DB-logged stepIndex (= array pos) and step.index
  will be fully resolved in P4-F when the cursor model aligns them.

EC-2: Failed step does not write to stepOutputs (correct behavior)
  The P4-A write:
    ctx.stepOutputs[step.index] = result.output ?? {}
    ctx.resolvedData = buildResolvedData(ctx.stepOutputs)
  ...is only reached when result.success === true AND result.waitForApproval !== true.
  Failed steps write to the DB step log (status='failed') and then return early.
  They do NOT pollute stepOutputs. This is correct: subsequent steps in a resumed
  execution (future Phase 5) will not see partial outputs from failed steps.

EC-3: waitForApproval returns before P4-A write (correct behavior)
  The return for waitForApproval also happens BEFORE the P4-A write.
  The approval step's output is therefore NOT stored in stepOutputs.
  This is intentional for Phase 4: approval steps are blocked by the validator
  and cannot be activated. When Phase 5 enables approval resume, the resume
  mechanism will need to store the approval step's output at pause time.
  No action required in Phase 4.

EC-4: result.output undefined/null (handled)
  The executor always writes:
    ctx.stepOutputs[step.index] = result.output ?? {}
  The ?? {} guard means: even if a step returns { success: true } with no output
  field, an empty object is stored. buildResolvedData handles empty objects by
  spreading nothing ({} spread adds no keys). No throw, no corruption.

EC-5: structuredClone on non-serializable payload values
  structuredClone cannot clone certain values (functions, Symbols, WeakMaps, DOM
  nodes, etc.). EventPayload.data comes from the event bus, which already JSON-
  serializes event data for the DB (via workspace_event_logs). Any value that
  survives the bus is structuredClone-safe.
  Risk: NONE for current event types. If future steps inject non-serializable
  values into payload.data, structuredClone would throw. Documented for awareness.
