OBJECTIVE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Convert raw short-term governance history (≤ 30 days) into compressed
long-term operational intelligence (up to 365 days) via a deterministic
append-only rollup lifecycle.

Prior to P7-C:
  • Analytics horizon was hard-capped at 30 days.
  • Snapshots older than 30 days were deleted without aggregation.
  • No long-term trend visibility existed.
  • Storage grew to ≤ 8,640 rows/workspace then pruned raw — no long-term
    continuity.
  • Pruning was unconditional — no safety gate between rollup and prune.

After P7-C:
  • governance_snapshot_rollups table provides 90-day (hourly) and
    365-day (daily) compressed history.
  • Query cascade automatically routes requests to the correct storage tier.
  • Pruning is gated on rollup success — raw data is never lost before it
    has been rolled up.
  • All rollup operations are strictly append-only and idempotent.
  • The full analytics window: 0–30d (raw) → 30–90d (hourly) → 90–365d (daily).

ROLLUP ARCHITECTURE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Table: governance_snapshot_rollups
File:  lib/db/src/schema/governance.ts

Two granularities share one table, distinguished by the `granularity` column:

  'hourly' tier:
    • Source: raw governance_snapshots (5-min resolution)
    • One row per workspace per clock-hour
    • Retention: 90 days
    • Resolution: 1-hour granularity for trend charts
    • Row count at capacity: 90 × 24 = 2,160 rows/workspace (vs 8,640 raw)

  'daily' tier:
    • Source: hourly rollup rows
    • One row per workspace per UTC calendar-day
    • Retention: 365 days
    • Resolution: 1-day granularity for long-term health history
    • Row count at capacity: 365 rows/workspace

Total maximum rows per workspace (all tiers combined):
  Raw:    8,640  (30d × 288/day)
  Hourly: 2,160  (90d × 24/day)
  Daily:    365  (365d × 1/day)
  Total: 11,165  rows/workspace at full retention capacity

Compared to raw-only (no rollups, keep forever): 105,120 rows/workspace/year
Compression ratio: 91.4% row reduction via the tiered rollup model.

AGGREGATION SEMANTICS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

All aggregation rules are pure, deterministic, and aligned with P7-A semantics.

── dominant_severity ────────────────────────────────────────────────────────

Rule: HIGHEST SEVERITY WINS.

Severity ranking (escalation order):
  healthy(0) < warning(1) < degraded(2) < critical(3)

Algorithm: map each severity to its rank, take the numeric maximum,
map back to string.  Ties resolve deterministically (numeric max is unique).

Example:
  ["healthy", "warning", "critical", "degraded"] → "critical"

Rationale: the dominant severity communicates "what was the WORST this
workspace experienced during this period?"  An hour with 11 healthy snapshots
and 1 critical snapshot is "critical" for operational reporting purposes.

For daily rollups: dominantSeverity = max(hourly.dominantSeverity)
This escalation is monotone — a day is never classified as less severe than
its constituent hours.

── avgErrorRate, avgApprovalBacklog, avgDelayBacklog, avgStuckCount ──────────

Rule: ARITHMETIC MEAN of the source metric across all source records.

For hourly from snapshots:
  avgErrorRate       = mean(snap.metricsSnapshot.workflowErrorRate)
  avgApprovalBacklog = mean(snap.metricsSnapshot.approvalBacklogCount)
  avgDelayBacklog    = mean(snap.metricsSnapshot.delayBacklogCount)
  avgStuckCount      = mean(snap.stuckCount)   ← top-level column, not JSONB

For daily from hourly rollups:
  avgErrorRate       = mean(rollup.avgErrorRate)
  avgApprovalBacklog = mean(rollup.avgApprovalBacklog)
  avgDelayBacklog    = mean(rollup.avgDelayBacklog)
  avgStuckCount      = mean(rollup.avgStuckCount)

Note: daily averages are technically "averages of hourly averages."
This is valid when hourly bucket sizes are similar (≈ 12 snapshots/hour
at 5-min interval).  No weighting by snapshotCount is applied — uniform
hourly bucket sizes are the expected operating mode.

For daily snapshotCount: SUM(rollup.snapshotCount) — total raw snapshots
aggregated into the day, providing provenance visibility.

── chronicAlertCodes ────────────────────────────────────────────────────────

Rule: GOV-* codes appearing in STRICTLY MORE THAN 50% of source records.

Threshold: ROLLUP_CHRONIC_THRESHOLD_PCT = 50 (aligned with P7-A CHRONIC_THRESHOLD_PCT)
Comparison: strict greater-than — a code at exactly 50% is NOT chronic.
Deduplication: each distinct code counts at most once per source item.
Output: sorted lexicographically for a stable, deterministic response.

For hourly from snapshots:
  Count appearances of each GOV-* code across snapshot.alertCodes arrays.
  Code is chronic when (count / total_snapshots) * 100 > 50.

For daily from hourly rollups:
  Count appearances of each code across rollup.chronicAlertCodes arrays.
  Code is chronic in the day when it was chronic in > 50% of the hours.

This creates a two-level chronic filter: a code must be chronic within each
hour AND appear in most hours of the day to reach the daily chronic level.
This prevents spurious alerts from one bad hour from tainting the daily view.

── stormFrequency ────────────────────────────────────────────────────────────

Rule: FRACTION of source records with active storm signal.

For hourly from snapshots:
  stormFrequency = count(stormSeverity != "none") / total_snapshots
  Range: 0.0–1.0 (e.g. 0.25 = storm present in 25% of the hour's snapshots)

For daily from hourly rollups:
  stormFrequency = arithmetic mean(rollup.stormFrequency)
  Range: 0.0–1.0 (e.g. 0.5 = on average, half the hours had some storm activity)

Rationale: mean of hourly frequencies propagates the storm intensity signal
correctly from hourly to daily without losing magnitude.

PRUNING INTEGRATION GUARANTEES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The prune-safe guarantee is enforced at the scheduler level.  The cycle
now has five ordered steps (up from four in P7-B):

  Step 1: Enumerate active workspaces (bounded, deterministic order)
  Step 2: Capture snapshots  (append-only INSERT, one row per workspace)
  Step 3: Build rollups      (P7-C — NEW — append-only, idempotent)
  Step 4: Prune raw          (ONLY for workspaces where step 3 succeeded)
  Step 5: Emit observability events

Step 3 → Step 4 coupling (rollupSuccessSet):
  • For each workspace, buildWorkspaceRollups() returns { success: boolean }.
  • If success=true  → workspace ID added to rollupSuccessSet → pruning enabled.
  • If success=false → workspace ID NOT in rollupSuccessSet → pruning SKIPPED.
  • A Set<number> tracks the gate per workspace within each cycle.
  • Per-workspace failure isolation: one workspace's rollup failure does not
    prevent pruning for other workspaces.

What "rollup success" means:
  • buildWorkspaceRollups() completed without throwing.
  • Both hourly and daily rollup windows were processed (even if zero new
    rows were inserted — idempotent re-runs return success=true with built=0).
  • The try/catch in buildWorkspaceRollups() catches DB errors, serialization
    errors, and any other unexpected failure, and returns success=false.

What happens when rollup fails:
  1. GOVERNANCE_ACTION_ROLLUP_FAILED is logged with the error message.
  2. Workspace is excluded from rollupSuccessSet.
  3. pruneSnapshotsOlderThan() is never called for that workspace.
  4. Raw snapshots for that workspace are preserved intact for the next cycle.
  5. The cycle continues for other workspaces — per-workspace isolation.

Guarantee: no raw snapshot is ever deleted before a rollup covering its
time window has been successfully persisted.

This covers the most important failure modes:
  • DB connection failure during rollup → prune skipped → retry next cycle.
  • Partial rollup (insertRollupIfNotExists throws) → prune skipped.
  • Clock skew or unexpected timestamp → rollup may skip some hours (future
    bucket guard: `if (bucket.end > now) continue`) → prune still skipped for
    the window.

Function: selectQueryLayer(daysBack: number): QueryLayer

  daysBack range         Storage tier    Table
  ─────────────────────  ──────────────  ──────────────────────────────────
  0–30 days (inclusive)  "raw"           governance_snapshots
  31–90 days             "hourly"        governance_snapshot_rollups (hourly)
  91–365 days            "daily"         governance_snapshot_rollups (daily)
  > 365 days             "daily"         clamped — caller should validate max

Boundary semantics:
  • daysBack=30 → "raw" (the boundary is inclusive for raw)
  • daysBack=31 → "hourly" (first day beyond raw retention)
  • daysBack=90 → "hourly" (the boundary is inclusive for hourly)
  • daysBack=91 → "daily"  (first day beyond hourly retention)

The cascade boundaries are derived directly from the retention constants:
  RAW_TO_HOURLY_THRESHOLD_DAYS  = RETENTION_RAW_DAYS = 30
  HOURLY_TO_DAILY_THRESHOLD_DAYS = ROLLUP_HOURLY_RETENTION_DAYS = 90

Alignment guarantee: if a retention constant is updated, the cascade
boundary moves automatically — no separate update required.

Query functions:

  Raw tier (existing, P7-A):
    querySnapshotsByRange(workspaceId, since, until, database)
    → SELECT from governance_snapshots ORDER BY captured_at ASC

  Hourly tier (P7-C):
    queryRollupsByRange(workspaceId, 'hourly', since, until, database)
    → SELECT from governance_snapshot_rollups WHERE granularity='hourly'
       ORDER BY bucket_start ASC

  Daily tier (P7-C):
    queryRollupsByRange(workspaceId, 'daily', since, until, database)
    → SELECT from governance_snapshot_rollups WHERE granularity='daily'
       ORDER BY bucket_start ASC

Response contract:
  Raw tier:    returns StoredSnapshot[]  → serialize to SeverityDataPoint / NumberDataPoint
  Rollup tiers: returns StoredRollup[]   → serialize to RollupDataPoint[]

Both tiers return time-ordered arrays suitable for trend chart rendering.
RollupDataPoint includes all aggregated metric fields (avgErrorRate, etc.)
plus bucketStart/bucketEnd/granularity for chart labeling.

Cascade resolution observability:
  resolveQueryLayer(daysBack, workspaceId) emits:
    governance_rollup_query_resolved

ROLLUP PIPELINE DESIGN
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Bucket computation (pure):
  computeHourBucket(ts) — truncates to UTC hour start, returns [hourStart, hourStart+1h)
  computeDayBucket(ts)  — truncates to UTC midnight, returns [dayStart, dayStart+24h)
  bucketKey(bucket)     — stable Map key = bucketStart.toISOString()

Grouping (pure):
  groupSnapshotsByHourBucket(snapshots) — groups by capturedAt hour bucket
  groupRollupsByDayBucket(rollups)      — groups by bucketStart day bucket
  Both return sorted ascending by bucket start for deterministic iteration.

Bucket enumeration (pure):
  enumerateHourBuckets(windowStart, windowEnd) — all hour buckets in window
  enumerateDayBuckets(windowStart, windowEnd)  — all day buckets in window
  Bounded: max 8,760 hour buckets (365d × 24h), max 365 day buckets.

Pipeline (async, DB layer):
  buildHourlyRollupsForWorkspace(workspaceId, windowStart, windowEnd, now, db)
    For each hour bucket:
    1. Skip if bucket.end > now (don't roll up incomplete hours)
    2. querySnapshotsInBucket(workspaceId, bucketStart, bucketEnd) — READ-ONLY
    3. If snapshots exist: buildHourlyRollupPayload() → insertRollupIfNotExists()
    4. Count inserted rows (conflicts = 0 → already existed → idempotent)
    5. Emit governance_rollup_hourly_completed

  buildDailyRollupsForWorkspace(workspaceId, windowStart, windowEnd, now, db)
    For each day bucket:
    1. Skip if bucket.end > now (don't roll up the current day)
    2. queryHourlyRollupsInBucket(workspaceId, dayStart, dayEnd) — READ-ONLY
    3. If hourly rollups exist: buildDailyRollupPayload() → insertRollupIfNotExists()
    4. Emit governance_rollup_daily_completed

Orchestration window in buildWorkspaceRollups(workspaceId, now, db):

  Hourly window: [now - (RETENTION_RAW_DAYS + 2d), now - (RETENTION_RAW_DAYS - 2d)]
  = [now - 32d, now - 28d]  (4-day overlap zone around the 30d boundary)

  Daily window: [now - (ROLLUP_HOURLY_RETENTION_DAYS + 2d), now - (ROLLUP_HOURLY_RETENTION_DAYS - 2d)]
  = [now - 92d, now - 88d]  (4-day overlap zone around the 90d boundary)

  The ±2 day overlap zones handle:
    • Clock skew (up to 48 hours)
    • Scheduler downtime (missed cycles)
    • DST transitions (bucket boundary edge cases)
    • Gradual coverage build-up on first deployment

  Per-cycle cost: at most 96 hour queries (4d × 24h) + 4 day queries per
  workspace.  At 12 snapshots/hour, this is at most 1,152 snapshot rows
  read per cycle per workspace — a very low DB load.

INSERT semantics:
  insertRollupIfNotExists(payload, db) uses:
    INSERT INTO governance_snapshot_rollups VALUES (...) ON CONFLICT DO NOTHING

  The UNIQUE index on (workspace_id, granularity, bucket_start) handles the
  conflict.  Returns true if inserted, false if the row already existed.
  Re-running the same cycle is always safe — no duplicates, no errors.

SAFETY RULES SUMMARY
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Nine safety rules enforced by P7-C:

  RULE 1 — Append-only rollup inserts:
    insertRollupIfNotExists() uses INSERT ON CONFLICT DO NOTHING.
    No UPDATE path exists for rollup rows anywhere in the codebase.

  RULE 2 — Never rewrite existing rollups:
    The unique index on (workspaceId, granularity, bucketStart) ensures
    the second run for the same bucket is a no-op (conflict silently skipped).

  RULE 3 — Never mutate surviving raw snapshots:
    querySnapshotsInBucket() is SELECT-only.  The rollup pipeline never
    touches governance_snapshots with INSERT, UPDATE, or DELETE.

  RULE 4 — Never prune before rollup success:
    The scheduler checks rollupSuccessSet before calling pruneSnapshotsOlderThan().
    Workspaces not in the success set are silently skipped for pruning.

  RULE 5 — Never roll up the current (incomplete) hour or day:
    buildHourlyRollupsForWorkspace: `if (bucket.end > now) continue`
    buildDailyRollupsForWorkspace: same guard.
    Ensures only complete, closed buckets are rolled up.

  RULE 6 — Never affect runtime executions:
    governance-rollup.ts never imports from or writes to workflow_executions,
    approval_requests, or any other operational runtime table.

  RULE 7 — Idempotent pipeline:
    Running the rollup pipeline multiple times for the same workspace and
    now produces identical results — new rollups for any previously unprocessed
    hours, and silently skips already-rolled-up hours.

  RULE 8 — Per-workspace failure isolation:
    A DB error for one workspace's rollup does not affect other workspaces.
    The scheduler's try/catch per workspace ensures the cycle continues.

  RULE 9 — Rollup rows are immutable after insertion:
    Once inserted, a rollup row is never touched again.  No background job
    or API endpoint modifies rollup rows.  Historical analytics data is
    write-once audit data.

   Fields: workspaceId, error, action

   Level:  INFO
   When:   resolveQueryLayer(daysBack, workspaceId) is called
   Fields: workspaceId, daysBack, layer, action
   Emitter: governance-rollup.ts (query cascade resolution)

All five actions are distinct string constants exported from governance-rollup.ts
for use by any future handler, test, or monitoring system.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. ROLLUP RETENTION PRUNING (not in P7-C):
   Hourly rollup rows older than 90 days and daily rollup rows older than
   365 days are never pruned.  The governance_snapshot_rollups table grows
   indefinitely.  A future prune pass needs:
     • pruneOldRollups(workspaceId, 'hourly', cutoff90d, database)
     • pruneOldRollups(workspaceId, 'daily',  cutoff365d, database)
   This requires the same safety rule: prune daily only after daily rollup
   exists, prune hourly only after daily rollup covers that window.

2. NO API ENDPOINTS FOR ROLLUP TIERS (not in P7-C):
   queryRollupsByRange() and selectQueryLayer() exist but are not wired into
   any HTTP route.  The existing GET /governance/snapshots only queries raw
   snapshots.  A future GET /governance/trends?range=90d endpoint would
   use the cascade to serve the correct tier automatically.

3. NO FRONTEND ROLLUP CHARTS (not in P7-C):
   No Recharts components consume RollupDataPoint[].  The cascade model
   is ready but not surfaced in the UI.

4. FIRST-RUN BACKFILL GAP (not in P7-C):
   On first deployment, the rollup overlap zone is [28d–32d ago].  If raw
   snapshots exist for days 0–30 but no rollups exist yet, the first cycle
   will only build 4 days of hourly rollups.  The remaining 26 days of
   raw history before the overlap zone has no rollups.  A one-time backfill
   job scanning [0d–30d ago] would fill this gap.

5. WEIGHTED AVERAGING FOR UNEQUAL HOURLY BUCKET SIZES (not in P7-C):
   When a workspace goes offline mid-hour (e.g. 6 snapshots in that hour
   instead of 12), daily averages weight that hour equally with 12-snapshot
   hours.  A future enhancement could weight by snapshotCount, but this
   adds complexity and the current bias is minor for operational reporting.

6. CROSS-WORKSPACE ROLLUP ANALYTICS (super_admin — not in P7-C):
   No endpoint compares rollup trends across multiple workspaces simultaneously.
   A super_admin endpoint returning platform-wide avg error rate trends
   would require a CROSS-JOIN or aggregation across all workspace rollups.

7. ROLLUP SCHEDULER STATUS ENDPOINT (not in P7-C):
   No GET endpoint exposes last rollup cycle stats (hourlyBuilt, dailyBuilt,
   success/failure counts per workspace) to the super-admin dashboard.

8. HOURLY ROLLUP COVERAGE GAPS FROM SCHEDULER DOWNTIME (not in P7-C):
   If the server is down for > 32 days, the overlap zone misses snapshots
   from before (now - 32d).  A longer overlap window or a full historical
   backfill pass would be needed to recover from extended downtime.
