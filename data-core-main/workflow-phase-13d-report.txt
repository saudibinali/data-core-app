API SURFACE
────────────────────────────────────────────────────────────────────────────────

GET  /api/platform/tenants/:tenantId/entitlements
  Auth: super_admin only
  Response: { entitlementProfile, overrides }

PATCH /api/platform/tenants/:tenantId/entitlements/overrides
  Auth: super_admin only
  Body: { overrides: [{ moduleCode, overrideType, limitCode?, limitValue?, reason }], confirmation: true }
  Response: { entitlementProfile, overrides, tenant, auditEvent }
  Validation: ENTITLEMENT_REASON_MIN_LENGTH=10, max 50 overrides per batch,
              known module codes only, known limit codes only for limit_override,
              limitCode required for limit_override, limitValue ≥ 0 or null

NOTES
────────────────────────────────────────────────────────────────────────────────

- No payment, billing, invoice, or tax logic introduced anywhere.
- No enforcement logic — entitlements are informational only.
- No automatic workspace suspension or lifecycle state changes.
- governance_console entitlement is tenant-workspace-scoped future feature only;
  it does NOT expose the Super Admin Governance Console to workspace users.
- Module implementations are entirely separate from entitlement access controls.
- All overrides are audit-logged to activityLogsTable with full payload.
- Upsert strategy: delete-then-insert (no DB unique constraint required on overrides).
- Frontend renders enabled module badges, key limits grid, and override list
  directly from the /entitlements GET response.
