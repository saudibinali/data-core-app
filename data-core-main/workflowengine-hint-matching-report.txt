File: artifacts/api-server/src/lib/workflows/engine.ts

The `handleEvent()` method was updated to implement a two-tier trigger matching
strategy.  All execution logic, context building, condition evaluation, and
workflow queueing are completely unchanged.  Only the definitions lookup changed.

BEFORE (single-tier):
  const definitions = await db
    .select()
    .from(workflowDefinitionsTable)
    .where(
      and(
        eq(workflowDefinitionsTable.workspaceId, payload.workspaceId),
        eq(workflowDefinitionsTable.triggerEvent, payload.event),
        eq(workflowDefinitionsTable.isActive, true),
      ),
    );
  if (definitions.length === 0) return;

AFTER (two-tier):
  // TIER 1 — primary match: exact event type (unchanged logic)
  const primaryDefs = await db
    .select()
    .from(workflowDefinitionsTable)
    .where(and(
      eq(...workspaceId),
      eq(triggerEvent, payload.event),
      eq(isActive, true),
    ));

  // TIER 2 — secondary hint match (form.submitted events only)
  let definitions = primaryDefs;
  if (payload.event === "form.submitted") {
    const hint = payload.data["workflowEventHint"];
    if (hint && typeof hint === "string") {
      const hintDefs = await db.select()...WHERE triggerEvent = hint;
      if (hintDefs.length > 0) {
        logger.debug({ hint, hintMatches, primaryMatches }, "secondary hint match");
        const primaryIds = new Set(primaryDefs.map(d => d.id));
        const newDefs = hintDefs.filter(d => !primaryIds.has(d.id));
        definitions = [...primaryDefs, ...newDefs];
      }
    }
  }

  if (definitions.length === 0) return;
  // ... execution loop unchanged

SECTION 2 — HOW MATCHING NOW WORKS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

For every event that reaches handleEvent():

  ┌─────────────────────────────────────────────────────────────────────────┐
  │ 1. TIER 1 — Always runs                                                 │
  │    Query: WHERE workspaceId = X AND triggerEvent = payload.event        │
  │    Result: primaryDefs (may be empty)                                   │
  └─────────────────────────────────────────────────────────────────────────┘
                              │
               payload.event = "form.submitted"?
                    YES ──────────────────────────────────────────────────┐
                    NO:  definitions = primaryDefs → proceed to exec loop │
                                                                          ▼
  ┌─────────────────────────────────────────────────────────────────────────┐
  │ 2. TIER 2 — Only for "form.submitted"                                   │
  │    Extract: hint = payload.data["workflowEventHint"]                    │
  │    If hint is a non-empty string:                                       │
  │      Query: WHERE workspaceId = X AND triggerEvent = hint               │
  │      Deduplicate: filter out any def already in primaryDefs (by id)     │
  │      Merge: definitions = [...primaryDefs, ...newDefs]                  │
  │    If hint is absent or empty string:                                   │
  │      definitions = primaryDefs (no second query, no overhead)           │
  └─────────────────────────────────────────────────────────────────────────┘
                              │
  ┌─────────────────────────────────────────────────────────────────────────┐
  │ 3. Execution loop — unchanged for all cases                             │
  │    if definitions.length === 0: return (early exit)                     │
  │    for each def: evaluate conditions → insert execution → run workflow  │
  └─────────────────────────────────────────────────────────────────────────┘

  DB queries per event:
    Non-form events:    1 query (TIER 1 only)
    form.submitted with hint:    2 queries (TIER 1 + TIER 2)
    form.submitted without hint: 1 query (TIER 1 only, TIER 2 short-circuits)

  Performance note: both queries are indexed on (workspaceId, triggerEvent, isActive).
  No N+1 issue — at most 2 queries per form submission event.

SECTION 3 — WORKFLOW CONFIGURATIONS NOW SUPPORTED
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Configuration 1 — Generic "catch-all forms" workflow
    trigger_event = "form.submitted"
    Fires on: ALL form submissions (any module, any hint)
    Matched by: TIER 1
    Example use case: "notify the ops team when any form is submitted"

  Configuration 2 — Module-scoped form workflow
    trigger_event = "hr.form.submitted"
    Fires on: HR module forms (Leave Request, any future HR form)
    Matched by: TIER 2 when payload.data.workflowEventHint = "hr.form.submitted"
    Example use case: "route HR form submissions to the HR manager for review"

  Configuration 3 — Module-scoped approvals workflow
    trigger_event = "approvals.form.submitted"
    Fires on: Approval-routed forms (Expense Reimbursement, etc.)
    Matched by: TIER 2 when hint = "approvals.form.submitted"
    Example use case: "auto-create an approval task when expense form is submitted"

  Configuration 4 — HR service-specific workflow
    trigger_event = "hr.annual-leave.submitted"  (or any hr.{slug}.submitted)
    Fires on: That specific HR service form only
    Matched by: TIER 2 when hint = "hr.annual-leave.submitted"
    Example use case: "trigger leave balance check for annual leave requests only"

  Configuration 5 — Domain event workflow (unchanged, TIER 1 only)
    trigger_event = "ticket.created"
    trigger_event = "approval.created"
    trigger_event = "employee.created"
    etc.
    Fires on: those events as before.  TIER 2 never activates for these.
    Zero behavioral change for all non-form workflows.

  Configuration 6 — Dual match (both tiers)
    trigger_event = "form.submitted" (TIER 1) AND another workflow with
    trigger_event = "hr.form.submitted" (TIER 2) for the same workspace.
    When an HR form is submitted: BOTH workflows fire.
    Deduplication only prevents the SAME definition from firing twice.
    Two DIFFERENT definitions with different IDs can both fire — this is correct
    and expected behavior (catch-all + specific workflows co-exist).

  Configuration 7 — Edge: hint = "form.submitted" (same as primary event)
    trigger_event = "form.submitted" in TIER 1 matches.
    In TIER 2: hint = "form.submitted" → query returns the same definition.
    Deduplication catches it (same ID already in primaryIds set).
    Definition runs exactly ONCE.  No duplicate execution.

SECTION 4 — DEDUPLICATION BEHAVIOR
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Algorithm:
    primaryIds = Set of IDs from TIER 1 results
    newDefs    = TIER 2 results filtered to exclude IDs already in primaryIds
    definitions = [...primaryDefs, ...newDefs]

  Guarantee:
    Each workflow definition appears at most ONCE in the definitions array.
    Therefore each workflow executes at most ONCE per event.

  Scope of deduplication:
    Within a single handleEvent() invocation.
    Across events (e.g. two rapid form.submitted events): not in scope.
    The existing idempotency mechanisms (if any) in the execution layer apply.

  Why primary takes precedence:
    primaryDefs are listed first to preserve execution order.
    In the dedup filter, a definition already in primaryDefs is excluded from
    newDefs — it runs in primary position, not duplicated in hint position.
