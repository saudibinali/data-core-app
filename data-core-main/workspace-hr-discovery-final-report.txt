EXECUTIVE SUMMARY
================================================================================
The project has a mature **Workspace = Tenant** model (`workspaces.id`) with
40+ workspace-scoped tables. **HR Foundation is largely built**: hierarchical
org units, job grades/titles, positions, work locations, configurable lookups,
a full **employees** domain (contracts, documents, custom fields, payroll,
attendance, services), and extensive REST APIs in a single `hr.ts` router
(~130 endpoints).

Major risks before any redesign:
  1) **departments** (legacy, user-centric) vs **hr_org_units** (HR-centric)
  2) **hr_employee_leaves** vs **leave_requests** (schema for latter may not
     be in applied migration 0000_sad_midnight.sql)
  3) **employees** vs **users** as duplicate person records
  4) Almost **no automated HR tests**
  5) Coarse HR permissions (hr.view / hr.manage only)

Recommendation: **Consolidate and extend existing models** — do not add parallel
tables or a greenfield HR stack.

================================================================================
BIGGEST DESIGN ISSUES
================================================================================
1. tenant/workspace naming used interchangeably
2. Monolithic hr.ts (~4000+ lines) — maintenance hazard
3. self_service.view missing from permission registry API
4. Schema–migration drift for leave_requests
5. No HR test coverage

================================================================================
FINAL RECOMMENDATION (BEFORE ANY CODE CHANGES)
================================================================================
1. Audit production/staging DB against Drizzle schema (especially leave_requests).
2. Decide canonical org (hr_org_units) and leave (leave_requests) models.
3. Document employee↔user linking rules.
4. Extend workspace RBAC — do not copy platform P17 override model.
5. Only then plan API split / UI consolidation / new HR permissions.

Do NOT start greenfield HR tables or parallel Foundation.
