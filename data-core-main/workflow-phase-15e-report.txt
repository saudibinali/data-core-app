1) PAYMENT RECORDS MODEL
================================================================================

Table: commercial_payment_records
Schema: lib/db/src/schema/commercial-payment-records.ts

Fields:
  - id, workspaceId, commercialAccountId, invoiceId
  - paymentReference, paymentDate
  - receivedAmount, currency
  - paymentMethod: bank_transfer | cheque | cash | internal_adjustment | other
  - collectionStatus: pending_verification | verified | rejected | partially_applied | reversed
  - recordedByUserId, verifiedByUserId (optional), verificationDate (optional)
  - internalNotes, rejectionReason (optional)
  - createdBy, updatedBy, createdAt, updatedAt

Rules:
  - Multiple payment records per invoice allowed.
  - No payment gateway / card / bank account / IBAN columns.
  - receivedAmount > 0 enforced on create/update.
  - currency must match invoice currency (strict reject on mismatch).
  - invoice must belong to same workspaceId as tenant route param.
  - commercialAccountId must match invoice account on create.
  - No hard delete — reversal via collectionStatus=reversed.

Migration: run `pnpm --filter @workspace/db run push` when Postgres is available.

================================================================================
2) COLLECTION SUMMARY LOGIC (COMPUTED)
================================================================================

Module: artifacts/api-server/src/lib/invoice-collection-summary.ts
(also used by GET collection-summary endpoint)

Not stored in DB — computed per request from payment rows.

Fields returned:
  - invoiceAmount, currency (from invoice)
  - totalRecordedPayments — sum of pending_verification + verified + partially_applied
  - totalVerifiedPayments — sum of verified + partially_applied only
  - outstandingAmount — max(0, invoiceAmount - totalVerifiedPayments)
  - collectionState:
      unpaid          — totalVerifiedPayments == 0
      partially_paid  — 0 < verified < invoiceAmount (and not disputed)
      paid            — verified >= invoiceAmount (exact)
      overpaid        — verified > invoiceAmount
      disputed        — has rejected payment(s) and not fully paid
  - paymentCount, verifiedPaymentCount, hasRejectedPayments

Rejected and reversed amounts are excluded from verified totals.
invoiceAmount is never modified by payment records.

4) BACKEND APIs (PLATFORM ONLY)
================================================================================

Router: artifacts/api-server/src/routes/commercial-payments.ts

GET /platform/tenants/:tenantId/commercial-payments
  - commercial.payments.read
  - Filters: invoiceId, collectionStatus, paymentMethod, from, to (paymentDate)

GET /platform/tenants/:tenantId/commercial-invoices/:invoiceId/collection-summary
  - commercial.payments.read
  - Returns computed summary object

POST /platform/tenants/:tenantId/commercial-invoices/:invoiceId/payments
  - commercial.payments.record
  - Creates pending_verification payment

PATCH /platform/tenants/:tenantId/commercial-payments/:paymentId
  - commercial.payments.record
  - Editable only when collectionStatus = pending_verification

PATCH .../payments/:paymentId/verify
  - commercial.payments.verify — requires reason (min 10 chars)

PATCH .../payments/:paymentId/reject
  - commercial.payments.verify — sets rejected + rejectionReason

PATCH .../payments/:paymentId/reverse
  - commercial.payments.verify — sets reversed + reason in internalNotes

Forbidden: DELETE, tenant payment routes, gateway/bank API routes.

Validation:
  - Rejects cardNumber, bankAccount, iban, stripe*, checkout* in JSON body
  - Tenant + invoice + account cross-checks
  - Currency match with invoice
