EXECUTIVE SUMMARY
================================================================================

The platform implements a **dual workforce model**:

  users (+ legacy flat departments + lineManagerId)
  employees (+ hr_org_units tree + directManagerId + optional userId link)

These are **not synchronized**. Only `fullName` is mandatory when creating an
employee; department/org unit, manager, job title, and position are **optional**.

Real hierarchical org exists in **hr_org_units** (parentId, typed levels).
Legacy **departments** remain flat and drive **user directory and workflows**.

Manager resolution **splits by module**:
  • Canonical leave → employees.directManagerId (manager must have userId)
  • Workflows → users.lineManagerId
  • Legacy dept → departments.managerId (limited use)

Verdict: **Metadata-rich HCM schema with under-governed runtime** — closer to
**user management + HR add-on** than unified enterprise workforce platform.

KEY ANSWERS (PRIMARY OBJECTIVE)
================================================================================

How is employee created?
  POST /hr/employees (fullName required) | provision | import
  User creation separate (POST admin/users) — no auto employee

How linked to department?
  employees.orgUnitId → hr_org_units (HCM)
  users.departmentId + user_departments → departments (legacy)

Is department mandatory?
  NO — orgUnitId optional at API and create UI

Real organizational hierarchy?
  YES in hr_org_units tree — NOT in legacy departments
  NOT unified platform-wide

Reporting / manager hierarchy?
  STORED optionally: directManagerId (employee), lineManagerId (user)
  NOT enforced; NOT synced; weak FK integrity

Position hierarchy?
  Job grades/titles operational; position SEATS (hr_positions) not assigned
  to employees in application layer (positionId unused)

Real workforce model?
  PARTIAL — employee table is canonical for HR ops; platform still user-centric

Enterprise-ready?
  NO — dual models, dangerous optional fields, approval fallbacks

================================================================================
ENTITY MODEL (CONDENSED)
================================================================================

users
  workspace_id, role (member|manager|admin), department_id, line_manager_id
  position (text), employment_status

employees
  workspace_id, user_id (optional unique), full_name (required)
  org_unit_id, job_title_id, job_grade_id, position_id (unused), direct_manager_id
  company/branch/location (text), status, employment_type, onboarding_data

hr_org_units
  type: company|branch|division|department|team, parent_id (tree)

departments (legacy)
  flat: name, manager_id → user

hr_positions
  seat definition: title, org_unit_id, headcount, status vacant|filled

================================================================================
ARCHITECTURAL RISKS (TOP 7)
================================================================================

1. Dual org systems (departments vs hr_org_units)
2. Dual manager fields (lineManagerId vs directManagerId)
3. Zero mandatory org relationships on employee create
4. User ↔ employee link without field sync
5. positionId / hr_positions disconnected from roster
6. Job movement history does not update current employee org
7. Leave/workflow approver sources disagree
