0. الملخص التنفيذي
================================================================================

Data Core منصة enterprise متعددة المستأجرين (multi-tenant) تجمع:
  • Core Ops (تذاكر، بريد، تقويم، مستخدمون، أقسام)
  • HCM/HR (موظفون، foundation، حضور، رواتب، استيراد)
  • Self-service + نماذج ذكية + سير عمل
  • Platform Super-Admin (مستأجرون، تجاري، اشتراكات، حوكمة)
  • On-Prem readiness (setup wizard، Docker، PM2)

نموذج النشر: monolith modular — API واحد (Express) + SPA (React/Vite) +
              PostgreSQL واحد + عزل منطقي بـ workspace_id.

التقييم العام للهندسة:
  ┌─────────────────────┬──────────┬────────────────────────────────────────┐
  │ المحور              │ النضج    │ ملخص                                   │
  ├─────────────────────┼──────────┼────────────────────────────────────────┤
  │ البنية المعمارية    │ جيد      │ طبقات واضحة، monorepo منظم              │
  │ عزل المستأجرين      │ جيد−     │ FK + JWT scope؛ يعتمد على انضباط الكود │
  │ RBAC                │ جيد      │ workspace + platform؛ bypass للadmin   │
  │ قاعدة البيانات      │ متوسط    │ schema غني؛ drift migrations           │
  │ الأمن               │ متوسط−   │ JWT؛ CORS مفتوح؛ لا rate limit عام     │
  │ الأداء/Scalability  │ متوسط−   │ monolith + PG؛ workers DB-backed       │
  │ الاختبارات          │ متوسط    │ ~175 ملف vitest؛ فجوات routes كبيرة    │
  │ التوثيق (OpenAPI)   │ ضعيف     │ ~79% مسارات API غير موثّقة              │
  │ CI/CD               │ ضعيف     │ لا GitHub Actions في الم repo            │
  │ المراقبة            │ متوسط−   │ pino فقط؛ لا APM/tracing مدمج          │
  │ On-Prem             │ جيد      │ wizard + docker-compose + PM2          │
  └─────────────────────┴──────────┴────────────────────────────────────────┘

1. تصميم البنية المعمارية (Architecture)
================================================================================

── 1.1 نمط عام ──

  [ops-platform SPA]  ──HTTP/JSON──►  [api-server Express]  ──Drizzle──►  [PostgreSQL]
         │                                    │
         │ @workspace/api-client-react        │ src/lib/* (domain logic)
         │ (Orval من OpenAPI جزئي)             │ src/routes/* (HTTP adapters)
         └────────────────────────────────────┘

  Monorepo pnpm:
    artifacts/api-server     → @workspace/api-server (Node 24, esbuild bundle)
    artifacts/ops-platform   → Vite/React 19 SPA
    lib/db                   → schema + migrations + pool
    lib/api-spec             → openapi.yaml
    lib/api-client-react     → hooks مولّدة
    lib/core-*               → events, workflows, permissions, audit (shared kernels)

── 1.2 طبقات المسؤولية (Layered responsibilities) ──

  ┌─────────────────────────────────────────────────────────────────────────┐
  │ Presentation   │ ops-platform/pages, components, hooks                │
  │ API boundary   │ routes/*.ts — validation, auth, HTTP mapping         │
  │ Application    │ api-server/src/lib/* — use cases, orchestration      │
  │ Domain cores   │ lib/core-events, core-workflows, core-permissions    │
  │ Persistence    │ lib/db schema + Drizzle queries                      │
  │ Infrastructure │ pino, SMTP, object storage, SSE stream               │
  └─────────────────────────────────────────────────────────────────────────┘

  مبدأ: Schema في lib/db؛ HTTP في routes؛ UI في ops-platform.
  انحراف: كثير من business logic داخل route handlers (خصوصاً hr.ts ~161 handler).

── 1.3 حدود الخدمات (Service boundaries) ──

  حالياً: **خدمة واحدة deployable** (api-server) — ليست microservices.

  Domains منطقية داخل monolith:
    • Core Ops (tickets, messages, calendar, users)
    • HCM (hr, workforce, leave, payroll)
    • Platform (tenants, commercial, subscriptions, governance)
    • Automation (workflows, forms, approval-runtime, events)

  نقاط قوة:
    • نشر واحد = تعقيد تشغيل أقل لـ On-Prem
    • transaction boundaries أسهل داخل PG واحد

  نقاط ضعف:
    • hr.ts + platform.ts + payroll = blast radius كبير
    • scaling أفقي يتطلب scale كامل API وليس module واحد
    • لا فصل read/write أو queue workers مستقل physically

── 1.4 مسارات Platform vs Workspace ──

  Platform (super_admin):
    /platform/tenants, /platform/users, commercial-*, governance-*

  Workspace (tenant):
    /tickets, /users, /hr/*, /self-service/*, /workflows

  Setup (public tier):
    /setup/*, /auth/*, /health

  Middleware chain:
    public → requireDatabase → workspaceAccessWriteGuard → routers

  مرجع: artifacts/api-server/src/routes/index.ts
         artifacts/api-server/src/middlewares/requireAuth.ts

4. الأمن والصلاحيات (Security & RBAC)
================================================================================

── 4.1 المصادقة (Authentication) ──

  • JWT Bearer في Authorization header
  • bcrypt لكلمات المرور
  • Payload: userId, workspaceId, role, customRoleId
  • Token في localStorage (ops-platform)
  • JWT_SECRET default "dev-secret-change-in-production" — خطر production

  مرجع: artifacts/api-server/src/routes/auth.ts
         artifacts/api-server/src/middlewares/requireAuth.ts

── 4.2 RBAC على مستوى Workspace ──

  أدوار built-in: admin | manager | member | super_admin

  Custom roles:
    workspace_custom_roles + workspace_role_permissions
    تُحمّل لـ member + customRoleId فقط

  requirePermission("hr.view") — admin/manager/super_admin يتجاوزون!

  API: /workspace-roles, /permissions
  UI: /roles — مصفوفة صلاحيات

── 4.3 RBAC على مستوى Platform (Super Admin) ──

  platform_user_permission_overrides
  requirePlatformPermission("tenants.read") إلخ
  Root owner policy: root-platform-owner-policy.ts

── 4.4 Guards إضافية ──

  workspaceAccessWriteGuard (P16-E):
    • يمنع POST/PUT/PATCH/DELETE عند read-only workspace
    • super_admin bypass
    • exempt: /platform/, /auth/, /setup, /health

  requireDatabase: 503 حتى يُكوَّن DB

── 4.5 فجوات أمنية ──

  | الفجوة                         | الخطورة | ملاحظة                          |
  |--------------------------------|---------|----------------------------------|
  | CORS origin: true (any origin) | متوسط   | cors_origins في settings غير م wired |
  | لا helmet                        | متوسط   | headers أمن HTTP                  |
  | لا rate limit على /auth/login  | عالي    | brute force                       |
  | webhook attendance بدون auth     | متوسط   | يحتاج HMAC/secret                 |
  | GET /platform/branding public  | منخفض   | مقصود؟                            |
  | RBAC inconsistent على routes   | متوسط   | hr/categories, leave = auth فقط   |
  | ~79% API بدون OpenAPI contract | متوسط   | drift UI/backend                  |

── 4.6 كيف تبني RBAC «حقيقي» (enterprise-grade) — توصيات ──

  1. Policy layer موحّد (CASL/OPA/custom) بدل requirePermission scattered
  2. Scope attributes: org_unit_id, department_id, employee_id في JWT claims
  3. admin/manager لا bypass تلقائي — roles = permission bundles
  4. Separation of Duties: approve ≠ create لنفس الكيان
  5. Audit immutable لكل grant/revoke permission
  6. Platform permissions منفصلة تماماً في middleware chain
  7. Periodic access review (موجود UI — يحتاج enforcement أقوى)

5. الأداء و Scalability
================================================================================

── 5.1 الوضع الحالي ──

  • Monolith Node single process (+ PM2 cluster optional)
  • Connection pool PG عبر @workspace/db
  • SSE stream للإشعارات (/stream)
  • لا CDN مدمج (settings: Coming soon)
  • لا Redis cache layer
  • hr.ts queries — بعضها heavy (import, bulk, export)

── 5.2 Bottlenecks متوقعة ──

  | Bottleneck              | عند                        | تأثير        |
  |-------------------------|----------------------------|--------------|
  | PostgreSQL writes       | bulk HR import, payroll run| latency عالي |
  | Single API process CPU  | workflow engine + HTTP     | saturation   |
  | N+1 queries             | employee detail nested tabs| بطء UI       |
  | Large esbuild bundle    | api-server 8.6mb dist      | cold start   |
  | SSE connections         | آلاف users                 | memory       |

── 5.3 كيف يتحمل آلاف المستخدمين — مسار نضج ──

  المرحلة 1 (0–500 concurrent):
    • PM2 cluster (ecosystem.config.cjs)
    • PG connection pool tuning
    • indexes على workspace_id + hot FKs
    • CDN للـ static assets

  المرحلة 2 (500–5000):
    • Read replica PG للتقارير/GET heavy
    • فصل workers (notifications, exports) process منفصل
    • Redis للـ session/rate-limit/cache
    • pagination إلزامي على كل list APIs

  المرحلة 3 (5000+):
    • Horizontal API behind load balancer (stateless JWT)
    • Queue (Redis/Rabbit) للـ import/payroll
    • Sharding أو DB-per-tenant للعملاء enterprise كبار
    • Object storage خارجي (S3-compatible) للمرفقات

  الوضع الحالي: مناسب لـ **مئات** users/workspace؛ يحتاج عمل لـ **آلاف**.

10. عزل Workspaces ومنع تسرب البيانات بين الشركات
================================================================================

── 10.1 نموذج العزل ──

  **Logical isolation** (shared database):
    • workspaces.id = tenantId
    • كل جدول تشغيلي: workspace_id NOT NULL FK
    • JWT يحمل workspaceId — handlers filter eq(table.workspaceId, req.workspaceId)

  **ليس** DB-per-tenant ولا schema-per-tenant.

── 10.2 آليات الدفاع ──

  1. FK cascade من workspaces
  2. requireAuth يثبت workspaceId من user row (ليس من client body)
  3. workspaceAccessWriteGuard — read-only mode
  4. Platform routes منفصلة (/platform/*) — super_admin permissions
  5. Subscription entitlements — quota limits

── 10.3 مخاطر تسرب (Tenant leakage) ──

  | السبب                              | الوقاية                          |
  |------------------------------------|----------------------------------|
  | query بدون workspaceId filter      | code review + lint rule + tests  |
  | IDOR: /resource/:id بدون scope     | always join workspace_id         |
  | super_admin cross-tenant access      | audit + least privilege          |
  | export/report cross-workspace      | validate workspace in job        |
  | cached data بدون tenant key        | prefix cache keys                |
  | SQL injection (نادر مع Drizzle)    | parameterized queries only       |
  | logs تطبع PII                      | pino redaction + policy          |

── 10.4 Enterprise isolation upgrade path ──

  • Row Level Security (RLS) في PostgreSQL:
      SET app.workspace_id = $
      POLICY tenant_isolation USING (workspace_id = current_setting(...))
  • Automated tenant isolation tests (create 2 workspaces, assert no cross-read)
  • DB-per-tenant للعملاء regulated (healthcare, gov)

── 12.3 Forms + smart workflow ──

  • forms.ts — audience, workflow binding, self-service
  • form-workflow-builder, form-audience-builder (UI)

── 12.4 فجوات Events/Workflows ──

  • ازدواج: technical workflows vs HR process templates
  • approval inbox vs legacy approvals.tsx
  • لا outbox pattern — event loss risk if listener fails mid-flight
  • Idempotency داخل metadata فقط — ليس HTTP header

── 12.5 Best practices roadmap ──

  1. Complete bus migration — remove direct dispatcher calls
  2. Transactional outbox table
  3. Unified approval model (leave + forms + tickets)
  4. Process template authoring UI
  5. Event schema registry (platform_event_registry exists — extend)

13. Cloud → On-Prem بدون انهيار
================================================================================

── 13.1 ما يدعمه اليوم ──

  ✓ Setup wizard (DB + owner account)
  ✓ data/.platform.json لـ DATABASE_URL
  ✓ Docker Compose (postgres + api + nginx)
  ✓ PM2 cluster
  ✓ No hard cloud dependency in core path (PG + SMTP optional)
  ✓ Single-tenant deploy possible (workspace واحد)

── 13.2 مخاطر الهجرة ──

  | Risk                         | Mitigation                           |
  |------------------------------|--------------------------------------|
  | Migration drift              | reconcile journal قبل cutover        |
  | Missing env secrets          | .env.example checklist               |
  | Object storage cloud-only    | S3-compatible MinIO on-prem          |
  | Email delivery               | local SMTP relay                     |
  | License/phone-home           | document offline mode                |
  | Large bundle cold start      | PM2 + adequate RAM                   |

── 13.3 Runbook Cloud → On-Prem ──

  1. Export PG dump من cloud
  2. Provision on-prem: Docker/VM + secrets
  3. Restore dump → run api (auto-migrate baseline)
  4. Configure SMTP, APP_URL, JWT_SECRET (rotate!)
  5. DNS cutover أو VPN
  6. Parallel run period — compare healthz + critical flows
  7. Decommission cloud after validation window

  لا تُنقل secrets أو JWT_SECRET بين environments.
