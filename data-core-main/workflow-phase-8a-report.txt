1 — TOPOLOGY GRAPH ARCHITECTURE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

File: artifacts/api-server/src/lib/workflows/topology.ts

The topology engine is a pure static module with zero runtime dependencies.
It sits entirely outside the execution stack and never writes to the DB,
never triggers the scheduler, and never invokes the executor.

Dependency graph:
  topology.ts → logger.ts   (structured observability events only)
  topology.ts → (nothing else)

WorkflowTopologyGraph shape:
  {
    nodes             TopologyNode[]        all steps as graph nodes
    edges             TopologyEdge[]        directed labelled edges
    entryPoints       number[]              stepIndex(es) of entry nodes
    terminalNodes     number[]              reachable nodes with no outgoing edges
    branchingNodes    number[]              condition steps with ≥2 distinct targets
    approvalNodes     number[]              all approval-type steps
    delayNodes        number[]              all delay-type steps
    unreachableNodes  number[]              not reachable from entry by any path
    isolatedSubgraphs number[][]           connected components disconnected from main
  }

TopologyNode shape:
  { stepIndex, stepName, stepType, arrayPos }

TopologyEdge shape:
  { fromIndex, toIndex, label: "linear" | "true_branch" | "false_branch" }

2 — STRUCTURAL EXTRACTION MODEL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Function: extractWorkflowTopology(rawSteps: unknown[]) → WorkflowTopologyGraph

Internal pipeline (all pure, no async):

  Step 1 — parseSteps()
    Pre-pass: builds stepIndex → arrayPos map for routing resolution.
    For each element: extracts index, type, name, config.
    For condition steps: resolves onTrueStepIndex / onFalseStepIndex to array
    positions — only valid forward references (same semantics as validation engine).

  Step 2 — buildSuccessorMap()
    For condition step:
      trueTarget  = onTrueArrayPos  ?? linearNext
      falseTarget = onFalseArrayPos ?? linearNext
      successors  = deduplicated [trueTarget, falseTarget] (filtered to valid positions)
    For non-condition step:
      successors  = [linearNext] if < length, else []

  Step 3 — buildEdges()
    Condition step, two distinct targets:
      true_branch edge  → trueTarget
      false_branch edge → falseTarget
    Condition step, both targets same: single true_branch edge only.
    Non-condition step: linear edge → linearNext (if exists).

  Step 4 — bfsReachable(pos=0, succs)
    Standard BFS. Returns Set<number> of reachable array positions.

  Step 5 — classify nodes
    terminalNodes:    reachable AND successors.length === 0
    branchingNodes:   condition type AND distinct successors ≥ 2
    approvalNodes:    type === "approval"
    delayNodes:       type === "delay"
    unreachableNodes: not in reachable set

  Step 6 — findIsolatedSubgraphs()
    Builds undirected adjacency (both edge directions).
    BFS/DFS to find all connected components.
    Main component = the one containing arrayPos=0.
    All other components = isolated subgraphs.

Routing model faithfulness:
  Matches the validation engine's parseSteps + buildSuccessors semantics exactly.
  Only forward references accepted. Self-loops and backward jumps treated as
  "no routing target" (fallback to linear advance).

3 — COMPLEXITY ANALYTICS FORMULAS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Function: computeTopologyAnalytics(graph, rawSteps) → TopologyAnalytics

All metrics use topological sort (Kahn's BFS) + DP. Deterministic on identical input.

  nodeCount
    parsed.length

  edgeCount
    edges.length

  maxBranchDepth   [consecutive condition chain depth]
    DP in topological order:
      dp[pos] = if type === "condition":
                  1 + max(dp[pred] for pred where pred.type === "condition")
                  minimum value: 1 (condition step with no condition predecessors)
                else: 0
    Result: max(dp.values())
    Invariant: resets to 0 at any non-condition step.

  longestPathEstimate   [max nodes on any entry→terminal path]
    DP in REVERSE topological order:
      terminal (no successors): dp[pos] = 1
      else:                     dp[pos] = 1 + max(dp[succ])
    Result: dp[0]

  terminalPathCount   [distinct execution paths, capped at 1024]
    DP in REVERSE topological order:
      terminal: dp[pos] = 1
      else:     dp[pos] = sum(dp[succ]), cap each accumulation at 1024
    Result: dp[0]
    Cap prevents integer overflow on deep branching graphs.

  branchingFactor   [avg outgoing edges per branching node]
    If branchingNodes.length > 0:
      sum of distinct successor counts / branchingNodes.length
    If no branching nodes:
      1.0 (linear flow)
    If no nodes:
      0.0

  approvalDensity   = approvalNodes.length / nodeCount   (0 if nodeCount=0)
  delayDensity      = delayNodes.length / nodeCount       (0 if nodeCount=0)

  unreachableCount         = unreachableNodes.length
  isolatedSubgraphCount    = isolatedSubgraphs.length

  conditionallyExecutedCount   [reachable but not on all paths]
    For each condition step with ≥2 distinct branches B1, B2:
      reach1 = BFS from B1
      reach2 = BFS from B2
      conditionally executed += (reach1 - reach2) ∪ (reach2 - reach1)
    Result: set.size (deduplication preserves correctness across multiple conditions)

4 — DEAD STRUCTURE DETECTION BEHAVIOR
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Two categories of dead structure are detected, each by a different algorithm:

A) Unreachable nodes
   Definition: steps not reachable from arrayPos=0 by any directed path.
   Algorithm: BFS from pos=0 → any step not in the visited set.
   Cause: condition step where both branches skip the same set of intermediate steps,
          and those steps have no other incoming edges.
   Reported in: graph.unreachableNodes, analytics.unreachableCount
   Risk: triggers TOPO-DEAD-MOD (1–2 steps) or TOPO-DEAD-HIGH (3+ steps)
   Observability: workflow_dead_structure_detected event

B) Isolated subgraphs
   Definition: connected components in the UNDIRECTED graph that have no
               undirected path to the main component (the one containing pos=0).
   Algorithm: undirected BFS → find all components → exclude main component.
   Cause: theoretically possible when steps have no edges connecting them to
          the main flow even in the undirected sense.
   Reported in: graph.isolatedSubgraphs, analytics.isolatedSubgraphCount
   Risk: triggers TOPO-ISOLATED (high) if count >= 1
   Observability: workflow_dead_structure_detected event

Structural note:
   In the current routing model (consecutive array positions with linear advance),
   purely isolated subgraphs are rare because undirected connectivity is preserved
   by linear advance edges. Unreachable nodes (in directed sense) are the more
   common dead-structure case.

5 — RISK ASSESSMENT MODEL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Function: assessTopologyRisk(graph, analytics) → TopologyRiskAssessment

Output shape:
  { level: "low"|"moderate"|"high"|"critical", reasons, dominantDrivers }

Thresholds (all tunable constants at top of topology.ts):

  RISK_HIGH_NODE_COUNT          = 20    → TOPO-SIZE-HIGH
  RISK_CRITICAL_NODE_COUNT      = 40    → TOPO-SIZE-CRITICAL
  RISK_HIGH_BRANCH_DEPTH        = 4     → TOPO-NEST-HIGH
  RISK_CRITICAL_BRANCH_DEPTH    = 7     → TOPO-NEST-CRITICAL
  RISK_HIGH_BRANCHING_PATHS     = 8     → TOPO-PATHS-HIGH
  RISK_CRITICAL_BRANCHING_PATHS = 32    → TOPO-PATHS-CRITICAL
  RISK_HIGH_APPROVAL_DENSITY    = 0.5   → TOPO-APPROVAL-OVERLOAD
  RISK_HIGH_DELAY_DENSITY       = 0.4   → TOPO-DELAY-HIGH
  MAX_PATH_COUNT_CAP            = 1024

Risk codes emitted:
  TOPO-SIZE-HIGH / TOPO-SIZE-CRITICAL      excessive step count
  TOPO-NEST-MOD  / TOPO-NEST-HIGH / TOPO-NEST-CRITICAL   condition chain depth
  TOPO-PATHS-MOD / TOPO-PATHS-HIGH / TOPO-PATHS-CRITICAL terminal path explosion
  TOPO-APPROVAL-MOD / TOPO-APPROVAL-OVERLOAD              approval density
  TOPO-DELAY-MOD / TOPO-DELAY-HIGH                        delay density
  TOPO-DEAD-MOD / TOPO-DEAD-HIGH                          unreachable steps
  TOPO-ISOLATED                                            isolated subgraphs
  TOPO-FANOUT-MOD                                         avg branching factor > 2

Overall level: max(all individual reason levels)
dominantDrivers: reason codes at the overall maximum level (moderate or above only)

Rule: "low" is the default when zero risk reasons are triggered.

All four events emitted via logger.info (pino structured JSON) inside analyzeTopology().
No DB writes. Fire-and-forget.

A) workflow_topology_extracted
   Emitted: always, on every analyzeTopology() call
   Fields:  action, workflowId, workflowVersion, workspaceId,
            nodeCount, edgeCount, branchDepth, unreachableCount, riskLevel

B) workflow_topology_risk_assessed
   Emitted: when risk.level >= "moderate"
   Fields:  action, workflowId, workflowVersion, workspaceId,
            riskLevel, branchDepth, nodeCount, terminalPaths, dominantDrivers

C) workflow_dead_structure_detected
   Emitted: when unreachableCount > 0 OR isolatedSubgraphCount > 0
   Fields:  action, workflowId, workspaceId,
            unreachableCount, isolatedSubgraphCount, nodeCount, riskLevel

D) workflow_branching_pressure_detected
   Emitted: when maxBranchDepth >= 4 OR terminalPathCount >= 8
   Fields:  action, workflowId, workspaceId,
            branchDepth, terminalPathCount, branchingFactor, riskLevel

Route-level event (additional):
   workflow_topology_requested — emitted via req.log.info in the route handler.
   Fields: event, workflowId, workspaceId, nodeCount, riskLevel, actorId

9 — REMAINING TOPOLOGY INTELLIGENCE GAPS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The following are intentionally deferred and NOT part of P8-A:

A) Graph visualization UI
   No visual rendering of the topology graph. The API returns the raw graph
   data structure — rendering is left to a future frontend phase.

B) Cross-workflow topology comparison
   No diffing between two workflow versions' topologies. Useful for change
   auditing but requires historical version fetching logic.

C) Dynamic reachability (runtime-aware)
   The current engine is purely structural. It does not account for runtime
   conditions such as field values, user roles, or external data that would
   affect which branches are actually taken in practice.

D) Topology-based activation gating
   Currently topology analysis is informational only. A future phase could
   block activation of critical-risk workflows or require admin override.

E) Topology trends over versions
   No tracking of how topology metrics change across workflow versions.
   A future phase could store topology snapshots per version in governance_snapshots.

F) Multi-step condition chain merging (convergence detection)
   The current model detects which steps are conditionally executed, but does
   not identify explicit convergence points (where branches re-merge). A future
   phase could mark convergence nodes explicitly.

G) Circular reference detection in malformed inputs
   The validator rejects backward/self-loop references before topology runs,
   but topology.ts does not independently detect cycles. Defensive cycle
   detection (DFS visited-stack) could be added for belt-and-suspenders safety.

H) Weighted path cost estimation
   longestPathEstimate counts steps, not wall-clock time. A future phase could
   combine with delay step config to estimate maximum calendar duration.
