2. REMEDIATION OUTCOME INTELLIGENCE ARCHITECTURE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

2.1  Layer Design

  P10-F adds a pure read-only analytics layer above P10-E's execution records.
  No new DB table — analytics are computed on-demand from existing execution data.

  Layer F — Pure Analytics Engine (read-only, sync, no DB, no mutations)
    artifacts/api-server/src/lib/workflows/remediation-outcome-intelligence.ts

    Core functions:
      computeOutcomeProfile(ws, type, records)  → RemediationOutcomeProfile
      evaluateRemediationOutcomes(records)       → RemediationOutcomeProfile[]
      computeOperatorProfile(opId, records)      → OperatorRemediationProfile
      evaluateOperatorProfiles(records)          → OperatorRemediationProfile[]
      buildPlatformEffectivenessSummary(records) → PlatformEffectivenessSummary

    Analytics primitives:
      computeSuccessRate(records)              → number (0.0–1.0)
      computeRollbackFrequency(records)        → number (0.0–1.0)
      computeAbandonmentRate(records)          → number (0.0–1.0)
      computeAverageRecoveryDuration(records)  → number (ms, or -1 if no data)
      computeMttrTrend(records)               → MttrTrend_Result
      detectChronicRecurrence(records, days)  → ChronicRecurrenceResult
      scoreEffectiveness(metrics)             → RemediationEffectivenessScore

  Route Layer — DB Access + Analytics Dispatch
    3 read-only routes in artifacts/api-server/src/routes/platform.ts:
      GET /platform/reliability/outcomes      → outcome profiles
      GET /platform/reliability/operators     → operator profiles
      GET /platform/reliability/effectiveness → platform effectiveness summary

    Route handler pattern:
      1. Query remediationExecutionAttemptsTable with optional filters
      2. Map DB rows to ExecutionRecord slim projection
      3. Pass to pure engine function
      4. Emit structured observability event
      5. Return JSON

2.2  Full Intelligence Chain (P10-A → P10-F)

  P10-A: Live reliability domain evaluation (per-capture snapshot)
  P10-B: Append-only snapshot persistence + incident lifecycle tracking
  P10-C: Operator advisory recommendations + trend analysis
  P10-D: Human-in-the-loop orchestration + remediation lifecycle
  P10-E: Explicit-confirmation execution tracking + rollback recording
  P10-F: Remediation outcome analytics + effectiveness intelligence (read-only)

  Full data flow:
    Capture → Incident → Recommendation → Orchestration → Execution → Analytics

  P10-F reads exclusively from: remediation_execution_attempts (P10-E table)
  P10-F writes to: nothing
  P10-F produces: pure value objects (serialized to JSON in API responses)

2.3  ExecutionRecord Projection

  Route handlers project DB rows to a slim ExecutionRecord type before
  passing to engine functions. This decouples the analytics engine from the
  DB schema — the engine only sees the fields it needs:

    executionId, workspaceId, executionType, initiatedBy, confirmedBy,
    executionStatus, rollbackStatus, createdAt, confirmedAt, executedAt,
    completedAt, rolledBackAt, abandonedAt

  DB columns not needed for analytics (executionNotes, executionEvidence,
  confirmationMode, actionId) are not projected — reducing type surface.

3. EFFECTIVENESS SCORING SEMANTICS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

3.1  Five-Tier Score: RemediationEffectivenessScore

  Tiers (evaluated in order, first match wins):

  ineffective      — successRate < 0.30  OR  rollbackFrequency > 0.60
                     Indicates: remediation attempts are mostly failing or being
                     reversed. The underlying issue is likely not addressed by
                     the execution type in use.

  unstable         — successRate < 0.50  OR  rollbackFrequency > 0.40
                     OR  chronicRecurrence = true
                     Indicates: execution outcomes are inconsistent. Chronic
                     recurrence alone elevates to unstable regardless of rates.

  acceptable       — successRate >= 0.50 AND rollbackFrequency <= 0.40
                     Indicates: more than half of executions complete, with
                     moderate rollback activity. Satisfactory but improvable.

  effective        — successRate >= 0.70 AND rollbackFrequency <= 0.25
                     AND abandonmentRate <= 0.15
                     Indicates: execution quality is high. Rollbacks are
                     infrequent and most operators follow through to completion.

  highly_effective — successRate >= 0.85 AND rollbackFrequency <= 0.10
                     AND abandonmentRate <= 0.10
                     Indicates: near-optimal execution outcomes. Rollbacks and
                     abandonments are rare edge cases, not systemic issues.

3.2  Input Metrics for scoreEffectiveness()

  successRate:        completed / (completed + rolled_back)
                      Denominator excludes abandoned — abandonment is a separate
                      signal from failure. An operator who abandoned 9/10 executions
                      is not the same as one who completed 9/10.

  rollbackFrequency:  rolled_back / total_executions
                      Denominator includes all records — rollback frequency is
                      meaningful relative to total execution volume, not just
                      completed ones.

  abandonmentRate:    abandoned / total_executions
                      Measures operator follow-through. High abandonment may
                      indicate confirmation friction or scope mismatch.

  chronicRecurrence:  bool from detectChronicRecurrence()
                      When true, elevates the score to "unstable" regardless
                      of rate metrics — repeated executions on the same
                      workspace+type suggest the root cause is not resolved.

3.3  Score Semantics — What They Do NOT Mean

  ✗ Scores are NOT used for operator enforcement or HR actions.
  ✗ Scores are NOT used to automatically adjust fairness policies.
  ✗ Scores are NOT used to block or gate future orchestration creation.
  ✗ Scores are NOT used to trigger automatic remediation actions.
  ✗ "ineffective" does NOT mean an execution type should be disabled.

  Scores are operator-visible intelligence ONLY. They surface patterns for
  human review. Acting on them requires a new orchestration action (P10-D)
  followed by a new execution attempt (P10-E) — both operator-initiated.

4. RESILIENCE ANALYTICS MODEL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

4.1  Six Analytics Primitives

  A) execution success rate
     completed / (completed + rolled_back)
     Measures: how often executions that reach an outcome are successful.
     Excludes: abandoned (no outcome), active (no outcome yet).

  B) rollback failure rate
     rolled_back / total_executions
     Measures: fraction of all attempted executions that were reversed.
     High values suggest: execution steps are hard to sustain, or
     operators are reversing too quickly (low confidence in changes).

  C) MTTR improvement trend
     Compare: average recovery duration (confirmedAt → completedAt) of
     older half vs newer half of completed records.
     Improving: newer half faster by ≥ 10%
     Degrading: newer half slower by ≥ 10%
     Stable: within ±10%
     Insufficient data: fewer than 4 qualifying records.

  D) chronic incident recurrence
     executionsInWindow > 3 AND recurrenceRate > 1.0 executions/day
     Signals: the underlying reliability issue is not resolving; execution
     attempts are recurring at a rate that suggests systematic failure.

  E) execution abandonment frequency
     abandoned / total_executions
     Measures: how often operators create and then abandon execution attempts.
     High values may indicate: confirmation flow friction, scope mismatch,
     or operators creating attempts prematurely.

  F) remediation stability ratio
     completed / (completed + rolled_back)
     Equivalent to successRate. When viewed across execution types, reveals
     which types are inherently more stable vs. those prone to reversal.

4.2  RemediationOutcomeProfile Fields

  profileId                  — "<workspaceId>:<executionType>" (unique key)
  workspaceId                — target workspace
  executionType              — one of 8 P10-E execution types
  totalExecutions            — all records for this workspace+type
  successfulExecutions       — count with executionStatus="completed"
  rolledBackExecutions       — count with executionStatus="rolled_back"
  abandonedExecutions        — count with executionStatus="abandoned"
  successRate                — rounded to 3 decimal places (0.000–1.000)
  rollbackFrequency          — rounded to 3 decimal places
  abandonmentRate            — rounded to 3 decimal places
  averageRecoveryDuration    — ms (confirmedAt→completedAt avg); -1 if no data
  chronicFailureRecurrence   — bool (from detectChronicRecurrence with 30d window)
  mttrTrend                  — improving | stable | degrading | insufficient_data
  effectivenessScore         — five-tier score
  lastEvaluatedAt            — ISO 8601 of when this profile was computed

4.3  MttrTrend_Result Fields

  trend           — improving | stable | degrading | insufficient_data
  firstHalfAvgMs  — average recovery ms, older half of qualifying records
  secondHalfAvgMs — average recovery ms, newer half of qualifying records
  improvementPct  — positive = shorter recovery (improving); rounded to 1dp
  sampleSize      — count of qualifying (completed + both timestamps) records

4.4  ChronicRecurrenceResult Fields

  isChronicRecurrent  — bool (count > 3 AND rate > 1.0/day)
  recurrenceRate      — executions per day in the window
  windowDays          — the window used (default: 30)
  executionsInWindow  — count of records with createdAt within window

5. OPERATOR VISIBILITY MODEL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

5.1  OperatorRemediationProfile Fields

  operatorId               — initiatedBy value (operator identifier)
  initiatedExecutions      — count of all executions initiated by operator
  confirmedExecutions      — count where confirmedBy is non-null
  completedExecutions      — count with executionStatus="completed"
  rolledBackExecutions     — count with executionStatus="rolled_back"
  abandonedExecutions      — count with executionStatus="abandoned"
  completionRate           — completedExecutions / initiatedExecutions
  rollbackFrequency        — rolledBackExecutions / initiatedExecutions
  executionStabilityScore  — completedExecutions / (completed + rolledBack)
                             Range 0.0–1.0. 0.0 if none have reached outcome.
  lastActivityAt           — ISO 8601 of most recent execution created by operator

5.2  Operator Profile Safety Semantics

  ✗ NO ranking system. Operators are NOT ranked against each other.
  ✗ executionStabilityScore is NOT a performance indicator for HR purposes.
  ✗ completionRate is NOT used to gate operator access to create executions.
  ✗ Profiles are NOT streamed to any external HR or performance system.

  Profiles exist for: operator self-awareness, incident response coordination,
  and platform-level audit of who is performing remediation actions.

5.3  No Confirmed-By Attribution Gap

  completionRate is computed as completed / initiated (not confirmed / initiated).
  Some executions may be confirmed by a different operator than the one who
  initiated them. The confirmedExecutions count reflects executions where any
  confirmation occurred, not necessarily by the initiating operator.
  This is intentional — the initiatedBy field is the stable identifier for
  operator profile grouping.

Four structured events emitted by the analytics layer:

A) remediation_outcome_profile_evaluated
   Fired: inside GET /platform/reliability/outcomes route.
   Signals that an outcome analytics query was processed.
   Fields: workspaceId, executionType, effectivenessScore="evaluated",
           rollbackFrequency=0, operatorId="", action="outcomes_queried"

B) execution_effectiveness_scored
   Fired: inside GET /platform/reliability/effectiveness route.
   Signals that a platform effectiveness summary was computed.
   Fields: workspaceId, executionType, effectivenessScore (real score),
           rollbackFrequency (real rate), operatorId="",
           action="effectiveness_evaluated"

C) rollback_trend_detected
   Fired: inside GET /platform/reliability/effectiveness route,
          only when overallRollbackRate > 0.40.
   Signals: high rollback rate detected, warrants operator review.
   Fields: workspaceId, executionType, effectivenessScore (real score),
           rollbackFrequency (real rate), operatorId="",
           action="high_rollback_rate_detected"
   Important: event is informational only — no automatic action is taken.

D) operator_effectiveness_updated
   Fired: inside GET /platform/reliability/operators route.
   Signals that an operator analytics query was processed.
   Fields: workspaceId=0, executionType="all", effectivenessScore="evaluated",
           rollbackFrequency=0, operatorId (filtered or "all"),
           action="operators_queried"

All events: logger.info() with structured JSON — no console.log.
All events are informational read-path events — no write or mutation
is triggered by any event emission.

  - Protected by requireAuth + requireSuperAdmin middleware chain
  - READ-ONLY: no INSERT, UPDATE, or DELETE on any table
  - Deterministic: same DB state + same inputs → same outputs
  - Append-only semantics preserved: analytics never modify source records
  - Audit-safe: all response fields are JSON-safe ISO strings and numbers

GET /platform/reliability/outcomes
  Auth:    super_admin
  Params:  workspaceId?, executionType?, limit (1–200, default 100), offset
  Compute: evaluateRemediationOutcomes(records)
  Returns: { profiles: RemediationOutcomeProfile[], total, limit, offset }
  Note:    profiles are grouped by (workspaceId, executionType) and sorted
           by profileId alphabetically.

GET /platform/reliability/operators
  Auth:    super_admin
  Params:  operatorId?, limit (1–200, default 100), offset
  Compute: evaluateOperatorProfiles(records) → optional operatorId filter
  Returns: { operators: OperatorRemediationProfile[], total, limit, offset }
  Note:    profiles are sorted by operatorId alphabetically. No ranking.

GET /platform/reliability/effectiveness
  Auth:    super_admin
  Params:  workspaceId?, executionType?
  Compute: buildPlatformEffectivenessSummary(records)
  Returns: { summary: PlatformEffectivenessSummary }
  Note:    byExecutionType breakdown always included.
           rollback_trend_detected event emitted if overallRollbackRate > 0.40.

10. REMAINING RESILIENCE GAPS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

P10-F is intentionally minimal and read-only. Remaining gaps:

  1. ANALYTICS ARE ON-DEMAND, NOT CACHED
     Every GET /outcomes request re-queries all execution records and
     recomputes all profiles from scratch.
     Gap: for platforms with thousands of execution records, this becomes
          expensive. A materialized analytics cache or periodic pre-computation
          job would improve performance. Not added — premature optimization
          without load data.

  2. NO TIME-WINDOWED ANALYTICS
     Profiles use the full lifetime of execution records. There is no
     built-in mechanism to compute "outcomes in the last 30 days" vs.
     "outcomes in the last 7 days" for trend comparison.
     Gap: windowDays parameter for GET /outcomes would enable temporal slicing.

  3. NO CROSS-WORKSPACE COMPARISON
     Profiles are independent per workspace — there is no "how does workspace X
     compare to workspace Y for the same execution type" view.
     Gap: aggregate analytics across workspaces would require careful tenant
          isolation design to avoid information leakage between workspace admins.

  4. NO OUTCOME-LINKED INCIDENT ANALYSIS
     Execution outcomes are not correlated to incident resolution timelines (P10-B).
     Gap: "did this execution correlate with incident resolution?" analysis
          would require linking execution.actionId → orchestration.incidentId
          → incident.resolvedAt — a three-hop join not currently performed.

  5. NO EXECUTION TYPE RECOMMENDATION FEEDBACK LOOP
     Analytics are surfaced to the operator but do not feed back into P10-C's
     advisory recommendations. A workspace with effectivenessScore="ineffective"
     for operational_intervention type does not receive a differently-weighted
     recommendation as a result.
     Gap: a read-only advisory signal from analytics back to recommendations
          would close the feedback loop without introducing automation.

  6. ROLLBACK FAILURE RATE NOT SEPARATELY ANALYSED
     rollbackStatus values (pending/completed/failed) are stored in P10-E
     but the analytics engine does not separately surface "rollback completion
     rate" (i.e., how often rollbacks themselves succeed vs. fail).
     Gap: a dedicated rollbackOutcomeRate metric would distinguish between
          "execution reversed cleanly" and "rollback attempt also failed."

  7. NO EVIDENCE PATTERN ANALYSIS
     executionEvidence is a string array in P10-E but is not projected into
     ExecutionRecord for analytics — it is excluded from the slim projection.
     Gap: evidence code frequency analysis (e.g., "backlog_depth:* appears in
          78% of escalation_stabilization executions") would identify common
          patterns without requiring AI.

  8. MTTR TREND REQUIRES MINIMUM 4 RECORDS
     Workspaces with fewer than 4 completed executions of a given type always
     return trend="insufficient_data". For newly onboarded workspaces this is
     a permanent "no data" state until execution volume grows.
     Gap: a lower threshold (2 records) with a lower-confidence flag would
          provide earlier trend signal.
