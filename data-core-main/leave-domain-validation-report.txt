  These two raw failures were caused by test-setup issues, not system bugs.
  Both behaviors are correct and desirable.

  [E7] Sarah withdrawing John's approved request
  ───────────────────────────────────────────────
  Expected : 422 (cannot withdraw approved)
  Received : 403 ("You can only withdraw your own leave requests")
  Analysis : The ownership guard runs before the status guard. Returning 403
             for a cross-user withdraw is MORE correct than 422: it does not
             reveal the state of someone else's request. Both guards fire in
             the right order. No code change needed.

  [F6] Balance insufficiency with overlapping dates
  ──────────────────────────────────────────────────
  Expected : 422 (insufficient balance)
  Received : 409 (overlapping dates)
  Analysis : The test accidentally used Aug 2026 dates that overlapped with an
             existing request left from the F4 overlap test. The overlap check
             correctly runs before the balance check. The balance validation
             itself was confirmed working via F6b: a 23-day request against a
             10-day entitlement returned 422 with a clear error message:
             "Insufficient leave balance. Available: 10.0 days, Requested: 23 days"
             No code change needed.

  POST /hr/leave-requests — validation priority:
    1. Auth (requireAuth)
    2. Schema validation (Zod: dates, leaveType, policyId required)
    3. Date ordering (startDate ≤ endDate)
    4. Business-day count > 0 (no weekend-only ranges)
    5. Policy exists and belongs to workspace
    6. Balance record exists for employee + policy
    7. Date overlap check against existing active requests (409)
    8. Sufficient balance check (422)
    9. Submit + reserve balance (FOR UPDATE) + create approval step

  PATCH /hr/leave-requests/:id/approve|reject|withdraw:
    1. Auth (requireAuth)
    2. Request exists in workspace
    3. Identity/role check (403 if not authorized actor)
    4. Status check — valid transition (422 if terminal state)
    5. DB update under transaction (FOR UPDATE)
    6. Balance update (release pending / commit used)
    7. Emit domain event → activity_log + notification

§8  RECOMMENDATIONS FOR PHASE 2
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Phase 1 is production-ready. For Phase 2 consider:

  1. Multi-step approval chains
     Leave requests that require multiple sequential approvers (e.g., manager
     → HR → department head). step_order column already in schema.

  2. Cancellation of approved requests
     Allow employees to cancel an already-approved future request; requires
     releasing used balance and HR notification.

  3. Public holidays calendar
     Business-day counting currently skips Sat/Sun. Integrate a holidays
     table to exclude country-specific public holidays.

  4. Carry-over & accrual rules
     End-of-year entitlement carry-over, monthly/quarterly accrual schedules.

  5. Leave request attachments
     Medical certificates for sick leave, supporting documents.
     Object Storage integration is already available in the platform.

  6. Frontend UI
     - Employee leave dashboard (balance widget, history, submit form)
     - Manager pending approvals queue
     - HR admin overview (all requests, policy management)
     - Calendar view of team leave

  7. Delegation
     Allow a manager to delegate approval authority during their own absence.
