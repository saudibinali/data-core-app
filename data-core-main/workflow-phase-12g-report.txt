One page, fully rewritten from the P12-A placeholder:
  /super-admin/governance/evidence-packages
  → SuperAdminGovernanceEvidence  (rewritten, 8 sections)

Route is already registered in App.tsx from P12-A — no routing change needed.
Protected by SuperAdminRoute — non-super-admins redirected to /sign-in.

Page sections (top to bottom):
  1. GovernanceSectionHeader + GovernanceReadOnlyNotice
  2. Evidence readiness overview banner (useGovernanceEvidenceReadiness)
  3. Package integrity summary cards (5 status buckets, count per status)
  4. Scope selector — 6 tabs, drives useGovernanceEvidencePackages(scope)
  5. Filter bar — scope · integrityStatus · workspaceId · section (text)
  6. Expandable package list with full detail views
  7. Export placeholder notice (read-only info card)
  8. Safety annotation footer line

Data flow (read-only):
  useGovernanceEvidencePackages(scope)  → evidenceQuery.data.package
  useGovernanceEvidenceReadiness()      → readinessQuery.data.package

Both hooks are pre-existing from P11-F. No new hooks added.
All API calls are GET-only. No mutation hooks present.

Normalisation: API may return either a single package object or an array.
The page normalises both to a packages[] array with:
  Array.isArray(rawPkg) ? rawPkg : [rawPkg]

13. REMAINING EVIDENCE PACKAGE UI GAPS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Intentionally deferred — require backend work or are out of P12-G scope:

a) Multi-package API response
   Current API returns a single package per scope. A future list endpoint
   (GET /evidence-packages?scope=x&limit=n) would allow browsing multiple
   packages and comparing integrity across time. The page already handles
   array responses via the normalisation step.

b) Per-package detail navigation
   Currently all packages load in the same page via accordion. A dedicated
   /super-admin/governance/evidence-packages/:id route would allow deep
   linking to a specific package for sharing in governance review sessions.

c) Section-to-layer cross-linking
   GovernanceEvidenceSectionCoverageGrid shows expectedSourceLayer labels.
   A future pass could deep-link each section row to the relevant topology
   page (topology or audit integrity page) for boundary context.

d) Evidence reference deep resolution
   GovernanceEvidenceReferenceList shows referenceId values. When a
   reference type is "audit_chain_entry" or "execution_record", a future
   pass could resolve the ID to a specific audit trail event on the
   Audit Integrity page.

e) Package history / version list
   No package history endpoint exists. When implemented, a sidebar of
   "previous versions by scope" would allow comparing hash changes over time,
   supporting tamper detection review without needing diff execution.

f) Readiness dimension ↔ evidence section linking
