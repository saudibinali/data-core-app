2. IMMUTABLE AUDIT ARCHITECTURE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

2.1  Design Principles

  The audit chain is modelled as a hash-linked list of immutable entries.
  Each entry is a row in `compliance_audit_chains` with:

    currentAuditHash = SHA-256(
      (previousAuditHash ?? "GENESIS") |
      eventType |
      entityId |
      operatorId |
      occurredAt.toISO() |
      JSON.stringify(payload, sortedKeys)
    )

  Any modification to any field in a persisted row would produce a different
  recomputed hash — detected immediately by verifyAuditIntegrity().

  Payload keys are deterministically sorted in JSON.stringify() to eliminate
  hash variability from insertion-order differences in JavaScript objects.

2.2  Chain Structure per Entity

  Each (entityType, entityId) combination forms its own sub-chain:

    Entry 0: previousAuditHash = null ("GENESIS" in hash input)
    Entry 1: previousAuditHash = Entry 0's currentAuditHash
    Entry 2: previousAuditHash = Entry 1's currentAuditHash
    ...

  Multiple entities can have their audit chains recorded in the same table —
  they are logically separate chains sharing the same storage.

  Each new entry's previousAuditHash must be supplied by the caller (it is the
  currentAuditHash of the most recent entry for that entityId). The engine
  computes the new currentAuditHash from inputs — it is never caller-supplied.

2.3  Entity Types

  incident              — reliability domain incidents from P10-B
  recommendation        — advisory recommendations from P10-C
  orchestration_action  — orchestration lifecycle from P10-D
  execution_attempt     — execution lifecycle from P10-E
  platform_event        — cross-workspace platform governance events

2.4  No Blockchain, No Distributed Ledger

  The chain is implemented as a standard PostgreSQL table with hash values
  stored as text columns. The hash chain is NOT a blockchain:
  - No consensus mechanism
  - No distributed validation
  - No block structure
  - No cryptographic signing of entire blocks
  It is an integrity check mechanism: if the DB is tampered with, the recomputed
  hashes will not match the stored hashes, revealing the modification. It does
  not prevent a sufficiently privileged actor from modifying both the data and
  the stored hashes — that would require external signing infrastructure, which
  is out of scope for P11-A.

3. INTEGRITY VERIFICATION SEMANTICS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

3.1  AuditIntegrityStatus — Five Tiers

  verified      Hash chain is intact. Recomputed hash matches stored hash.
                previousAuditHash resolves correctly to a known entry.
                This is the expected state for all entries under normal operation.

  warning       Minor anomaly detected. Currently reserved for future use
                (e.g., large time gap between occurredAt and recordedAt,
                which may indicate delayed audit recording).

  compromised   Recomputed hash does NOT match stored currentAuditHash.
                This is the definitive signal that a stored field (payload,
                operatorId, eventType, occurredAt, or previousAuditHash) has
                been modified after the entry was written.
                Severity: CRITICAL — requires immediate forensic review.

  orphaned      Entry's previousAuditHash is non-null but does not match any
                currentAuditHash in the evaluated entry set.
                This indicates a gap: either a preceding entry was deleted, or
                the entry was inserted with an incorrect previousAuditHash.
                Severity: HIGH — chain continuity is broken.

  incomplete    Reserved for future use: expected entries are missing
                (e.g., gap in sequence number when sequence IDs are used).
                Currently not emitted by the engine — included for forward
                compatibility with future sequence-based chain validation.

3.2  Severity Ordering (for overallStatus computation)

  compromised > orphaned > incomplete > warning > verified

  overallStatus = worst individual status across all evaluated entries.
  A single compromised entry elevates the entire report to "compromised".

3.3  verifyAuditIntegrity() Process

  1. Build a Set of all currentAuditHash values in the entry set.
  2. For each entry:
     a. Recompute its hash from its own fields.
     b. Compare recomputed hash to stored currentAuditHash.
     c. Check if previousAuditHash (if non-null) is present in the hash Set.
     d. Assign computedStatus: compromised / orphaned / verified.
  3. Count by status, compute overallStatus.
  4. Return AuditIntegrityReport with per-entry results + summary counts.

  NO SELF-REPAIR: the engine never updates stored hashes or integrity statuses.
  It only computes and reports. Acting on findings requires a human operator.

3.4  detectTamperAnomalies() — Targeted Tamper Detection

  For each entry, recomputes the hash and compares to the stored value.
  Returns only the entries where the hashes diverge, with:
    - chainId
    - storedHash (the hash currently in DB)
    - recomputedHash (what the hash should be given the current field values)
    - severity ("compromised")
    - detectedAt (ISO timestamp of the detection run)

  Used internally by verifyAuditIntegrity() and available as a standalone
  function for targeted integrity checks.

3.5  detectOrphanedRecords() — Link Verification

  Scans for entries whose previousAuditHash references a hash not present
  in the evaluated entry set.
  - Genesis entries (previousAuditHash = null) are never considered orphaned.
  - An orphan may indicate: a deleted predecessor, a cross-batch query
    that excluded the predecessor, or a corrupt previousAuditHash value.

  NOTE: When querying a subset of entries (e.g., filtered by workspaceId),
  orphan detection may produce false positives if the predecessor entry
  exists in the DB but was excluded by the filter. This is expected behavior —
  callers should be aware that orphan detection is reliable only when the
  full entry set for an entityId is loaded.

4. FORENSIC RECONSTRUCTION MODEL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

4.1  reconstructAuditTimeline(entityId, allEntries) → ForensicTimeline

  Process:
    1. Filter allEntries to those matching entityId.
    2. Sort chronologically by occurredAt (ASC) — deterministic forensic ordering.
       occurredAt represents when the event happened in the source system,
       not when it was recorded in the audit table.
    3. For each event: recompute hash, check orphan status, assign integrityStatus.
    4. Compute worst-case chainIntegrity across all events.
    5. Return ForensicTimeline with fully annotated event list.

  ForensicTimeline fields:
    entityId         — the queried entity
    entityType       — inferred from first matching entry (null if no entries)
    totalEvents      — count of audit entries for this entity
    chainIntegrity   — worst-case integrity status (compromised/orphaned/verified)
    events[]         — ForensicTimelineEvent[], sorted ASC by occurredAt
    reconstructedAt  — ISO timestamp of when this reconstruction was run

  ForensicTimelineEvent fields:
    chainId                  — unique identifier of this audit entry
    eventType                — what happened (e.g., "execution_confirmed")
    operatorId               — who performed the action
    occurredAt               — when the event happened (ISO 8601)
    recordedAt               — when the audit entry was written (ISO 8601)
    integrityStatus          — per-event integrity classification
    retentionClassification  — governance retention tier for this event
    hashValid                — bool: recomputed hash === stored hash
    payload                  — frozen snapshot of event-specific data

4.2  Full Lifecycle Chain Reconstruction

  The audit trail connects to the full P10-A → P10-F resilience chain:

    incident                  (P10-B)
      └─ recommendation       (P10-C)
           └─ orchestration   (P10-D)
                └─ execution  (P10-E)
                     └─ outcome analytics (P10-F — read-only)

  Each layer records audit entries with its own entityId:
    - incident entityId      = incidentId (P10-B)
    - recommendation entityId = recommendationId (P10-C)
    - orchestration entityId = actionId (P10-D)
    - execution entityId     = executionId (P10-E)

  The GET /platform/compliance/forensics/:entityId endpoint surfaces the
  full audit chain for any of these entity IDs. Cross-entity reconstruction
  (e.g., "all audit events for incident X and all its child entities") is
  a future enhancement — currently each entity's chain is independent.

4.3  Forensic Ordering Guarantee

  Events are always sorted by occurredAt (ASC), not recordedAt.
  This ensures forensic timelines reflect real-world event sequence even when:
  - Audit entries are recorded with slight delays
  - Entries are added retroactively (backfill scenarios)
  - Clock skew produces slight recordedAt ordering inversions

  The distinction between occurredAt and recordedAt is preserved in every
  ForensicTimelineEvent — investigators can see both timestamps.

5. RETENTION GOVERNANCE FOUNDATIONS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

5.1  RetentionClassification — Four Tiers

  operational           Day-to-day events (status updates, captures, progressions).
                        Standard retention period.
                        Examples: snapshot_captured, status_updated

  governance            Policy, configuration, and orchestration lifecycle events.
                        Longer retention — required for governance audit trails.
                        Examples: orchestration_resolved, policy_updated,
                                  recommendation_acknowledged, workspace_created

  compliance_sensitive  Confirmation, approval, authentication, and access events.
                        Treated as sensitive — may be subject to GDPR / SOC 2 rules.
                        Examples: execution_confirmed, access_granted,
                                  password_changed, approved

  forensic_critical     Rollbacks, abandonments, cancellations, and integrity
                        failures. Maximum retention — cannot be deleted before
                        review. Triggers anomaly notifications.
                        Examples: execution_rolled_back, execution_abandoned,
                                  integrity_anomaly_detected, tamper_detected

5.2  Classification Algorithm

  classifyRetention(entityType, eventType) uses keyword matching on the
  lowercase eventType string, evaluated in strict priority order:

    1. forensic_critical — matches: rollback, rolled_back, abandon, cancelled,
                           tamper, compromised, integrity_failure, integrity_anomaly
    2. compliance_sensitive — matches: confirm, approved, rejected, access,
                              permission, auth, sign_in, password
    3. governance — matches: orchestrat, policy, configuration, resolution,
                    resolved, recommendation, workspace_created, workspace_updated;
                    OR entityType is orchestration_action or recommendation
    4. operational — all other cases

  Classification is fully deterministic and pure — same eventType always
  produces the same classification. No DB access, no async.

5.3  Retention Semantics — READ-ONLY in P11-A

  The RetentionClassification field is:
  ✓ Set at write time via classifyRetention()
  ✓ Stored in the DB for each entry
  ✓ Surfaced via API for operator visibility
  ✓ Aggregated in ComplianceSummary (byRetentionClassification counts)

  Retention policies (actual deletion schedules, legal holds, export rules)
  are NOT implemented in P11-A. The classification provides the semantic
  foundation for future retention enforcement without building enforcement now.
  Implementing retention schedules requires legal/compliance review — P11-A
  intentionally stops at classification only.

A) audit_chain_recorded
   Level:  INFO
   Fired:  POST /platform/compliance/audit-chains on successful INSERT.
   Fields: chainId, entityType, entityId, integrityStatus="verified",
           retentionClassification (computed), action="audit_chain_entry_recorded"
   Purpose: confirms a new immutable entry has been successfully appended.

B) audit_integrity_verified
   Level:  INFO
   Fired:  GET /platform/compliance/audit-integrity (no anomalies found)
           GET /platform/compliance/audit-chains (read path)
   Fields: chainId="", entityType, entityId, integrityStatus (overall),

   Purpose: records that an integrity verification pass completed cleanly.

C) audit_integrity_anomaly_detected
   Level:  WARN
   Fired:  GET /platform/compliance/audit-integrity (when compromisedCount > 0)

           is "compromised" or "orphaned")
   Fields: chainId="", entityType, entityId, integrityStatus="compromised",
           retentionClassification="forensic_critical",
           action="integrity_anomaly_detected" or "forensic_integrity_anomaly"
   Purpose: WARN-level signal that tamper evidence was found. Requires human review.

D) forensic_timeline_reconstructed
   Level:  INFO
   Fired:  GET /platform/compliance/forensics/:entityId on every call.
   Fields: chainId="", entityType (from timeline), entityId,
           integrityStatus (chainIntegrity), retentionClassification="forensic_critical",
           action="forensic_timeline_reconstructed"
   Purpose: audit trail of every forensic reconstruction request.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

All routes:
  - Require Authorization: Bearer <jwt> with role = super_admin
  - Protected by requireAuth + requireSuperAdmin middleware chain
  - APPEND-ONLY: no UPDATE or DELETE on compliance_audit_chains table
  - Forensic-safe serialization: all timestamps as ISO 8601 strings
  - Deterministic pagination via limit/offset with max 200 per page

POST /platform/compliance/audit-chains
  Auth:    super_admin
  Body:    { entityType, entityId, workspaceId?, previousAuditHash?,
             eventType, operatorId, payload, occurredAt }
  Process: buildAuditChainEntry() → validation → INSERT
  Returns: { entry: row } (201 Created)
  Errors:  400 { error: "AUDIT_VALIDATION_FAILED", codes: [...] }
           if operatorId, entityId, or eventType is empty, or occurredAt is invalid
  Safety:  currentAuditHash computed server-side — never accepted from caller.
           No UPDATE path exists — each call is a fresh append-only INSERT.

GET /platform/compliance/audit-chains
  Auth:    super_admin
  Params:  entityType?, entityId?, workspaceId?, integrityStatus?,
           retentionClassification?, limit (1–200, default 100), offset
  Returns: { entries: row[], total, limit, offset }
  Order:   DESC recordedAt (most recent first)

GET /platform/compliance/audit-integrity
  Auth:    super_admin
  Params:  entityType?, entityId?, workspaceId?
  Process: load all matching entries → verifyAuditIntegrity() → buildComplianceSummary()
  Returns: { report: AuditIntegrityReport, summary: ComplianceSummary }
  Events:  audit_integrity_anomaly_detected (WARN) if compromisedCount > 0
           audit_integrity_verified (INFO) if clean

GET /platform/compliance/forensics/:entityId
  Auth:    super_admin
  Params:  entityId (path)
  Process: load all entries for entityId → reconstructAuditTimeline()
  Returns: { timeline: ForensicTimeline }
  Order:   timeline.events sorted ASC by occurredAt
  Events:  forensic_timeline_reconstructed (INFO) always
           audit_integrity_anomaly_detected (WARN) if chainIntegrity is
           "compromised" or "orphaned"

8. DB SCHEMA — compliance_audit_chains
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Table:   compliance_audit_chains
Columns:

  id                      serial PK
  chain_id                text UNIQUE NOT NULL        — "audit:<entityType>:<entityId>-<msTs>"
  entity_type             text NOT NULL               — one of 5 entity types
  entity_id               text NOT NULL               — the specific entity instance ID
  workspace_id            int FK → workspaces.id      — nullable (platform events have no workspace)
                          ON DELETE CASCADE
  previous_audit_hash     text                        — nullable for genesis entry
  current_audit_hash      text NOT NULL               — SHA-256 computed hash
  event_type              text NOT NULL               — what happened
  operator_id             text NOT NULL               — who performed the action
  payload                 json NOT NULL               — frozen event snapshot
  occurred_at             timestamp NOT NULL          — when the event happened (source time)
  recorded_at             timestamp NOT NULL DEFAULT now() — when audit entry was written
  integrity_status        text NOT NULL DEFAULT "verified"
  retention_classification text NOT NULL DEFAULT "operational"

Indexes:
  audit_entity_id_idx          (entity_id)
  audit_entity_type_id_idx     (entity_type, entity_id)
  audit_workspace_idx          (workspace_id)
  audit_recorded_at_idx        (recorded_at)
  audit_retention_idx          (retention_classification)
  audit_integrity_status_idx   (integrity_status)

No UPDATE or DELETE operations are ever issued against this table.
workspaceId CASCADE ensures workspace audit data is removed when a workspace
is deleted — intentional for tenant data lifecycle management. If forensic-
critical data must survive workspace deletion, a legal hold mechanism would
need to be added as a future phase.

10. REMAINING COMPLIANCE GAPS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

P11-A is intentionally minimal — foundations only. Known remaining gaps:

  1. AUDIT CHAIN NOT AUTO-WIRED TO EXISTING OPERATIONS
     The compliance_audit_chains table is populated via the explicit POST route.
     Existing routes (P10-D orchestration, P10-E execution, P10-B incident)
     do NOT automatically append audit entries. Each operation that should be
     audited must call POST /platform/compliance/audit-chains explicitly.
     Gap: a middleware hook or event listener on existing operation routes would
          auto-record audit entries without requiring callers to be modified.

  2. PREVIOUS_AUDIT_HASH CONTINUITY NOT ENFORCED AT DB LEVEL
     The previousAuditHash of a new entry is supplied by the caller and is not
     validated against the actual most-recent entry for that entityId at INSERT time.
     Gap: a DB-level constraint or application-level check that the supplied
          previousAuditHash actually matches the currentAuditHash of the most
          recent entry for that entityId would strengthen chain integrity.

  3. ORPHAN FALSE POSITIVES IN PARTIAL QUERIES
     detectOrphanedRecords() and verifyAuditIntegrity() may report false positives
     when operating on a filtered subset of entries (e.g., filtered by workspaceId
     or paginated) because the predecessor entries may be excluded from the subset.
     Gap: full chain loading or a "load full chain for entityId" API path would
          enable accurate orphan detection regardless of filter context.

  4. NO RETENTION ENFORCEMENT
     RetentionClassification is stored and surfaced but has no enforcement.
     Records classified as forensic_critical are not distinguished from
     operational records for deletion protection.
     Gap: a soft-delete flag (deletionProtected) or legal_hold boolean column
          would enable the application layer to block deletion of protected entries.

  5. NO EXTERNAL SIGNING / NON-REPUDIATION
     The hash chain proves tampering within the application layer but provides
     no non-repudiation guarantee — a sufficiently privileged actor could modify
     both the data and the stored hashes simultaneously.
     Gap: external signing (e.g., timestamping entries with an RFC 3161
          trusted timestamp, or exporting audit hashes to an external immutable
          log service) would close the non-repudiation gap.

  6. AUDIT CHAIN NOT EXPOSED IN SUPER ADMIN FRONTEND
     The three GET routes and one POST route are available in the API but
     no frontend views exist yet for:
       - Audit chain browser (list + filter)
       - Integrity dashboard (overall status + compromised count)
       - Forensic timeline viewer for a specific entity
     Gap: P11-B or a dedicated frontend phase could add these views.

  7. CROSS-ENTITY FORENSIC RECONSTRUCTION
     GET /platform/compliance/forensics/:entityId returns the chain for one
     entity. It does not follow the P10-A–P10-F lifecycle chain:
     "for incident X, show all audit events for its recommendations,
     orchestrations, and executions."
     Gap: a compound forensics endpoint that resolves the full entity tree
          (incident → recommendations → orchestrations → executions) and
          returns a unified timeline would enable end-to-end lifecycle forensics.

  8. WORKSPACE CASCADE DELETES AUDIT DATA
     workspaceId uses ON DELETE CASCADE — when a workspace is deleted, all
     compliance_audit_chains entries for that workspace are deleted.
     Gap: for compliance regimes requiring retention of audit data after tenant
          offboarding, the cascade should be replaced with a soft-delete or
          a legal hold mechanism that detaches rather than deletes.
