SECTION 1 — GOVERNANCE CONSOLE SHELL ARCHITECTURE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

P12-A converts the Phase 11 governance backend into a usable operations console
by adding a frontend shell layer. The shell is frontend-only, read-only, and
super-admin-restricted. It connects to the 17 existing governance API routes
without adding any backend changes.

Architecture principle:
  Every governance route that existed in Phase 11 now has a corresponding
  frontend page that displays its data in the super-admin console.
  No new backend routes were created. No existing routes were modified.

Layer separation maintained:
  P11 backend (API routes, engine files, DB tables) — UNTOUCHED
  P12-A frontend (pages, hooks, layout) — ADDITIVE ONLY

Dependency graph:
  governance-console-config.ts  (pure TS constants — no dependencies)
         ↓
  governance-console-hooks.ts   (TanStack Query hooks — depends on api-client)
         ↓
  7 × super-admin-governance-*.tsx  (React pages — depends on hooks + ui)
         ↓
  super-admin-layout.tsx  (nav section added — depends on lucide-react + wouter)
         ↓
  App.tsx  (7 new routes registered — depends on page components)

SECTION 2 — NAVIGATION STRUCTURE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The super-admin sidebar now has a "GOVERNANCE" section below the existing 5

2.2  Active route highlighting

  Governance nav items use the same active-state styling as platform nav items:
  `bg-sidebar-accent text-sidebar-accent-foreground` when the current location
  matches the item's href. Exact-match logic is used for the overview (to avoid
  highlighting it when viewing a sub-section). Prefix-match logic is used for
  all sub-section links.

2.3  data-testid attributes

  Each governance nav link carries a `data-testid` attribute derived from the
  label:  `governance-nav-overview`, `governance-nav-audit-integrity`, etc.
  These enable future automated E2E test targeting.

2.4  Governance nav is super-admin-only

  The governance nav items live inside `super-admin-layout.tsx`, which is only
  rendered by `SuperAdminRoute`. Non-super-admin users never receive this layout
  and therefore never see the governance nav — not even as a disabled state.
  The governance nav items are not conditionally hidden; they are structurally
  absent from non-super-admin views.

  Route path                                        Component
  ──────────────────────────────────────────────────────────────────────────────
  /super-admin/governance/audit-integrity           SuperAdminGovernanceAudit
  /super-admin/governance/violations                SuperAdminGovernanceViolations
  /super-admin/governance/workflows                 SuperAdminGovernanceWorkflows
  /super-admin/governance/analytics                 SuperAdminGovernanceAnalytics
  /super-admin/governance/topology                  SuperAdminGovernanceTopology
  /super-admin/governance/evidence-packages         SuperAdminGovernanceEvidence
  /super-admin/governance                           SuperAdminGovernance (overview)

Route ordering note:
  Sub-section routes are registered BEFORE the overview route
  (/super-admin/governance). This prevents wouter's prefix-matching from
  capturing all sub-paths as the overview. The overview route acts as the
  catch-all for the /governance prefix.

Route guard:
  All 7 routes use SuperAdminRoute which enforces:
  - auth.isLoaded: true
  - auth.isSignedIn: true
  - auth.user?.role === "super_admin"
  Any failure condition redirects to /sign-in. The governance console is
  never rendered for non-super-admin users at any point.

SECTION 4 — ACCESS CONTROL BEHAVIOR
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

4.1  Unauthenticated access

  Direct navigation to /super-admin/governance/* by an unauthenticated user
  triggers SuperAdminRoute's guard. The user is immediately redirected to
  /sign-in before any page component is mounted. No governance data is fetched,
  no governance UI is rendered.

4.2  Authenticated non-super-admin access

  A user with role admin/manager/member who navigates directly to
  /super-admin/governance/* is blocked by:
  - SuperAdminRoute checks `auth.user?.role !== "super_admin"` → /sign-in
  Their role does not entitle them to reach the SuperAdminLayout, so the
  governance nav is never visible even transiently.

4.3  Authenticated super_admin access

  A super_admin user gets the full SuperAdminLayout with governance nav.
  All 7 governance pages are accessible. API calls from the hooks carry the
  JWT Bearer token (via setAuthTokenGetter). The backend's requireSuperAdmin
  middleware then validates the token server-side before returning any data.

4.4  Governance data protection in client

  The governance console hooks only execute when the component is mounted
  inside SuperAdminLayout (which is only rendered for super_admin users).
  No governance API call is ever triggered for non-super-admin sessions.

4.5  No governance data in workspace routes

  Governance data (audit chains, violations, workflows, analytics, topology,
  evidence packages) is never fetched or displayed in any workspace-scoped route
  (ProtectedRoute). No cross-contamination between the super-admin governance
  console and workspace user routes.

SECTION 6 — UI STATE HANDLING
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Every governance page handles all 5 required states:

  LOADING
    - Skeleton components replace data placeholders during query loading.
    - Stat cards show Skeleton (h-7 w-12) for individual values.
    - List sections show rows of Skeleton elements matching expected row height.
    - The overview readiness banner shows a Skeleton for the status text.

  EMPTY
    - List sections (audit entries, violations, workflows, policies, etc.) show
      a centered empty state with an icon, title, and explanatory sub-text.
    - Example (violations): checkmark icon, "No policy violations detected",
      "All 8 governance policies are currently satisfied."
    - Example (workflows): checkmark icon, "No open investigations".

  ERROR
    - GovernanceErrorState component renders a red-bordered error notice with
      the message: "Unable to load governance data" + contextual explanation.
    - Error state is shown per-section (a query failing in one card does not
      collapse the entire page).
    - data-testid: governance-error-state

  UNAUTHORIZED
    - Handled by SuperAdminRoute before any page mounts.
    - Redirects to /sign-in — no partial rendering of governance UI occurs.

  READ-ONLY NOTICE
    - GovernanceReadOnlyNotice renders on every page below the header.
    - Always visible regardless of loading/empty/error state.
    - data-testid: governance-read-only-notice

Read-only guarantee:
  The hooks file exports no useMutation hooks.
  The hooks file calls no HTTP POST, PATCH, PUT, or DELETE endpoints.
  The hooks file has no side effects except dispatching GET queries.

9.5  Why no React component imports in tests

  components that use JSX, wouter, lucide-react, or shadcn/ui would fail at
  module resolution. The existing test files (governance-utils.test.ts,
  governance-trend-utils.test.ts) import only pure TypeScript utility functions.
  P12-A follows the same pattern by testing the pure constants file
  (governance-console-config.ts) exclusively — which has zero dependencies.
